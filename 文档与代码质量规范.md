# 文档与代码质量规范

> 状态：首版正文
>
> 定位：本文件是仓库内文档与代码质量要求的唯一真源。
>
> 目的：把已经在 `01_python` 中验证过的高质量模式，抽象为可复用、可评审、可持续演进的通用规范，并能覆盖 `02_llm`、`03_foundation` 以及后续 `04_rag`、`05_agent`、`06_application`。

---

## 1. 适用范围

本规范适用于以下内容：

1. `docs/*` 下的阶段文档、章节文档、阶段级 README、阶段级任务与设计文档。
2. `source/*` 下与学习材料配套的示例代码、练习代码、参考答案、最小项目骨架和项目 README。
3. 章节级内容、项目级内容、阶段级内容之间的映射与完成标准。

本规范不直接承担以下职责：

1. 不决定某一章具体讲哪些知识点。
2. 不规定某项技术只能采用一种实现方式。
3. 不替代阶段实施方案、项目设计说明和任务拆解文档。
4. 不承载时效性很强的平台推荐或一次性决策。

与其他文件的关系如下：

1. `README.md` 负责仓库导航和学习主线，不承载完整规范正文。
2. `CLAUDE.md` 负责执行摘要和协作约束，不复制主规范全文。
3. 各阶段 `README.md` 负责阶段入口、学习顺序和链接组织。
4. 各阶段 `implementation.md`、`design.md`、`tasks.md` 负责阶段特有落地方案。
5. 若出现冲突，以本文件为准；阶段文档只能在本规范允许的范围内做适配补充，不能推翻通用规则。

---

## 2. 目标

本规范的目标不是“让文档写得更多”或“让代码文件变多”，而是建立稳定闭环。

统一目标如下：

1. 文档可学：学习者知道先看什么、先跑什么、先写什么。
2. 代码可跑：关键示例、最小项目和主要入口必须能直接运行或具备清晰运行说明。
3. 映射清晰：文档、代码、练习、实现方案之间的对应关系必须显式写出。
4. 结构稳定：同阶段内容在栏目职责、目录组织、命名风格上保持一致。
5. 完成标准明确：什么算完成、什么不算完成，不依赖临时判断。
6. 便于复用：未来写 `04`、`05`、`06` 时，不再反复重建规则。
7. 便于评审：文档和代码都可以按统一清单 review，而不是凭感觉判断。

---

## 3. 核心原则

### 3.1 一节一闭环

每一节、每一章、每一个小项目都应形成局部闭环。不能长期处于“文档先写一半、代码以后再补、映射靠脑补”的状态。

闭环最少应包含：

1. 明确目标
2. 清晰入口
3. 对应代码
4. 学习或实施顺序
5. 完成标准

### 3.2 先建立入口，再补扩展

优先提供最小可运行入口，再补练习、扩展案例、对照实现和工程化封装。

禁止把以下内容当成第一入口：

1. 只有半成品的练习文件
2. 需要读者自己拼接的零散代码片段
3. 缺少运行说明的项目骨架

### 3.3 显式映射，禁止靠脑补

文档必须明确说明：

1. 当前内容对应哪些代码文件
2. 哪个文件是第一入口
3. 哪些文件属于练习或对照
4. 哪些文档对应设计、任务拆解或运行说明

### 3.4 文档服务学习路径，不只是留档

文档首先是学习与实施路线，而不是资料归档。优先解决的问题是：

1. 为什么学
2. 学到什么程度
3. 先做什么
4. 容易错在哪里
5. 学完如何判断自己达标

### 3.5 代码服务理解与实践，不只是堆文件

代码的首要作用是帮助理解和动手验证，而不是为了“显得工程化”而堆结构。文件数量、抽象层次和目录复杂度都应服务于当前阶段的学习目标。

### 3.6 先做最小正确闭环，再做完整工程化

对章节级内容，先保证核心链路正确、可跑、可解释。

对项目级内容，先保证最小骨架成立，再逐步补：

1. 配置
2. 分层
3. 日志
4. 测试
5. API 或 CLI 入口

### 3.7 通用规范优先，阶段适配随后补充

先抽取所有阶段共用的不变量，再为 `01`、`02`、`03` 以及未来阶段补适配规则。阶段适配是补充，不是重新发明一套新规范。

### 3.8 规范服务复用与评审，不追求形式化堆叠

规范的价值在于降低返工、统一风格、提高 review 质量。凡是不能提升复用性、可执行性或可评审性的要求，都不应成为强制项。

## 4. 术语说明

### 4.1 节

最小学习单元，通常对应一篇章节文档和一组配套代码。

### 4.2 章

同一阶段中的一个完整主题块。根据实际编排，一章可以等同于一节，也可以由多个节组成。

### 4.3 阶段

仓库学习主线中的一级目录，例如 `01_python`、`02_llm`、`03_foundation`。

### 4.4 主示例文件

当前节或当前主题的第一运行入口，用于先建立直觉和最小闭环。

### 4.5 练习文件

用于手写和巩固的代码文件。练习文件不承担首次讲解职责。

### 4.6 参考答案文件

用于卡住时对照和校验思路的文件。可以公开，也可以作为本地保留文件。

### 4.7 最小项目

为某一阶段核心目标服务的独立、轻量、可运行骨架，不等于完整业务产品。

### 4.8 实施方案

用于说明阶段级落地边界、实施顺序、交付物和完成标准的文档，例如 `implementation.md`。

### 4.9 完成定义

用于判断某一节、某一章、某一阶段是否真正完成的明确标准。

### 4.10 代码映射表

用于显式说明文档小节和代码文件、练习文件、项目模块之间关系的表格或等价结构。

---

## 5. 文档质量标准

### 5.1 文档固定结构

章节文档不要求所有标题名字完全一致，但必须覆盖以下职责：

1. 本节目标或学习目标
2. 本节在整体路径中的作用
3. 学习边界或本章不展开的内容
4. 学习顺序或实施顺序
5. 第一运行入口或第一阅读入口
6. 核心概念、核心流程或核心抽象展开
7. 代码映射或项目映射
8. 练习说明、任务说明或实施说明
9. 完成标准

允许的变体：

1. `01_python` 可更强调“本节怎么学”“先跑一下”“练习说明”。
2. `02_llm` 可更强调“运行方式”“环境变量”“CLI / API / 服务层职责”。
3. `03_foundation` 可更强调“边界说明”“设计说明”“任务拆解”“阶段收口”。

不允许的情况：

1. 只有知识点堆积，没有学习顺序。
2. 只有理论解释，没有对应代码或项目落位。
3. 只有代码链接，没有说明为什么先看它。
4. 只有目录图，没有职责说明。

### 5.2 本节目标的写法

本节目标应写成“学完后你能做什么”，而不是抽象概述。

要求：

1. 3-7 条为宜。
2. 尽量使用可验证动词，如“能运行”“能解释”“能实现”“能区分”“能修改”。
3. 目标要与本节代码和完成标准对应。

不建议写法：

1. “了解某技术”
2. “熟悉一些基础知识”
3. “掌握相关概念”但没有后续验证方式

### 5.3 学习顺序或实施顺序的写法

这是强制项。必须显式说明：

1. 先运行哪个文件或先读哪份文档
2. 再看哪些核心部分
3. 哪些内容必须手写
4. 卡住时先回看哪里
5. 最低通过标准是什么

推荐表达方式：

1. 用有顺序的步骤写出“先看什么、再跑什么、再写什么”。
2. 对项目级内容写出“先搭什么、后补什么、最后如何收口”。

### 5.4 第一入口的写法

每节都应该有第一入口。第一入口可以是：

1. 一个主示例文件
2. 一个最小脚本
3. 一个最小项目入口
4. 一份阶段 README 中指向的第一章节

若是代码入口，文档必须说明：

1. 运行命令
2. 运行前提
3. 预期输出或观察点
4. 为什么要先跑这份代码

### 5.5 核心内容的展开方式

每个核心概念、核心流程或核心抽象，优先采用以下顺序：

1. 一句话白话解释
2. 为什么它重要
3. 最小例子或最小流程
4. 常见坑或边界提醒
5. 与当前项目或后续阶段的关系

要求：

1. 一个代码块只解决一个主要问题。
2. 不要在一个最小示例里同时引入太多新概念。
3. 对初学者高频误区要显式提醒。
4. 对工程化阶段要显式说明“这一层为什么单独存在”。

### 5.6 代码映射表的写法

章节文档原则上都应有代码映射说明。形式可以是表格，也可以是等价的清单结构，但必须显式。

映射说明至少回答：

1. 文档这一部分对应哪个文件
2. 哪个文件是主入口
3. 哪个文件是练习或扩展
4. 哪个文件是参考答案、对照实现或项目模块
5. 当前内容与后续章节或项目的衔接点是什么

推荐表头：

| 文档部分 | 对应代码/文档 | 角色 | 说明 |
|----------|---------------|------|------|

### 5.7 练习说明、任务说明、实施说明的写法

不同阶段名称可以不同，但职责必须清楚。

应明确说明：

1. 这一部分是练习、任务还是实施动作
2. 输入是什么
3. 输出是什么
4. 验收标准是什么
5. 卡住时回看什么

禁止把“读者自己想想怎么做”当成主要说明方式。

### 5.8 完成标准的写法

每节文档都必须有完成标准。

完成标准应同时覆盖：

1. 理解层
2. 操作层
3. 代码层
4. 映射层

例如：

1. 能用自己的话解释核心概念
2. 能跑通主示例
3. 能完成至少一个练习或最小任务
4. 能说清楚文档与代码之间的对应关系

## 6. 代码质量标准

### 6.1 每节最少产物

章节级内容最少应具备以下产物中的适用部分：

1. 主示例文件或等价入口
2. 练习文件、任务文件或扩展示例
3. README 或运行说明（当目录内文件不止一个时）
4. 文档中的代码映射说明

若某节天然不适合练习文件，必须用以下任一方式替代：

1. 任务清单
2. 对照实现
3. 项目骨架中的明确待办模块

### 6.2 主示例文件标准

主示例文件应满足：

1. 是第一入口
2. 能独立运行或有清晰运行前提
3. 覆盖本节主线
4. 文件职责单一明确
5. 代码长度适中，便于阅读

主示例文件不应承担：

1. 把所有扩展场景都塞进一个大脚本
2. 同时承担练习、答案、测试和工程骨架四种职责
3. 依赖读者手动拼接多个片段后才能运行

### 6.3 练习文件标准

练习文件应满足：

1. 目标明确
2. 题目边界明确
3. 输入输出或预期结果清楚
4. 与主示例形成递进关系

练习文件不应承担：

1. 首次讲解职责
2. 替代主示例成为唯一入口
3. 用模糊描述让学习者猜题意

### 6.4 参考答案文件标准

当使用参考答案文件时，应满足：

1. 与练习目标对应
2. 主要用于对照，不喧宾夺主
3. 命名和位置稳定
4. 不与主示例混在同一文件中

若暂不公开答案，也应在文档或目录说明中明确是否存在本地保留版本。

### 6.5 README 与运行说明标准

当一个目录下存在多个文件、多个入口或一个独立小项目时，必须提供 README 或等价运行说明。

README 至少说明：

1. 目录定位
2. 关键文件职责
3. 运行方式
4. 环境变量或依赖前提
5. 建议阅读或运行顺序

### 6.6 独立小项目骨架标准

适用于 `02_llm` 后半段和 `03_foundation` 及以后阶段。

最小项目骨架通常应有：

1. README
2. 明确入口，如 CLI、API 或脚本
3. 配置说明，如 `.env.example`、配置模块或参数说明
4. 基本分层，如入口层、服务层、模型层、工具层或数据层中的适用部分
5. 至少一个能验证主线能力的最小流程

项目骨架不要求一开始就非常完整，但必须做到：

1. 目录职责清晰
2. 扩展方向可见
3. 最小流程可跑
4. 不把不同职责混成一个巨型文件

### 6.7 可运行性要求

“可运行”至少满足以下之一：

1. 在仓库当前依赖下可直接运行
2. 只需补充文档中写明的环境变量即可运行
3. 若依赖外部平台，可提供 mock 路径、对照路径或明确的运行替代方案

不算可运行的情况：

1. 只给目录不写入口
2. 只给代码不写依赖前提
3. 一运行就要求读者自行补大量隐藏上下文

### 6.8 注释、输出与样例要求

代码应优先做到自解释，必要时用简短注释说明复杂块。

推荐保留：

1. 关键打印输出或最小结果展示
2. 样例输入
3. 对易错点的短注释

禁止：

1. 大段重复解释型注释
2. 和代码明显不一致的陈旧注释
3. 把说明全部塞进注释而不是文档

## 7. 文档与代码映射规则

### 7.1 章节级映射

章节级映射必须明确：

1. 本文档对应的主示例文件
2. 练习文件或任务入口
3. 参考答案或对照实现
4. 章节与章节之间的承接关系

若一个文档对应多个代码文件，必须说明每个文件的角色，而不是只罗列文件名。

### 7.2 项目级映射

项目级内容必须同时说明：

1. 哪篇文档负责解释原理
2. 哪篇文档负责设计和结构
3. 哪篇文档负责任务拆解和实施顺序
4. 哪个目录承载真正代码
5. README 如何作为项目入口

`03_foundation` 是项目级映射的标准样本。此类场景不能只靠章节映射处理。

### 7.3 运行路径与测试路径说明

凡是需要运行、测试、对照或多入口切换的内容，都应显式说明：

1. 从哪里运行
2. 运行哪个文件或命令
3. 是否需要环境变量
4. 如果有测试，应如何执行
5. 如果存在 mock / real 双路径，应如何区分

### 7.4 反向映射要求

代码目录也应尽量支持反向追溯到文档。

推荐做法：

1. README 中链接对应章节
2. 文件名与章节编号保持对应
3. 在项目 README 中写出“先读哪篇文档，再跑哪些文件”

## 8. 标准产物定义

### 8.1 章节级标准产物

一节至少需要以下产物中的适用部分：

1. 一篇章节文档
2. 一个主示例入口
3. 一个练习或任务入口
4. 必要时提供参考答案或对照实现
5. 若目录内文件较多，则提供目录级 README

### 8.2 小项目级标准产物

一个最小独立项目至少需要：

1. 项目 README
2. 入口文件或入口命令
3. 目录职责说明
4. 运行方式
5. 配置说明
6. 关键模块说明

若项目承担教学主线中的关键作用，还应补充：

1. 设计说明
2. 任务拆解
3. 完成标准

### 8.3 阶段级标准产物

一个阶段至少应有：

1. 阶段 README
2. 阶段大纲或 outline
3. 章节文档集合
4. 与章节对应的代码目录

若阶段复杂度较高，还应有：

1. 实施方案
2. 独立项目
3. 项目设计说明
4. 任务拆解文档

---

## 9. 完成定义

### 9.1 章节完成定义

一节真正完成，至少同时满足以下条件：

1. 文档结构完整，承担了学习路径职责。
2. 有明确第一入口。
3. 有对应代码、练习或任务，不靠脑补建立联系。
4. 文档写清学习顺序或实施顺序。
5. 存在清晰完成标准。
6. 主示例或核心入口可运行，或具备明确运行说明。

### 9.2 章节不合格的常见情形

以下任一情况出现，该节都不应视为完成：

1. 只有文档，没有配套代码或任务入口。
2. 只有代码，没有文档说明如何学习。
3. 主示例不能跑，且文档没有说明前提。
4. 练习成为唯一入口。
5. 文档与代码没有显式映射。
6. 完成标准缺失。
7. 同节内容结构严重混乱，无法判断先后顺序。

### 9.3 阶段完成定义

一个阶段真正完成，至少满足：

1. 阶段 README、outline 和章节体系完整。
2. 阶段内部的命名、结构和风格基本一致。
3. 章节文档与代码目录之间存在稳定映射。
4. 若阶段需要独立项目，则项目已经能说明定位、边界、结构和运行方式。
5. 阶段完成标准能够覆盖理解、实现、运行和收口四个层面。

### 9.4 阶段不合格的常见情形

以下情形不能视为阶段完成：

1. 阶段只有大纲，没有稳定正文和配套代码。
2. 章节质量差异过大，没有统一结构。
3. 项目骨架存在，但没有 README、运行路径和职责说明。
4. 原理文档、设计文档和代码目录长期脱节。
5. 阶段收口标准缺失，无法判断是否可以进入下一阶段。

## 10. 分阶段适配规则

### 10.1 `01_python`

重点目标：

1. 建立基础语法和工程起手能力
2. 强化章节闭环
3. 强化 demo / exercise / reference 分工

强制关注点：

1. 每节尽量有第一入口代码
2. 优先解决“读得懂、跑得通、写得出”
3. 文档中要显式写学习顺序
4. 练习不能承担首次理解职责

### 10.2 `02_llm`

重点目标：

1. 从单个示例过渡到最小 AI 应用工程骨架
2. 建立配置、运行方式、服务层和多入口意识

强制关注点：

1. `.env.example` 或等价配置说明
2. CLI / API / 服务层职责边界
3. README 中写清运行方式和目录说明
4. 不同示例文件之间的递进关系明确

### 10.3 `03_foundation`

重点目标：

1. 解决“原理文档 + 设计说明 + 任务拆解 + 独立项目”并行的复杂场景
2. 为 `04_rag` 和 `05_agent` 建立统一工程心智模型

强制关注点：

1. 原理文档要写清理解边界，不做无关扩展
2. 实施方案要写清交付物、分阶段实施顺序和完成标准
3. `foundation_lab` 这类最小项目必须独立、轻量、可运行
4. 设计说明、任务拆解和代码目录之间必须形成项目级映射

### 10.4 `04_rag`

当前先约束交付物类型，不提前写死技术细节。

至少应关注：

1. 检索链路的数据流说明
2. 数据来源、切分、检索、生成之间的边界
3. 最小可运行检索问答路径
4. mock 路径和真实路径的关系
5. 评估、观测或对照方式的最小说明

### 10.5 `05_agent`

当前先约束交付物类型，不提前写死框架细节。

至少应关注：

1. Tool、状态、编排和执行边界
2. Agent 与普通聊天、普通 RAG 的区别
3. 最小可运行 agent 流程
4. 人工干预、错误处理和可观测性说明

### 10.6 `06_application`

当前先约束交付物类型，不提前写死产品形态。

至少应关注：

1. 从实验型项目过渡到应用型项目的分层变化
2. 前端、后端、模型服务、状态管理之间的职责
3. 配置、日志、错误处理、部署或运行说明
4. 用户路径和系统路径之间的映射
5. 文档、代码、项目结构和演示入口的一致性

## 11. 命名与目录规范

### 11.1 目录命名

1. 阶段目录使用统一编号前缀，如 `01_python`、`02_llm`。
2. 章节目录和文档文件尽量保持编号对应，便于映射。
3. 同阶段内避免同时出现多套命名风格。

### 11.2 文档命名

1. 章节文档采用稳定编号加语义名称。
2. 阶段入口优先使用 `README.md`。
3. 大纲优先使用 `outline.md`。
4. 实施方案、设计说明、任务拆解使用语义清晰的固定命名。

### 11.3 代码命名

1. 主示例文件优先使用语义明确的名称，如 `demo.py`、`main.py`、`01_xxx.py`。
2. 练习与答案优先使用稳定命名，如 `exercises.py`、`reference.py`。
3. 项目级代码按职责命名，不使用含糊文件名。

### 11.4 命名一致性要求

要求至少做到：

1. 文档编号和代码目录编号能相互对应。
2. README 中提到的文件名与实际文件一致。
3. 不再使用的旧命名应及时迁移或在 README 中说明。

## 12. 评审清单

### 12.1 文档评审清单

1. 是否写清本节目标。
2. 是否写清学习顺序或实施顺序。
3. 是否写清边界，说明本节不展开什么。
4. 是否有第一入口。
5. 是否有清晰映射。
6. 是否有练习说明、任务说明或实施说明。
7. 是否有完成标准。
8. 语言是否优先解决“能不能学会”，而不是只堆概念。

### 12.2 代码评审清单

1. 是否存在主示例或等价入口。
2. 是否可运行，或已写清运行前提。
3. 文件职责是否清晰。
4. 是否把多个层次的职责硬塞进一个文件。
5. 是否提供了练习、任务或对照实现中的适用部分。
6. README 是否足够支撑阅读和运行。
7. 项目级内容是否已有最小骨架和主线流程。

### 12.3 命名与映射检查

1. 文档编号、目录编号、代码文件是否能互相对应。
2. README、正文、代码中的命名是否一致。
3. 是否出现“文档说 A，代码实际是 B”的情况。

### 12.4 阶段收口检查

1. 阶段 README 是否能作为入口。
2. 阶段大纲、章节、代码和项目骨架是否互相对齐。
3. 是否已经具备进入下一阶段所需的最低能力闭环。
4. 是否仍存在需要靠口头解释才能成立的关键缺口。

## 13. 例外处理规则

允许偏离模板，但必须满足“有理由、有说明、有边界”。

### 13.1 允许例外的情形

1. 某一节是纯原理辨析，暂时不适合配独立练习文件。
2. 某一节的最佳入口不是单文件，而是小项目 README。
3. 某一阶段需要用设计说明和任务拆解替代传统练习说明。

### 13.2 例外时必须补充的说明

1. 为什么不能直接套标准模板。
2. 当前采用了什么替代结构。
3. 学习者或维护者应如何进入。
4. 完成标准如何调整。

### 13.3 如何防止例外失控

1. 例外只能改变表现形式，不能取消学习路径、映射和完成标准。
2. 同类例外重复出现后，应评估是否升级为正式适配规则。
3. 不能把一次性便利写成长期默认。

## 14. 与其他文件的关系

### 14.1 与 `README.md` 的关系

`README.md` 负责导航和入口，不复制完整规范正文。应链接到本文件。

### 14.2 与 `CLAUDE.md` 的关系

`CLAUDE.md` 负责执行摘要和协作约束。应引用本文件中的关键要求，而不是重写整份规范。

### 14.3 与历史规范种子的关系

本规范吸收了早期 `01_python` 改造过程中形成的规则，尤其是一节一闭环、先建立入口、显式映射和完成标准等模式。相关历史内容已经并入主规范，后续通用要求统一以本文件为准。

### 14.4 与各阶段实施方案的关系

阶段实施方案、设计说明、任务拆解文档属于本规范的落地补充。它们负责说明阶段特有边界和实施细节，不负责重定义通用质量标准。

## 15. 使用建议

### 15.1 新建章节时

先用本规范检查：

1. 本节目标是否可验证
2. 第一入口是否明确
3. 文档和代码如何映射
4. 完成标准是否已经想清楚

### 15.2 新建小项目时

先用本规范检查：

1. README 是否先行
2. 目录职责是否已分清
3. 主线流程是否有最小闭环
4. 是否已经说明配置、运行和扩展边界

### 15.3 做 review 时

优先检查：

1. 结构是否完整
2. 映射是否显式
3. 入口是否清晰
4. 完成定义是否成立

若这四项不成立，不应继续用局部润色掩盖结构问题。
