# AI Native 前端职业定位与学习路线思考

> 这份文档用于沉淀一次关于职业定位、AI 应用学习目标、个人项目方向与简历叙事的阶段性思考。
>
> 它不是简历文案，也不是项目详细设计文档，而是帮助后续反复校准方向的判断备忘录。

## 一、当前背景

### 1. 过往经验基础

当前主要经验集中在前端、Flutter、跨端客户端与复杂业务交付：

- 具备 Web 前端、Flutter、H5、Uniapp、React 等多端开发经验。
- 做过真实业务里的复杂客户端项目，包括金融交易 App、行情、K 线、长连接、交易链路等。
- 有混合开发、JS Bridge、Flutter 插件、原生能力接入、监控上报、动态化、工程化等经历。
- 当前简历中比较重要的真实公司项目包括：
  - YouFi 金融交易 App
  - ProtoBridge / AI Coding Agent 研发提效相关项目

这些经历说明，当前最强的基础不是纯后端、算法或 AI Infra，而是复杂客户端工程、跨端交付、AI 产品交互和研发提效工具落地。

### 2. 当前 AI 学习进度

当前 AI 应用开发学习项目中的进度大致是：

- `01_python` 已完成
- `02_llm` 已完成
- `03_foundation` 暂时跳过
- `04_rag` 已学习到“四、向量数据库 - 11 Chroma”
- `04_rag` 后续、`05_agent`、`06_application` 尚未系统完成

这说明当前已经具备 Python、LLM API、Prompt、结构化输出、基础 RAG 认知等入门能力，但距离完整掌握生产级 RAG、Agent、Workflow、评估、观测与部署仍有距离。

### 3. 工作中的 AI 现实

当前工作并不是完全强 AI 业务，更多仍然是金融客户端、复杂业务交付和内部 AI 提效。

也就是说，当前不是在公司里完整负责一个面向外部用户的 AI Agent 平台，而是处在：

- 真实复杂客户端业务
- AI Coding / AI 提效工具
- AI Native 前端能力探索
- 自学补齐 RAG / Agent / 后端闭环

这几个方向交汇的位置。

## 二、最初的核心疑虑

这次思考一开始主要围绕几个问题展开。

### 1. 两条路线差异是否很大

一条路线是：

> 高级 Flutter / 跨端客户端 + AI 提效 / AI 产品经验

另一条路线是：

> AI 应用后端 / Agent 工程师

疑虑在于：这两条路线就业前景和长期发展是否差异很大，是否需要尽快从前端/客户端切向 AI 后端/Agent 工程师。

### 2. 哪条路线更适合自己

过往经历明显偏前端、Flutter、客户端和金融业务，但现在又在学习 Python、LLM API、RAG、Agent。

核心疑虑是：

- 是否应该继续强化 AI Native 前端方向？
- 是否应该把 RAG、Agent、后端能力学深，转成 AI 应用后端？
- 如果继续做前端，会不会在 AI 时代不够“硬”？
- 如果转后端，会不会丢掉已有优势并进入别人的主场？

### 3. RAG / Agent 学习目标到底是什么

一开始容易把 RAG / Agent 学习理解成：

> 为了以后去竞争 AI 应用后端 / Agent 工程师岗位。

但进一步分析后，更合理的理解是：

> 学 RAG、Agent、后端基础，是为了补齐 AI 应用开发闭环，让自己能从 AI 能力、后端接口、业务流程到前端体验完整理解和落地。

这不是放弃前端，而是让前端能力不再停留在“接接口和做页面”，而是能参与 AI 应用产品的完整设计。

## 三、职业定位判断

### 1. 主定位

更适合当前阶段的主定位是：

> 高级跨端客户端 / AI Native 前端 / AI 应用前端工程师

或者更具体一点：

> 高级 Flutter / AI 应用前端工程师

这个定位能够承接过去的真实经验，也能自然加入 AI Native、AI Agent 交互、AI Coding 提效、动态 UI、流式输出、证据链展示等新能力。

### 2. 不适合立即主攻的定位

当前不适合直接把主定位切成：

- 纯 AI 应用后端工程师
- Agent 平台工程师
- RAG 平台后端工程师
- 大模型算法工程师
- AI Infra 工程师

原因不是这些方向不好，而是它们的主竞争场在后端工程、平台治理、模型工程、算法理解、部署稳定性、评测观测等方面。

当前已有优势并不在这里，直接切过去短期胜率不高，也容易失去前端/Flutter/金融客户端这条护城河。

### 3. 更准确的自我描述

更准确的表达是：

> 我能做 AI 应用全栈闭环，但我的优势不是和纯后端、算法、AI Infra 正面竞争，而是把 AI 能力做成可用、可控、可展示、可交付的前端/客户端产品。

这句话是当前定位的核心。

## 四、RAG / Agent 学习目标重新定义

### 1. 不是为了卷纯 AI 后端

继续学习 RAG、Agent、FastAPI、LangChain、LangGraph，并不是为了立刻转型成纯 AI 后端或 Agent 工程师。

这些内容仍然要学，但学习目标要从“岗位替换”转成“能力闭环”。

### 2. 是为了补齐 AI 应用闭环

对于 AI Native 前端来说，只会做页面不够。

未来更有价值的前端/客户端工程师，需要理解：

- 模型输出为什么不稳定
- Prompt 和结构化输出如何影响前端体验
- SSE 流式响应如何设计交互
- RAG 的 sources、metadata、score 如何展示为证据链
- Agent 工具调用过程如何可视化
- 什么时候要人工确认、拒答、降级、重试
- 如何通过评估集判断 AI 应用效果有没有变好

所以 RAG / Agent / 后端基础，是让自己不只会“接 AI 接口”，而是能理解 AI 应用主链路。

### 3. 目标深度

当前阶段更合适的目标不是成为检索平台或 Agent 平台专家，而是达到：

- 能独立跑通 LLM + RAG + FastAPI + 前端工作台
- 能理解 chunk、embedding、vector store、retrieval、sources、refusal、eval
- 能使用 Agent / Workflow 解决有限业务编排问题
- 能在前端展示 AI 执行过程、证据链、风险提示和人工协作状态
- 能和后端、算法、平台工程师有效协作

## 五、四类项目关系梳理

当前涉及四类项目或项目想法：

- YouFi
- ProtoBridge
- `course/04_rag/todo.md`
- `course/06_application/outline.md`

它们不应该被混为同一个项目，但可以形成一条统一的能力叙事。

### 1. YouFi

YouFi 是公司真实金融交易客户端项目。

它证明的是：

- 能做复杂 Flutter 客户端
- 能参与金融交易链路
- 能处理行情、K 线、长连接、Protobuf、FFI、订单状态、安全校验等复杂业务
- 能在高实时性、高安全性和高稳定性要求下做客户端交付

这是职业基本盘。

### 2. ProtoBridge

ProtoBridge 是公司内部已有的 AI Coding / 原型到 Flutter 实现辅助相关项目。

它证明的是：

- 能把 AI Coding Agent 引入研发流程
- 能围绕原型、工程规范、实现计划、验证产物构建提效链路
- 能做 CLI / MCP / AI 上下文工具
- 能把 AI 用在客户端交付流程中，而不是只停留在聊天 Demo

它属于产研内部提效方向。

### 3. `04_rag/todo.md`

`course/04_rag/todo.md` 是自学阶段构思的企业内部需求提效评审助手。

它不是公司真实已有项目，而是个人学习与作品化项目。

它适合承担的角色是：

- 补齐 ProtoBridge 上游的需求理解与评审能力
- 使用 RAG 处理 PRD、会议纪要、接口文档、历史评审记录等知识
- 输出带依据的评审问答和结构化评审报告
- 体现 LLM + RAG + FastAPI + React 工作台 + Eval 的完整闭环

它可以被理解成：

> 需求/文档/评审依据进入系统后，先被理解、检索、评审和结构化，然后再为后续原型、实现计划和代码生成提供更清晰的输入。

### 4. `06_application`

`course/06_application/outline.md` 是中长期的金融客服与合规 Copilot 综合项目。

它更偏真正面向业务部门的 AI Agent / AI Copilot 平台：

- 金融客服问答
- 合规审核
- 风险提示
- 知识库接入
- 人工协作
- 审核流
- 反馈沉淀

它更适合在 RAG 和 Agent 都更扎实后再完整推进。

## 六、todo 与 06 的关系

### 1. 不需要二选一

`todo.md` 和 `06_application` 不是互斥关系。

二者可以长期统一在一个更大的 AI 应用体系里，但当前阶段不需要急着合并成一个大而全的 Agent 平台。

更合理的是：

- 阶段上分开
- 叙事上统一
- 架构上预留未来统一编排的可能

### 2. 阶段定位不同

`todo.md` 更适合作为近期项目：

- 对应当前 RAG 学习阶段
- 不强依赖复杂 Agent
- 能集中打通固定 RAG 主链路
- 能快速形成可运行、可展示、可评估的作品
- 和 ProtoBridge 的上游需求评审链路更贴近

`06_application` 更适合作为中长期项目：

- 需要更完整的 Agent / Workflow 能力
- 需要更复杂的金融业务闭环
- 涉及客服、合规、人工协作、反馈回流、权限审计等更大范围
- 更适合作为完成 RAG + Agent 后的综合项目

### 3. 长期可以统一

长期看，它们可以处在同一个大方向下：

```text
需求评审 / 知识理解
-> 研发提效 / 原型到实现
-> 金融业务 Copilot / 客服与合规
-> 统一 AI Workflow / Agent 编排
```

但当前不应该过早设计为一个“统一多 Agent 平台”。

先把一个固定 RAG 项目做扎实，比一开始追求统一平台更重要。

## 七、建议路线

### 1. 短期：完成 RAG 需求评审助手

近期最值得做的是把 `course/04_rag/todo.md` 对应的项目做出来。

目标不是做成完整企业需求管理系统，而是跑通：

```text
文档接入
-> 切分
-> 向量化
-> 检索
-> 上下文组织
-> LLM 生成
-> sources 引用
-> 结构化评审
-> refusal
-> eval
-> React 工作台展示
```

这个项目做成后，可以证明：

- 不只是会用 AI 提效
- 能做 RAG 主链路
- 能做后端 API
- 能做前端可视化工作台
- 能做带依据和评估意识的 AI 应用

### 2. 中期：补 Agent / Workflow

在固定 RAG 稳定之后，再引入 Agent / Workflow。

中期目标可以是：

- 判断是否需要补充材料
- 判断是否需要查历史评审
- 生成待确认问题
- 根据评审维度选择不同检索策略
- 对高风险结论进入人工确认
- 展示 Agent 执行轨迹

这时 Agent 是为业务流程服务，而不是为了炫技。

### 3. 长期：推进金融 Copilot 综合项目

等 RAG 和 Agent 都更稳后，再推进 `06_application`。

这个项目可以承接金融背景，形成更完整的业务作品：

- 金融客服问答
- 合规审核
- 风险提示
- 证据链
- 人工协作
- 反馈回流
- 评估与观测

这会更适合展示“AI 应用全栈闭环 + 金融业务理解 + 前端产品化能力”。

## 八、简历与叙事边界

### 1. 可以包装，但必须真实

YouFi 和 ProtoBridge 是公司真实项目。

`todo.md` 和 `06_application` 是个人自学与作品化项目。

简历中可以把它们放在统一成长线里，但不能把个人项目包装成公司真实落地项目。

更稳妥的表达是：

> 基于金融客户端与 AI Coding 提效经验，个人进一步构建需求评审 RAG 与金融 Copilot 项目，用于补齐从需求理解、知识检索、评审依据、AI 交互到业务闭环的 AI 应用开发能力。

### 2. 不宜过早讲成统一平台

在真正实现前，不要把 todo、ProtoBridge、YouFi、06 讲成已经统一编排落地的平台。

更合理的是讲：

- 当前真实参与了 YouFi 和 ProtoBridge
- 个人基于这些业务观察，补充了 RAG 需求评审助手和金融 Copilot 方向的学习项目
- 这些项目共同指向 AI Native 前端与 AI 应用闭环能力

这样更可信。

## 九、主要风险

### 1. 过早做大而全

如果一开始就把 `todo.md` 和 `06_application` 合成统一多 Agent 平台，很容易导致：

- RAG 不扎实
- Agent 不稳定
- 前端工作台没有亮点
- 项目范围失控
- 最后难以完成

当前更应该先做小闭环。

### 2. 丢掉前端与客户端护城河

不能因为 AI 很热，就忽略 Flutter、金融客户端、复杂交易链路这些已有优势。

这些是真实业务经验，也是和普通 AI Demo 开发者拉开差距的地方。

### 3. 只停留在 AI 提效使用者

如果只是会用 Cursor、Codex、MCP、AI Coding Agent 提效，仍然不够。

需要通过 RAG 项目和后续 Agent 项目证明：

- 能设计 AI 应用链路
- 能做后端服务
- 能处理 sources、refusal、eval
- 能把 AI 过程做成前端可视化体验

### 4. 简历表述过度

简历包装必须基于事实。

可以强调能力迁移和个人项目，但不能把个人学习项目写成公司正式业务落地。

## 十、最终结论

当前最优路线不是：

> 前端转 AI 后端

而是：

> AI Native 前端 + AI 应用闭环能力

更完整的描述是：

> 以高级 Flutter / 跨端客户端为主场，以 RAG、Agent、FastAPI 补齐 AI 应用闭环，通过需求评审 RAG 和金融 Copilot 项目，把 AI 能力产品化、平台化、可视化，最终服务金融客户端和产研提效场景。

这条路线既不放弃已有优势，也不回避 AI 应用开发的核心能力。

下一步最重要的是：

> 先把 `course/04_rag/todo.md` 对应的需求评审 RAG 助手做成一个能运行、能展示、能评估、能讲清楚价值的个人项目。

这会让“AI Native 前端”不只是简历标签，而是有后端链路、有业务场景、有前端体验、有评估意识的真实能力。
