# 金融客服与合规 Copilot 学习大纲 v2

> 目标：构建一个真正体现 AI 必要性的金融 AI 应用，不只会“回答问题”，而是能够理解意图、检索规则、生成依据、执行审核、辅助决策，形成完整商业闭环

---

## 课程定位

- `v1` 更偏“完整 AI 应用骨架”。
- `v2` 更偏“AI-native 商业产品设计”，强调为什么这个系统必须用 AI，而不是普通后端加规则引擎就能替代。
- 主线产品定为：**金融客服与合规 Copilot**。
- 但它不是单一客服机器人，而是一套围绕金融业务问答、知识检索、合规审查、公告解读、风险提示构建的智能协作系统。

## 为什么这个方向更适合你

1. **贴合你的背景**
   - 你有金融交易 App 经验，理解行情、交易链路、复杂业务规则
   - 这会让你的 AI 项目不止是“会接接口”，而是更像懂业务的人在做产品

2. **AI 的必要性强**
   - 用户问题天然是非结构化的
   - 金融规则文档长、更新快、表达复杂
   - 合规判断不能只靠关键词匹配
   - 公告和财报分析需要多文档理解、抽取、归纳、比较

3. **适合包装**
   - 听起来不像“普通 demo”
   - 能兼顾前端体验、后端工程、RAG、Agent、合规、观测
   - 容易讲清楚业务价值和技术难点

## 什么叫“AI 不可替代”

这个项目的 AI 价值，不是“把客服话术换成 LLM 输出”，而是让系统具备以下能力：

1. **理解用户真实意图**
   - 用户常常不会准确描述问题
   - 同一句话可能涉及开户、交易规则、手续费、产品适当性、异常申诉

2. **跨知识源检索并综合**
   - FAQ
   - 产品规则
   - 交易制度
   - 运营手册
   - 历史问答
   - 公告与资讯

3. **生成带依据的回复与建议**
   - 不只是“回答”
   - 还要告诉你：依据来自哪里、风险点是什么、哪些地方需要人工确认

4. **执行语义级合规审核**
   - 识别夸大收益、风险提示缺失、表述不严谨、事实依据不足
   - 给出修改建议和风险解释

5. **形成多步工作流**
   - 先判断问题类型
   - 再决定调用哪个知识源 / 哪个 Agent
   - 再生成结果
   - 再进入审核或人工确认

## 产品定义

### 产品名称

金融客服与合规 Copilot

### 产品愿景

面向金融业务中的客服、运营、审核、投研支持等角色，提供一个“能理解问题、能引用依据、能识别风险、能辅助决策”的智能工作台。

### 核心用户角色

1. **终端用户**
   - 在 App 或 Web 中提问
   - 获取带来源依据的解答

2. **客服 / 运营人员**
   - 获得 AI 生成的回复草稿
   - 查看命中的规则、FAQ 和风险提醒

3. **合规 / 审核人员**
   - 审查回复、文案、资讯摘要
   - 查看风险原因、修改建议、审核记录

4. **内部知识运营人员**
   - 上传知识文档
   - 管理知识库
   - 追踪问答命中效果

---

## 完整商业闭环

这个产品不是四个孤立模块，而是一个闭环：

1. **知识接入**
   - 上传 FAQ、规则文档、产品手册、运营规范、公告数据

2. **智能问答**
   - 用户提问
   - 系统识别意图与场景
   - 选择知识源并生成答案

3. **风险审查**
   - 对回复草稿或营销文案执行合规检查
   - 高风险内容自动拦截或标记

4. **人工协作**
   - 客服 / 审核员查看 AI 结果
   - 人工确认、修改、通过、驳回

5. **反馈沉淀**
   - 命中失败的问题回流到知识库
   - 审核结论回流到审核策略
   - 高频问题用于优化 Prompt、检索和路由

这才是完整产品，而不是“一个聊天框 + 一个模型 API”。

## 产品能力矩阵

### 一、金融客服 Copilot

#### 不是普通客服机器人，而是：

- 自动识别用户问题属于哪类业务
- 从 FAQ、规则、文档中检索依据
- 生成“回复草稿 + 证据链 + 风险提示”
- 对高风险问题提示人工接管

#### 典型问题

- 开户条件是什么？
- 融资融券的门槛是什么？
- 期权交易开通需要满足哪些条件？
- 为什么我今天不能下单？
- 这个产品适合什么风险等级的人？

#### AI 核心价值

- 意图理解
- 检索增强回答
- 多轮追问补全
- 引用依据
- 风险兜底

---

### 二、合规审查 Copilot

#### 审核对象

- 客服回复
- 营销文案
- 资讯摘要
- 产品介绍
- 用户触达消息

#### 审核结果不只输出“通过/不通过”，还要输出：

- 风险等级
- 风险类型
- 问题句子定位
- 命中规则依据
- 修改建议
- 可发布版本草稿

#### AI 核心价值

- 语义理解而非纯关键词匹配
- 规则与上下文联合判断
- 自动生成修正建议

---

### 三、公告与财报智能解读

#### 不是泛泛“投研助手”，而是：

- 上传公告 / 财报 / 新闻
- 自动抽取核心事件
- 生成结构化事件卡片
- 输出影响点、风险点、待确认事项
- 支持多份文档对比

#### 可落地能力

- 事件抽取
- 指标变化摘要
- 时间线整理
- 多文档对比
- 风险提示生成

#### AI 核心价值

- 长文档理解
- 多文档综合
- 结构化抽取
- 解释型摘要

---

### 四、知识与规则决策引擎

#### 它不是单独暴露给用户的模块，而是整个平台的大脑底座：

- 管理 FAQ、规则、制度、产品说明、公告数据
- 为客服、审核、公告解读提供统一知识入口
- 支持多知识源路由
- 支持按角色、业务线、产品线做权限隔离

#### AI 核心价值

- 统一知识表示
- 统一检索能力
- 支持不同业务 Agent 的协同决策

## 系统工作流设计

### 1. 用户问答主链路

```text
用户问题
-> 意图识别
-> 场景路由
-> 检索 FAQ / 规则 / 文档 / 公告
-> 生成带依据的答案
-> 合规检查
-> 返回最终结果
```

### 2. 客服草稿生成链路

```text
客服输入问题
-> Copilot 识别业务意图
-> 命中知识源
-> 生成回复草稿
-> 给出证据链和风险说明
-> 人工确认 / 修改
```

### 3. 合规审核链路

```text
输入待审核文本
-> 风险分类
-> 命中规则与案例
-> 标注问题片段
-> 生成修改建议
-> 输出审核报告
```

### 4. 公告解读链路

```text
上传公告 / 财报 / 新闻
-> 文档解析
-> 事件与指标抽取
-> 生成摘要
-> 输出风险点 / 影响点 / 时间线
-> 支持追问与对比
```

## 三、Agent 与工作流

### 7. 金融 Copilot Agent 设计

#### 知识点

1. 不追求万能 Agent
   - 聚焦明确业务链路
   - 保证可控和可解释

2. 节点设计
   - router
   - retrieve
   - draft
   - compliance_check
   - human_review
   - finalize

3. Agent 的价值
   - 让多个能力形成闭环
   - 让系统有条件判断和状态流转
   - 不是让模型自由发挥

---

### 8. Human-in-the-loop

#### 知识点

1. 什么情况下必须人工介入
   - 高风险回复
   - 审核不通过
   - 证据不足
   - 多知识源冲突

2. 工作台设计
   - 审核员看到什么
   - 客服看到什么
   - 如何记录修改痕迹

3. 为什么这一层很重要
   - 它决定项目是否像企业产品
   - 纯自动化在金融场景通常不成立

## 验收标准

1. **不是** 只做一个聊天机器人，而是做出一套金融 AI 工作台
2. **实现** 问题理解、知识检索、依据生成、合规审核、人工协作的闭环
3. **支持** 至少三个高价值场景：
   - 客服 Copilot
   - 合规审查
   - 公告 / 财报解读
4. **具备** 明确的 AI 不可替代点，而不是简单 API 拼装
5. **能够** 用业务视角和工程视角同时讲清楚项目价值
6. **形成** 可迭代、可展示、可包装的主项目资产
