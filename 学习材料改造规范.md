# 学习材料改造规范

> 适用范围：`docs/01_python/*.md` 与 `source/01_python/*/`
>
> 目标：把现有材料从“资料堆积”改造成“可执行学习路径”。

---

## 一、改造背景

当前材料的核心问题不是内容太少，而是学习路径断裂：

1. 文档里有知识点，但部分表达不够直白，读起来不够顺。
2. 代码里很多是 `exercises.py`，但缺少完整可运行的参照示例。
3. 文档和代码缺少显式映射，学习时需要自己脑补对应关系。
4. 练习文件承担了过多“首次理解”职责，导致启动成本太高。

因此，后续改造不再采用“先把所有文档看一遍，再统一补代码”的方式，而是采用：

**一节一节滚动优化，一节一节闭环完成。**

---

## 二、总目标

每一节改造完成后，必须同时满足以下 5 个条件：

1. 文档能直接看懂，不靠猜。
2. 有一个完整、可运行、可对照的主示例文件。
3. 有一个练习文件，专门用于手写和巩固。
4. 文档明确写出学习顺序：先看什么、先跑什么、再写什么。
5. 文档内容和代码文件可以一一对应。

只要上述任意一条不成立，这一节就不算真正改造完成。

---

## 三、改造原则

### 1. 一节一闭环

不要全局大修。每次只处理一节，并确保该节文档、示例代码、练习、参考答案同时可用。

### 2. 先建立入口，再补练习

初学阶段最重要的是先有能跑通的示例，再做练习。`exercises.py` 不能作为第一入口。

### 3. 文档优先解决“看不懂”

文档不是资料堆积，而是学习路线。写法必须优先考虑可读性、白话表达和操作顺序。

### 4. 代码优先解决“怎么下手”

示例代码必须完整、可运行、有输出。练习代码必须明确、可执行、不依赖猜测。

### 5. 显式映射，禁止靠脑补

文档章节、示例代码、练习题必须建立明确映射关系。

---

## 四、每节的标准产物

每节固定产出以下 4 类内容：

### 1. 文档文件

路径示例：

```text
docs/01_python/01_basics.md
```

作用：

- 解释知识点
- 给出学习顺序
- 提供代码映射
- 说明哪些必须手写

### 2. 主示例文件

路径示例：

```text
source/01_python/01_basics/demo.py
```

作用：

- 本节第一入口
- 完整可运行
- 覆盖文档主线
- 用来先跑一遍建立感觉

说明：

- 如果当前节已经有类似 `basic.py`、`main.py` 这样的主示例文件，可以先继续使用，不必强制立刻改名。
- 后续有精力时再统一成 `demo.py`。

### 3. 练习文件

路径示例：

```text
source/01_python/01_basics/exercises.py
```

作用：

- 专门用于手写练习
- 只承担巩固职责
- 不承担首次讲解职责

### 4. 参考答案文件

路径示例：

```text
source/01_python/01_basics/reference.py
```

作用：

- 卡住时对照
- 用来验证思路
- 帮助修正练习结果

说明：

- 如果不想公开标准答案，也可以只作为本地文件保留。

## 六、文档结构规范

以后每一节文档统一使用以下结构。

~~~markdown
# XX. 章节标题

## 1. 本节目标
## 2. 本节怎么学
## 3. 先跑一下
## 4. 核心概念
## 5. 代码映射表
## 6. 练习说明
## 7. 本节完成标准
~~~

以下是每个栏目必须承担的职责。

### 1. 本节目标

要求：

- 写 3-5 条即可
- 必须写成“学完后你能做什么”
- 不要写成空泛概述

示例：

- 能写出变量、类型注解和基本输出
- 能使用 `if / elif / else` 处理简单判断
- 能读懂并修改本节示例代码

### 2. 本节怎么学

这是文档里最关键的栏目，必须直接回答以下问题：

1. 先运行哪个文件
2. 再阅读哪几个部分
3. 哪些知识点必须手写
4. 哪些知识点可以先理解再复现
5. 学完这一节的最低通过标准是什么

建议固定写法：

~~~markdown
## 2. 本节怎么学

1. 先运行 `source/01_python/01_basics/basic.py`
2. 再阅读本文第 3.1 到 3.4 小节
3. 把文档中的最小例子自己重新敲一遍
4. 再完成 `exercises.py`
5. 卡住时优先回看示例代码，最后再看 `reference.py`
~~~

### 3. 先跑一下

要求：

- 明确给出运行命令
- 告诉学习者运行后会看到什么
- 说明为什么要先跑这份代码

示例：

```bash
python source/01_python/01_basics/basic.py
```

### 4. 核心概念

每个知识点统一采用以下结构：

~~~markdown
### 4.1 概念名称 📌

**一句话解释**：用最白的话解释这个概念是什么。

**为什么需要**：它解决什么问题，为什么这一节要学它。

**最小例子**：
```python
# 只演示一个知识点
```

**常见坑**：
- 坑 1
- 坑 2

**与 JavaScript 对照**（仅在确实有帮助时写）：
- Python 写法
- JS 对照写法
~~~

要求：

- 先白话解释，再给术语
- 一个代码块只讲一个重点
- 不要一段代码塞进太多新概念
- 对初学阶段常见误区要显式提醒

### 5. 代码映射表

这是强制项。每节都必须有。

推荐格式：

| 文档小节 | 对应示例代码 | 对应练习 | 备注 |
|---|---|---|---|
| 4.1 变量 | `basic.py` 变量示例 | 练习 1 | 必须手写 |
| 4.2 数据类型 | `basic.py` 类型示例 | 练习 1 | 必须手写 |
| 4.3 条件判断 | `basic.py` 成绩判断 | 练习 2 | 建议先模仿 |
| 4.4 循环 | `basic.py` 循环示例 | 练习 3-4 | 重点练习 |

要求：

- 文档写完后必须能列出这张表
- 如果表写不出来，说明这一节还没整理清楚

### 6. 练习说明

必须明确写出：

- 哪几题必须自己手写
- 哪几题可以参考示例后再写
- 推荐完成顺序
- 卡住时应该先看哪里

示例：

~~~markdown
## 6. 练习说明

- 练习 1 和练习 2 必须手写
- 练习 3 可以先参考 `basic.py` 再完成
- 练习 4 是综合题，建议最后做
- 卡住时先回看 `basic.py`，再看 `reference.py`
~~~

### 7. 本节完成标准

要求：

- 用可验证的话描述
- 不要写成“理解了”“掌握了”这种模糊表述

示例：

- 能不看文档写出变量、类型注解、`if / for / while`
- 能解释示例代码每一段在做什么
- 能独立完成本节前 3 题练习

## 七、文档写法规范

这一部分用于解决“文档看不太懂”的问题，后续必须遵守。

### 1. 先说人话，再说术语

每个知识点先写一句白话解释，再补充正式概念说明。

例如：

- 不要直接写“变量是对对象引用的绑定”
- 应先写“变量就是给一个值起名字，后面可以反复使用”

### 2. 一次只讲一个重点

单个段落不要连续引入太多新概念。

错误示例：

- 变量、类型注解、字符串格式化、输入输出、条件判断放在同一段里一起解释

正确做法：

- 一个小节只讲一个重点
- 一个代码块只演示一个重点

### 3. 例子要短、可运行、可模仿

要求：

- 优先给最小例子
- 运行结果要容易观察
- 示例要能直接复制运行

### 4. 常见坑必须显式写出来

初学阶段要优先提醒下面这类问题：

- `input()` 返回的是字符串
- Python 用缩进，不用花括号
- `=` 是赋值，`==` 是比较
- `list.sort()` 会原地修改列表，`sorted()` 会返回新列表

### 5. 抽象句后面要补桥接句

如果一句话比较抽象，后面必须补一句“你可以把它理解成什么”。

示例：

- “字典是键值对结构。”
- “你可以把它理解成一个按名字取值的小型数据表。”

### 6. 删除为了“完整性”而写的内容

标准不是“内容越多越好”，而是“当前这一节是否更容易学了”。

如果某段解释不能直接服务当前学习动作，就应删掉或后移。

## 八、代码文件规范

### 1. 主示例文件规范

主示例文件可以叫 `demo.py`，也可以沿用现有的 `basic.py`、`main.py`。

要求：

1. 必须能直接运行
2. 必须覆盖文档主线
3. 必须有清晰输出
4. 不留 TODO
5. 尽量按知识点分段组织
6. 优先可读性，不追求炫技

建议结构：

```python
"""
01_basics 完整示例

运行方式：
python source/01_python/01_basics/basic.py
"""

# 1. 变量与类型

# 2. 字符串格式化

# 3. 条件判断

# 4. 循环
```

### 2. exercises.py 规范

要求：

1. 只放练习，不承担首次教学职责
2. 每题必须写清题目目标
3. 必要时给出最小提示
4. 前期章节以小题为主，少而稳
5. 练习必须能独立执行，不依赖猜测

前期章节推荐配置：

- 3-5 个非常明确的小练习
- 1 个综合小题

不建议：

- 一开始就布置太多开放题
- 让练习文件成为这一节唯一入口

### 3. reference.py 规范

要求：

1. 与 `exercises.py` 题号一一对应
2. 写法尽量直白
3. 优先和主示例代码风格一致
4. 以“能看懂、能对照、能修正”为目标

---

## 九、代码内容写法规范

### 1. 优先可读性，不优先高级写法

初学阶段示例代码首先要“看得懂”，不是“写得高级”。

### 2. 变量名必须直白

优先使用：

- `name`
- `age`
- `score`
- `student_scores`
- `file_path`

尽量避免：

- `x`
- `y`
- `tmp`
- `data2`

### 3. 输出必须可观察

学习示例要尽量通过 `print()` 把关键结果打出来，让运行行为可见。

### 4. 适量加短注释

可以加非常短的结构性注释，但不要写解释废话的注释。

正确例子：

```python
# 计算所有偶数的和
```

不推荐：

```python
# 给变量赋值
```

### 5. 一个逻辑块解决一个问题

不要把多个目标揉成一团。特别是教学代码，要尽量一段只演示一类行为。

## 十、文档与代码的一一对应规范

这一条是强制执行项。

每一节都必须做到：

1. 文档中的每个核心概念，都能在主示例文件中找到对应代码
2. 每个练习题，都能追溯到对应的文档小节或示例片段
3. 文档里必须显式给出映射表

推荐最小映射模板：

| 文档小节 | 示例代码 | 练习 |
|---|---|---|
| 4.1 变量 | `demo.py` 变量示例 | 练习 1 |
| 4.2 数据类型 | `demo.py` 类型示例 | 练习 1 |
| 4.3 条件判断 | `demo.py` 分数判断 | 练习 2 |
| 4.4 循环 | `demo.py` `for/while` 示例 | 练习 3 |

如果某个知识点在文档中讲了，但代码中没有落点，必须补代码或删文档。

## 十一、每次只改一节时的执行流程

以后每次运行，统一按下面顺序执行：

### 第 1 步：检查现有文档

重点找：

- 哪些解释不够白话
- 哪些段落一次讲了太多东西
- 哪些地方缺学习顺序说明
- 哪些地方缺和代码的对应关系

### 第 2 步：重写文档结构

优先补这 3 个部分：

1. `本节怎么学`
2. `先跑一下`
3. `代码映射表`

这三项没补齐前，不进入大规模练习设计。

### 第 3 步：补主示例文件

要求：

- 完整可运行
- 覆盖本节主线
- 输出清晰

### 第 4 步：整理 exercises.py

把练习改造成：

- 更明确
- 更分层
- 更容易下手

### 第 5 步：补 reference.py

确保：

- 题号能一一对上
- 答案写法直白
- 能作为卡住时的对照文件

### 第 6 步：做闭环检查

检查这一节是否已经满足本规范第二章中的 5 个条件。

## 十二、每节完成后的验收标准

一节完成后，必须能清楚回答以下 6 个问题：

1. 我这一节先看哪个文档部分？
2. 我这一节先跑哪个文件？
3. 我这一节必须手写哪些内容？
4. 我卡住时应该先回看哪里？
5. 文档每个重点对应哪段代码？
6. 学完这一节后，我最低应该会什么？

如果其中任意一个问题答不上来，这一节就还没有真正闭环。

## 十四、现阶段的落地标准

每节先不要追求“完美课程化”，先达到以下最低标准：

1. 文档中每个核心概念都有一句更白的话解释
2. 至少有 1 个完整可运行文件
3. `exercises.py` 可以独立完成，不依赖猜测
4. 文档明确写出学习顺序
5. 这一节结束后，知道自己到底要手写哪几个点

达到这个标准，材料就从“资料堆积”变成“可执行学习路径”。

## 十六、执行约束

后续实际改造时，统一遵守以下约束：

1. 每次只改一节，不跨节扩散。
2. 先改文档结构，再补示例代码，再调练习。
3. 不为追求完整而堆内容，优先降低理解成本。
4. 不让 `exercises.py` 承担首次讲解职责。
5. 不允许文档和代码脱节。

## 十七、最终判断标准

这套改造规范是否有效，不看内容多少，只看下面这件事是否成立：

**当我打开任意一节时，我能立刻知道：先跑哪个文件、先看哪部分、先写哪几题、卡住时回看哪里。**

如果能做到这一点，这套材料就已经从“信息集合”变成“学习系统”。
