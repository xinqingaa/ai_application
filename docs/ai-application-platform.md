# 企业级 AI 应用平台能力地图

这份文档用于沉淀本仓库对企业级 AI 应用平台的长期理解和远期方向。

它不是当前阶段的交付清单，也不是需求评审 RAG 助手的验收标准。现阶段以 [strategy.md](strategy.md) 中的课程路线和主项目闭环为准；本文只用于避免视野过窄，并在项目后续平台化时提供参考。

## 1. 核心判断

企业级 AI 应用不是聊天机器人，也不是单个 RAG Demo。

真正有价值的 AI 应用平台，需要把模型、知识库、工作流、智能体、工具生态、评测、安全和运营整合成一套稳定、可配置、可管理、可追踪、可持续优化的系统。

前端在这里不是普通展示层，而是 AI 应用平台的操作界面和控制台。它要让用户能理解 AI 在做什么、依据是什么、风险在哪里、什么时候需要人工介入、如何配置能力、如何追踪失败、如何持续优化。

## 2. 企业级项目架构

企业级 AI 应用需要先具备完整项目架构，而不是写完几个页面就结束。

需要考虑：

- 多环境配置：local、dev、staging、prod。
- 权限体系：用户、角色、资源、工具、知识库、审批权限。
- 菜单路由：面向知识库、智能体、工作流、评测、运行记录、系统配置。
- API 规范：普通 HTTP、SSE、stream event、workflow event、tool event。
- 状态管理：会话状态、任务状态、工具状态、工作流状态、审核状态。
- 异常处理：模型失败、检索失败、工具失败、流中断、权限失败、超时。
- 日志监控：请求日志、模型调用日志、工具调用日志、工作流日志、审计日志。
- 组件复用：对话、来源、风险、工具轨迹、审批、评测、知识库管理等组件。
- 多人协作：多人开发、多人审核、多人知识运营。
- 模块拆分：chat、rag、agent、workflow、eval、admin、infra。
- 灰度发布：Prompt、Retriever、Agent、Workflow、Model 配置都需要版本。
- 持续迭代：依赖评估、bad case、用户反馈和运行指标回流。

这部分能力不要求一开始全部实现。当前阶段只需要保留基础边界意识，避免后续无法扩展。

## 当前阶段使用方式

当前主项目是需求评审 RAG 助手。本文中的能力只按需取用，不要求完整实现。

当前阶段优先参考：

- 企业级 RAG 知识库中的文档生命周期、chunk、metadata、检索测试、引用溯源和知识回流。
- Agent 运行过程可视化中的工具轨迹、任务状态、人工介入和失败原因展示。
- 数据标注与模型评测中的 golden set、引用评估、拒答评估、bad case 和反馈回流。
- 企业级项目架构中的 API 分层、事件流、基础日志、成本延迟和 Docker Compose 本地运行。

当前阶段暂不作为重点：

- 完整多租户体系。
- 完整智能体管理平台。
- 完整 MCP 工具生态平台。
- 可视化低代码工作流平台。
- 企业级安全运营、告警、灰度和权限中台。

这些能力可以作为远期方向保留，但不应压迫当前学习节奏。

## 3. 可视化工作流编排

可视化工作流不是画几个节点，而是 AI 流程设计、校验、发布、运行和调试系统。

核心模块包括：

- 节点类型：模型调用、知识库检索、工具执行、条件判断、人工审核、结果输出。
- 节点配置：模型、Prompt、输入字段、输出 schema、工具参数、重试策略。
- 连线规则：哪些节点能连接，数据如何传递，分支条件如何定义。
- 参数校验：节点参数、schema、工具权限、知识库绑定是否完整。
- 版本管理：草稿、发布版本、回滚、复制、对比。
- 流程发布：从编辑态进入可执行态。
- 运行调试：单节点调试、全流程运行、模拟输入。
- 执行日志：每个节点输入、输出、耗时、错误。
- 失败节点重试：失败原因可见，允许从失败节点重跑。
- 人工介入：中断、审批、修改、恢复。

这类能力本质上是在做一个面向 AI 的低代码流程设计器。它和普通流程图的区别在于：节点背后连接模型、知识库、工具、Agent 和人工审核，运行过程必须可追踪和可恢复。

## 4. 企业级 RAG 知识库

企业级 RAG 前端不是文件上传和问答，而是知识库运营系统。

核心模块包括：

- 知识库创建：不同业务、部门、场景可以有不同知识库。
- 文档上传：PDF、Word、Markdown、PPT、图片、表格、网页等。
- 文档分类：PRD、接口文档、规则、FAQ、公告、会议纪要、历史评审。
- 解析进度：上传、排队、解析、切片、向量化、索引完成、失败。
- 切片管理：查看 chunk、标题、上下文、metadata、来源页码。
- metadata 管理：source、doc_type、section、version、owner、permission。
- 权限隔离：不同用户、部门、角色看到不同知识。
- 检索测试：输入问题，查看召回片段、score、rerank、过滤条件。
- 引用溯源：答案能定位到文档、页码、章节、原文片段。
- 版本治理：旧版本、废弃文档、增量更新、删除一致性。
- 知识回流：bad case、用户反馈、人工修订进入知识运营流程。

对企业项目来说，可追溯性和可治理性往往比单纯生成答案更重要。

## 5. 智能体管理平台

企业中的智能体不是一个聊天窗口，而是一组可以统一创建、配置、发布和管理的数字员工。

核心模块包括：

- Agent 创建：定义名称、角色、场景、能力边界。
- 模型配置：模型供应商、模型名、温度、最大 token、成本策略。
- Prompt 管理：system prompt、任务 prompt、版本、变量、示例。
- 知识库绑定：选择知识库、检索策略、权限过滤。
- 工具授权：绑定可用工具、限制高风险工具、设置人工确认。
- 参数设置：最大步数、超时、重试、检索 top-k、成本预算。
- 版本发布：草稿、发布、回滚、灰度。
- 运行记录：每次运行输入、输出、工具调用、耗时、成本、错误。
- 部门级智能体：客服助手、销售助手、合同审核助手、数据分析助手、知识运营助手。

智能体管理平台的关键不是“配置项越多越好”，而是让 Agent 的能力、边界、版本、工具和运行风险可见、可控、可审计。

## 6. MCP 与工具生态

智能体需要调用数据库、搜索、文件系统、企业接口和第三方服务。工具生态不能只停留在后端注册函数，还需要前端完成配置、测试、授权和审计。

核心模块包括：

- 工具目录：展示所有可用工具、描述、输入 schema、权限范围。
- 工具接入：配置 MCP server、HTTP API、数据库、搜索、文件系统、企业接口。
- 参数配置：根据 JSON Schema / UI Schema 渲染工具配置表单。
- 权限控制：哪些 Agent、用户、角色可以调用哪些工具。
- 连接测试：验证工具是否可用、凭证是否正确、返回格式是否符合预期。
- 调用记录：记录工具输入、输出、耗时、错误、调用者、Agent run id。
- 异常状态：连接失败、参数错误、权限不足、超时、服务不可用。
- 高风险操作确认：删除、修改、下单、发布、外部发送等操作必须人工确认。
- 操作审计：关键工具调用需要保留审计记录。

MCP 和工具生态的核心是标准化连接能力，但平台必须同时处理权限、确认、审计和失败可见性。

## 7. Agent 运行过程可视化

企业用户不仅关心最终结果，还需要知道 AI 任务执行到了哪一步、调用了什么工具、为什么失败、能否重试、是否需要自己介入。

核心模块包括：

- 任务状态：queued、running、waiting_review、failed、completed、cancelled。
- 执行步骤：计划、检索、工具调用、分析、生成、审核、输出。
- 工具调用：工具名、输入、输出、耗时、状态、错误。
- 运行日志：节点日志、模型日志、工具日志、人工操作日志。
- 错误节点：失败发生在哪一步，失败原因是什么。
- 重试机制：单节点重试、从 checkpoint 恢复、整任务重跑。
- 耗时统计：总耗时、节点耗时、工具耗时、模型耗时。
- 成本统计：token、模型调用次数、工具调用次数。
- Agent trace：完整轨迹可折叠展示，不把内部噪声直接暴露给普通用户。
- Workflow runtime：图结构、当前节点、历史路径、分支结果。
- 用户可干预节点：人工确认、修改草稿、补充材料、取消任务。

这是 AI 前端和普通后台最大的差异之一：普通后台展示业务数据，AI 前端还要展示智能系统的执行过程和不确定状态。

## 8. 数据标注与模型评测

企业 AI 应用上线后，需要持续判断模型回答是否准确、引用是否正确、工具调用是否合理、工作流是否稳定。

核心模块包括：

- 问答数据管理：保存用户问题、答案、来源、模型、成本、反馈。
- 人工标注：准确性、完整性、引用质量、拒答合理性、风险提示。
- 结果审核：审核 AI 输出是否可用，是否需要修改。
- 模型对比：不同模型、Prompt、Retriever、Agent 版本对比。
- 评分统计：通过率、引用正确率、拒答准确率、工具轨迹正确率。
- 问题回流：bad case 回流到知识库、Prompt、Retriever、Tool、Graph 或人工流程。
- RAG Eval：检索命中、生成质量、引用正确、拒答正确。
- Agent Trajectory Eval：工具选择、参数、顺序、禁止工具调用、最大步数。
- Workflow Eval：节点路径、分支命中、人工确认、恢复成功率。

评测和标注不是后期补充，而是企业级 AI 应用能否持续变好的关键模块。

## 9. 企业级安全与运营

企业级 AI 应用不只要能运行，还要安全、稳定、可控制、可追踪、可预算。

核心模块包括：

- 用户权限：用户、角色、部门、资源权限。
- 数据隔离：不同用户、部门、知识库、租户之间隔离。
- 敏感信息脱敏：输入、输出、日志、trace 中的敏感信息处理。
- 操作审计：知识库修改、工具调用、审批操作、Agent 发布。
- 调用限流：用户级、Agent 级、工具级、模型级限流。
- Token 统计：输入、输出、缓存、总成本。
- 成本分析：按用户、部门、Agent、任务类型统计。
- 告警监控：失败率、成本异常、工具异常、模型异常。
- 工具权限：高风险工具必须限制调用范围。
- 高风险审批：涉及金融、合规、发布、删除、外部发送的操作必须确认。
- 日志留存：运行记录和审计记录需要可追溯。

这部分能力决定项目是否像真实企业系统，而不只是学习 Demo。

## 11. 与主项目的关系

当前主项目是需求评审 RAG 助手。

- 重点验证企业知识库、需求文档接入、RAG 问答、引用溯源、拒答、单 Agent 追问、基础评估和最小工作台。
- 项目围绕 PRD、业务规则、接口文档、会议纪要和历史评审记录展开，不再并行维护多个主项目方向。

项目设计应遵循：先做可运行闭环，再逐步增加平台化能力；不要为了完整平台而牺牲阶段可展示成果。
