## 课程定位

- 本课程不是继续堆概念，而是把前面的能力组合成一个完整产品。
- 本课程的主项目不是普通聊天机器人，而是一个 **金融客服与合规 Copilot**。
- 这套系统要解决的不是“能不能回答”，而是：
  - 能不能理解真实意图
  - 能不能给出依据
  - 能不能识别风险
  - 能不能让人机协作形成闭环
- 课程会吸收 `llmops.md` 中有价值的工程化能力，例如：
  - 多模型接入
  - 知识库与记忆
  - 任务调度
  - 权限与审计
  - 部署、监控、成本控制
- 本课程会沉淀一批可复用的基础能力，但依然以业务闭环为主线，不把项目直接做成通用工作流平台或完整企业中台。

- `05_agent` 解决的是动态决策、工具调用、状态编排、记忆、guardrails、评测。
- 本课程不是再讲一次 Agent 原理，而是把这些能力装进真实业务流程：路由、审核、审批、人工接管、多角色协作。
- 可以理解成：`05_agent` 给的是“能力模块”，这门课做的是“业务闭环”。

- 这里更强调：先在真实金融场景里把链路跑通，再从中识别哪些部分值得后续抽象成通用层。

- 但它不追求多业务线、多租户、多团队共享的完整企业中台设计，也不追求组织级治理深度。
- 当前目标仍然是做出一个业务价值明确、工程闭环完整、能够展示你能力的主项目。

## 什么叫 AI 不可替代

这个产品的 AI 价值，不是“把客服系统接上 LLM”。

真正的 AI 价值体现在：

1. **理解用户真实意图**
   - 同一句话可能同时涉及费用、规则、产品适当性、异常申诉

2. **跨知识源综合判断**
   - FAQ、规则、产品说明、运营规范、公告、历史问答可能要一起参考

3. **生成带依据的结果**
   - 不只是结论，还要有来源、依据摘要、风险提示

4. **执行语义级合规审核**
   - 识别夸大收益、风险提示缺失、表述不严谨、事实依据不足

5. **形成多步工作流**
   - 先路由，再检索，再生成，再审核，再人工确认

如果一个功能只是“查数据库然后返回”，那不是本课程的 AI 核心能力。

4. **知识运营人员**

## 完整商业闭环

### 这一闭环如何对应前面课程

- **知识接入**：主要承接 `04_rag` 的文档处理、索引构建、知识治理。
- **智能问答**：主要承接 `02_llm` 的消息与输出控制，以及 `04_rag` 的检索与引用回答。
- **风险审查**：主要承接 `02_llm` 的结构化输出，以及 `05_agent` 的 guardrails、审核与人工协作思路。
- **人工协作**：主要承接 `05_agent` 的 Human-in-the-loop、状态流转和审批逻辑。
- **反馈沉淀**：主要承接 `04_rag` 的效果回归和 `05_agent` 的评估闭环。

这套系统不是几个平铺功能，而是一个完整闭环：

1. **知识接入**
   - FAQ、规则文档、产品说明、运营规范、公告资讯接入系统

2. **智能问答**
   - 用户提问后，系统先识别意图，再选择知识源，再生成带依据的答案

3. **风险审查**
   - 对回复草稿、营销文案、资讯摘要进行合规检查

4. **人工协作**
   - 客服 / 审核员确认、修改、通过或驳回 AI 结果

5. **反馈沉淀**
   - 命中失败的问题回流知识库
   - 审核结论回流审核策略
   - 高频问题回流 Prompt、路由和检索优化

## 产品能力矩阵

### 这一部分怎么看

- 前三个模块更偏 **业务场景层**：它们直接对应真实用户价值和展示价值。
- 第四个“知识与规则决策引擎”更偏 **通用能力底座**：它本身不是主要卖点，但决定前三个模块能否复用知识、统一路由、统一治理。

### 1. 金融客服 Copilot

#### 能力目标

- 自动识别问题属于开户、交易规则、手续费、产品适当性、异常申诉等哪个场景
- 从 FAQ、规则和文档中检索依据
- 生成“回复草稿 + 证据链 + 风险提示”
- 对高风险问题提示人工接管

#### 典型问题

- 开户需要哪些条件？
- 融资融券门槛是什么？
- 期权开通需要满足哪些条件？
- 为什么我今天不能下单？
- 这个产品适合什么风险等级的用户？

#### AI 核心价值

- 意图理解
- 检索增强回答
- 多轮追问补全
- 引用依据
- 风险兜底

---

### 2. 合规审查 Copilot

#### 审核对象

- 客服回复
- 营销文案
- 产品介绍
- 资讯摘要
- 用户触达消息

#### 输出结果

- 风险等级
- 风险类型
- 问题句子定位
- 命中规则依据
- 修改建议
- 可发布版本草稿

#### AI 核心价值

- 语义理解而非纯关键词匹配
- 上下文和规则联合判断
- 自动生成可执行的修订建议

---

### 3. 公告与财报智能解读

#### 能力目标

- 上传公告 / 财报 / 新闻
- 自动抽取核心事件和关键指标
- 生成结构化事件卡片
- 输出影响点、风险点、待确认事项
- 支持多份文档对比和追问

#### AI 核心价值

- 长文档理解
- 多文档综合
- 结构化抽取
- 解释型摘要

---

### 4. 知识与规则决策引擎

#### 它不是独立卖点，而是整个 Copilot 的底座

- 管理 FAQ、规则、制度、产品说明、公告数据
- 为客服、审核、公告解读提供统一知识入口
- 支持多知识源路由
- 支持按角色、业务线、产品线做权限隔离

#### AI 核心价值

- 统一知识表示
- 统一检索能力
- 支持不同业务 Agent 的协同决策

### 10. 上下文管理与记忆策略

#### 本节与前面课程的关系

- 承接 `02_llm` 的消息管理与上下文窗口认知。
- 承接 `05_agent` 的 Context Engineering 和记忆机制，同时把这些能力真正放进多角色业务场景。

#### 本节的通用能力

- Context Manager
- 多层记忆设计

#### 本节的业务特性

- 金融场景对上下文污染、敏感信息隔离、可审计性要求更高

#### 知识点

1. **为什么不能只靠原始聊天历史**
   - Token 成本会持续上升
   - 无关上下文会污染回答
   - 金融场景需要更强的可控性与可审计性

2. **三层记忆设计**
   - 会话记忆：当前轮次与最近若干轮对话
   - 用户记忆：用户角色、风险偏好、常见问题、关注产品

   - 滑动窗口
   - 摘要压缩
   - 按场景选择上下文
   - 高风险问题禁止带入无关历史

4. **记忆写入策略**
   - 不是每轮都写入长期记忆
   - 只有高价值信息进入用户记忆或任务记忆
   - 敏感信息需要脱敏、隔离与审计

5. **记忆更新与失效机制**
   - 会话记忆随会话窗口自然淘汰
   - 用户记忆需要区分长期偏好与短期关注点
   - 任务记忆应随任务完成、关闭或过期而清理
   - 记忆不是越多越好，过期和错误记忆要允许被覆盖或删除

### 16. 工作流分层设计

#### 本节与前面课程的关系

- 承接 `05_agent` 里的 Workflow、LangGraph、Human-in-the-loop。
- 也承接 `04_rag` 的文档索引异步流，把 AI 主链路和系统任务流真正分层。

#### 本节的通用能力

- 同步流 / 异步流 / 人工审核流分层
- 状态设计
- 失败与兜底策略

#### 本节的业务特性

- 金融回复审核和知识构建天然要求不同的时延、审计和责任边界
- 多知识源冲突升级人工也是非常业务化的策略

#### 知识点

1. **为什么要分层**
   - 同步聊天链路和异步处理链路的约束完全不同
   - 业务工作流、系统任务流、人工审核流不能混成一条链
   - 分层后更容易观测、重试、审计和演示

2. **同步工作流**
   - 用户提问 -> 路由 -> 检索 -> 草稿 -> 合规检查 -> 输出 / 人工接管
   - 适用于客服问答、规则咨询、截图问答

3. **异步工作流**
   - 上传 -> 入库 -> 解析 -> 切分 -> 向量化 -> 建索引 -> 完成 / 失败
   - 适用于知识库构建、批量审核、公告解读批处理

4. **人工审核流**
   - AI 草稿 -> 风险提示 -> 人工修改 -> 审核通过 / 驳回 -> 结果存档
   - 强调修改痕迹、责任归属、审计日志

5. **关键状态设计**
   - pending
   - running
   - waiting_human
   - passed
   - failed
   - cancelled

6. **失败与兜底策略**
   - 检索为空时返回可解释兜底
   - 多知识源冲突时升级人工
   - 模型失败时重试、降级或切换模型

#### 实践练习

```text
定义三类工作流：
1. 客服问答同步流
2. 文档索引异步流
3. 合规审核人工协作流

要求输出：
- 节点
- 状态
```

### 17. Human-in-the-loop

#### 本节与前面课程的关系

- 直接承接 `05_agent` 的 Human-in-the-loop 和 Guardrails。
- 这也是本课程区别于“只做一个聊天机器人”的关键点，因为真实金融业务很难完全自动化。

#### 本节的通用能力

- 审批流
- 人工修改与确认
- 修改痕迹记录

#### 本节的业务特性

- 高风险回复、证据不足、多知识源冲突在金融场景下都更容易触发人工介入
- 审核员和客服之间的协同方式本身就是业务流程的一部分

#### 知识点

1. **什么情况下必须人工介入**
   - 高风险回复
   - 审核不通过
   - 证据不足
   - 多知识源冲突

2. **工作台设计**
   - 审核员看到什么
   - 客服看到什么
   - 如何记录修改痕迹

3. **为什么这一层很重要**
   - 它决定项目是否像企业产品
   - 纯自动化在金融场景通常不成立

#### 实践练习

```text
设计人工确认流程：
草稿 -> 风险提示 -> 人工修改 -> 审核通过 -> 发布
```

## 验收标准

1. **不是** 只做一个聊天机器人，而是做出一套金融 AI 工作台
2. **实现** 问题理解、知识检索、依据生成、合规审核、人工协作的闭环
3. **支持** 至少三个高价值场景：
   - 客服 Copilot
   - 合规审查
   - 公告 / 财报解读
4. **具备** 明确的 AI 不可替代点，而不是简单 API 拼装
5. **具备** 明确的工作流状态流转、失败兜底与人工接管机制
6. **具备** 记忆与上下文管理能力，而不是简单堆叠聊天历史
7. **具备** 至少一种贴近金融场景的多模态能力：
   - PDF / 页面截图理解
   - 财报表格抽取
   - 图文联合问答
8. **打通** 前端、后端、LLM、RAG、Agent、数据库、缓存、部署的完整链路
9. **能够** 用业务视角和工程视角同时讲清楚项目价值
10. **形成** 可迭代、可展示、可包装的主项目资产
