# Learning Guide

这份文档是当前仓库学习设计、课程设计、项目设计、写作规范和代码组织规范的唯一真源。

## 1. 学习组织方式

当前学习以真实问题和项目闭环组织：

```text
真实问题
-> 基础原理
-> 最小实现
-> 主流框架实现
-> 失败分析与能力边界
-> 小项目实战
```

学习目标不是机械完成章节，而是形成可解释、可运行、可展示、可评估的 AI 应用能力。

## 2. 为什么这样组织

Python 和 LLM API 适合章节化学习，也适合用脚本快速获得反馈。

RAG、Agent 和 Workflow 更接近系统工程。如果仍然要求每个知识点都对应一组脚本，容易把学习变成“完成脚本”，而不是理解问题、设计链路和完成项目。

后续学习更关注：

- 真实问题是什么。
- 为什么需要这个能力。
- 最小实现如何验证原理。
- 主流框架如何封装同一件事。
- 能力边界和失败模式在哪里。
- 如何收敛到一个可运行的小项目。

## 3. 课程编排原则

保留已有基础阶段：

- `01_python`
- `02_llm`
- `03_foundation`
- `04_rag`

这些阶段中部分已有文档和代码属于早期课程式结构产物，后续新内容以当前规范为准。

从 RAG 开始，课程编排不再强制遵循“完整学完一个大阶段才能进入下一个大阶段”的线性方式。

RAG、Tool Calling、Agent、Workflow、Evaluation、Observability 和 Frontend Workbench 可以围绕项目需要交叉出现。

课程目录可以继续按阶段展开，例如后续保留 `05_agent`、`06_*`、`07_*` 等，但每个阶段都应该服务 AI 应用主链路，而不是孤立堆知识点。

## 4. RAG 与 Agent 以后后续课程的关系

RAG 和 Agent 不应该被理解成完全割裂的两门课。

更合理的关系是：

- 固定 RAG 是知识注入和依据生成的基础能力。
- Tool Calling 是模型连接外部能力的基础机制。
- Agentic RAG 是在固定 RAG 不够时，引入查询改写、动态检索、多知识源路由和质量判断。
- LangGraph / Workflow 用于复杂状态流转、分支、循环和人机协作。
- Evaluation 和 Observability 贯穿所有 AI 应用阶段。

学习顺序应该服务项目闭环：

```text
先跑通固定链路
-> 再按问题引入工具调用
-> 再按复杂度引入 Agent / Workflow
-> 始终保留评估、观测和前端可视化
```

## 5. 专题文档模板

后续新增专题文档时，优先使用下面的结构。

### 真实问题

说明遇到什么问题，为什么需要学习这个能力。

### 基础原理

用最少必要概念讲清楚核心机制。

### 最小实现

用尽量小的实现验证原理。不是每个专题都必须写大量脚本，但如果有代码，必须能说明入口、运行方式和观察重点。

### 主流框架实现

说明主流框架如何封装同一件事，例如 OpenAI SDK、LangChain、LangGraph、MCP 或其他工具。

### 失败分析与能力边界

说明常见失败、误区、不适用场景和需要人工介入的情况。

### 小项目实战

说明这个能力如何进入阶段项目，并和其他能力组成完整链路。

### 完成标准

用可验证动词描述完成标准，例如：

- 能解释
- 能运行
- 能实现
- 能判断
- 能改造
- 能在项目中使用

## 6. 项目文档模板

后续新增项目文档时，优先使用下面的结构。

### 项目定位

说明项目解决什么真实问题，不解决什么问题。

### 面向角色

说明项目服务哪些用户或协作角色。

### 业务主链路

用文本链路描述输入、处理、输出和反馈。

### V0 最小闭环

说明最小可运行版本包含什么、不包含什么，以及如何判断完成。

### V1 能力增强

说明在 V0 基础上新增哪些能力，为什么新增。

### V2 产品化完善

说明前端体验、评估、观测、部署和演示能力如何完善。

### 技术模块

说明各技术模块的职责边界。

### 失败边界

说明项目不能解决什么、依赖什么前提、什么时候需要人工介入。

### 完成标准

说明项目到什么程度算完成，而不是只写“实现某功能”。

## 7. 文档写作规范

- 文档先于代码。
- 问题先于概念。
- 最小闭环先于工程完整。
- 项目收敛先于章节堆叠。
- 不写迁移过程、历史流水账和过渡状态。
- 不维护多个重复真源。
- 不把实时学习进度写入长期文档。
- 不为了完整性堆概念。
- 不为了每个章节都造脚本。

文档至少要回答：

- 为什么学。
- 解决什么真实问题。
- 学到什么程度。
- 最小可验证结果是什么。
- 什么时候不该用。
- 如何进入项目实战。

## 8. 代码组织规范

- 示例代码服务理解。
- 项目代码服务闭环。
- 不强制每个知识点都有脚本。
- 新代码优先围绕项目阶段组织。
- 可以按 `V0 / V1 / V2` 多包逐步推进。
- 每个 package 必须有清晰职责、README、运行入口和完成标准。
- 涉及真实 API、密钥、模型配置时必须隔离。
- 涉及 RAG、Agent、评估和观测时，优先保留可调试输出。

示例代码可以简单，但必须能解释。

项目代码可以逐步工程化，但必须能运行。

## 9. 完成标准

一个专题完成，至少满足：

- 能解释真实问题。
- 能说明基础原理。
- 有最小实现，或明确说明为什么不需要代码。
- 能说明主流框架如何解决同类问题。
- 能说明失败边界。
- 能进入项目链路。

一个项目阶段完成，至少满足：

- 有可运行入口。
- 有业务输入输出。
- 有基础评估或验证方式。
- 能展示来源、过程或结果。
- 能讲清楚价值和边界。
