# Learning Guide

这份文档是当前仓库学习设计、课程设计、项目设计、写作规范和代码组织规范的唯一真源。

## 1. 学习组织方式

当前学习以真实问题和项目闭环组织。**学习认知链路**（全课通用）如下；单节正文在此之上叠加交付约定（本节实战、运行与观察等），见 §6.4、§7。

```text
真实问题
-> 基础原理
-> 最小实现
-> 主流框架实现
-> 失败分析与能力边界
```

学习目标不是机械完成章节，而是形成可解释、可运行、可展示、可评估的 AI 应用能力。

在 AI Coding 参与下，课程完成不以“代码由谁敲出”判断，而以是否真正拥有代码所有权判断。具体标准见 [ai-coding-mastery.md](ai-coding-mastery.md)。

## 2. 学习方式与项目驱动原则

本仓库的学习方式，是按**一个确定的业务项目**开展系统性的 AI 应用开发学习，而不是按孤立章节刷脚本。

### 唯一主项目

当前唯一长期主项目是「**需求评审助手**」：从 RAG 知识助手，逐步演进为具备多 Agent 协作、Workflow、人工确认、评估观测和 AI Native 工作台的完整评审系统。

通过这个项目，系统性掌握 LLM、RAG、Agent、Multi-Agent、Workflow、Eval、Observability 和 AI Native 工程能力。

### 能力路线与项目版本（两条轴，不混为一谈）

| 维度 | 回答的问题 | 载体 |
| --- | --- | --- |
| **能力路线** | 按什么主题、什么深度学 AI 应用能力 | `02_llm`–`07_projects` 课程分层；见 [strategy.md](strategy.md) 七层能力 |
| **项目版本** | 需求评审助手**产品**演进到哪一步 | V0–V6；见 [course/07_projects/outline.md](../course/07_projects/outline.md) |

二者**相关但不一一对应**：学 `03_rag` 是在建知识系统能力，不等于「做完 V2」；V3 可能同时用到 RAG 课与 Agent 课的能力。文档中避免把「第 X 门课」与「VX 版本」写成等同关系。

### 文档 + 代码

- **文档最重要**：定义真实问题、原理、边界与完成标准；当节可读、可学，不假设读者已学后续章节。
- **文档 + 代码同步**：当节文档与当节代码一起完成；正文默认引用 `source/` 真实代码，不以伪代码作为主路径（例外见 §6.6）。
- **一节一交付**：一次只推进一个专题（一篇 `course/.../*.md` + 对应 `source/` 变更），完成后再进入下一节，减少返工。主战场是 **`source/packages/` 增量**；不承诺每节新建 `demos/` 目录（见 §6.4）。
- 每门课 **00** 创建本课 package 骨架与第一个可运行入口；**01 起**每节在同一 package 上增量实现。
- 专题正文遵循本文 §7 模板；各课 `outline.md` 已对齐备课结构（**暂定**，实施时可能微调边界），正文不必机械复制 outline 的全部章节名。

### 章节 Git 约定

每节「文档 + 代码 + 完成标准」齐备后提交一次 Git，便于回溯：

**Commit message 格式：**

```text
course(02_llm/00): LLM 问题空间与 first_chat demo
course(02_llm/01): Provider 抽象与 LLMClient
```

**阶段 Tag（仅用户明确下令时使用）：**

整门课或一个学习阶段完成后再打 tag，例如 `course/02_llm`。日常每节 commit 即可，不必每节打 tag。

### 真实项目开发模式

以真实项目开发的逻辑边学边做：自己承担产品、前后端和 AI 开发角色，按业务版本递进交付，以**完整业务逻辑闭环**为验收目标，而不是以「章节完成」或「脚本跑通」为终点。

### AI Coding

大量借助 AI 写代码是预期方式。是否掌握，以 [ai-coding-mastery.md](ai-coding-mastery.md) 中的代码所有权标准判断：能解释、能验证、能修改、能调试、能迁移——而不是代码由谁生成。

### 能力分层，而非能力割裂

`02_llm`、`03_rag`、`04_agent` 等目录是教学分层，服务于项目递进；LLM、RAG、Tool Calling、Agent、Workflow、Eval、Frontend 可在项目需要时交叉出现，不强制线性学完一整门再进入下一门。

### 参考开源项目的方式

可参考 `other/` 下 MaxKB、RAGFlow 等项目的**设计思路**（如知识生产层、应用产品对象、Workflow 运行时），吸收机制与边界判断；不复制其目录结构、脚本组织或学习顺序。

### 与 archive 的关系

历史课程式资料在根目录 [archive/](../archive/)，与新课程无学习路径关系。编写新文档和新代码时默认不依赖 archive；详见 [AGENTS.md](../AGENTS.md)。

## 3. 为什么这样组织

Python 和 LLM API 适合章节化学习，也适合用脚本快速获得反馈。

RAG、Agent 和 Workflow 更接近系统工程。如果仍然要求每个知识点都对应一组脚本，容易把学习变成“完成脚本”，而不是理解问题、设计链路和完成项目。

后续学习更关注：

- 真实问题是什么。
- 为什么需要这个能力。
- 最小实现如何验证原理。
- 主流框架如何封装同一件事。
- 能力边界和失败模式在哪里。
- 如何收敛到一个可运行的小项目。

## 4. 课程编排原则

当前课程按主线和补充区区分。各课 `outline.md` 已与本文对齐并**定稿（暂定）**：实施时可能微调单节边界，结构以本文 §6–§7 为准。

- `python_base`：已完成的 Python 基础，后续不作为主线扩展。
- `02_llm`：模型交互、Prompt、结构化输出、上下文工程和调用底座。
- `03_rag`：RAG、LangChain 与单 Agent 知识助手。
- `04_agent`：Agent 工作流、工具系统和可控执行。
- `05_eval_observability`：评估、可观测、回归和质量工程。
- `06_ai_native`：AI Native 体验、运行态 UI、交互协议和项目必需后端工程。
- `07_projects`：需求评审助手的项目收敛与作品化。

其中 `python_base` 不作为后续课程扩展重点。非主线补充资料见 [archive/](../archive/)。后续新内容以当前规范为准。

当前主项目是需求评审助手。课程评估和新内容设计应优先服务这个项目的可运行闭环（V0–V6）；`docs/ai-application-platform.md` 是远期平台化能力地图，不作为当前阶段的完整交付要求。

从 RAG 开始，课程编排不再强制遵循“完整学完一个大阶段才能进入下一个大阶段”的线性方式。

RAG、Tool Calling、Agent、Workflow、Evaluation、Observability 和 Frontend Workbench 可以围绕项目需要交叉出现。

课程目录可以继续讨论和演进，但每个阶段都应该服务 AI 应用主链路，而不是孤立堆知识点。

## 5. RAG 与 Agent 以及后续课程的关系

RAG 和 Agent 不应该被理解成完全割裂的两门课。

更合理的关系是：

- 固定 RAG 是知识注入和依据生成的基础能力。
- Tool Calling 是模型连接外部能力的基础机制。
- Agentic RAG 是在固定 RAG 不够时，引入查询改写、动态检索、多知识源路由和质量判断。
- LangGraph / Workflow 用于复杂状态流转、分支、循环和人机协作。
- Evaluation 和 Observability 贯穿所有 AI 应用阶段。

学习顺序应该服务项目闭环：

```text
先跑通固定链路
-> 再按问题引入工具调用
-> 再按复杂度引入 Agent / Workflow
-> 始终保留评估、观测和前端可视化
```

## 6. 课程正文写作标准

课程正文不是 outline 的展开，也不是项目架构设计笔记。它首先是学习材料，然后才是项目设计输入。新增或重写专题时，必须同时服务两件事：

- 让学习者从当前认知水平理解一个概念。
- 让这个概念进入需求评审助手的项目闭环。

### 6.1 概念先行

每个阶段遇到新的一级概念时，第一篇必须先解释：

- 它是什么。
- 它解决什么问题。
- 它接收什么输入，产生什么输出。
- 它和已有概念有什么区别。
- 它在需求评审助手中处于哪一环。
- 它常见失败和误区是什么。

例如：

- `02_llm/00` 需要先解释 LLM 是什么、和普通程序 / 搜索 / 数据库有什么不同。
- `03_rag/00` 需要先解释 RAG 是什么、为什么不能只靠 Prompt 或长上下文。
- `04_agent/00` 需要先解释 Agent / Chain / Workflow 的差异，而不是直接进入 Tool Runtime。

不要假设学习者已经掌握后续章节的概念。正文不能带着完整系统知识回头总结；它必须让学习者按顺序读得懂。

### 6.2 三层真实问题

专题中的「真实问题」必须按三层写清楚，而不是只写项目架构问题：

| 层级 | 要回答的问题 |
| --- | --- |
| 学习者真实问题 | 我为什么要学这个？它和我已知的东西有什么不同？ |
| 产品真实问题 | 需求评审助手为什么需要这个能力？不用会怎样？ |
| 工程真实问题 | 引入这个能力后会出现什么失败、边界、成本和治理问题？ |

如果一个专题的真实问题只有「系统需要稳定、可观测、可扩展」这类抽象表达，通常说明还不够真实。应补充一个可感知的业务例子，例如：用户上传一份 PRD 后，系统为什么不能直接把全文丢给模型；模型为什么可能引用不存在的条款；前端为什么不能只等最终答案。

### 6.3 正文深度

课程正文要比 outline 更厚。每篇专题至少应包含：

- 概念解释：用学习者能理解的语言解释术语。
- 最小例子：给出一个输入、处理、输出的完整小场景。
- 对比表：说明它和相近概念或传统做法的区别。
- 数据流：说明信息从哪里来、到哪里去（仅当节相关，不画全栈架构图）。
- 常见误区：列出初学者容易误解的地方。
- 完成标准：用「能解释 / 能判断 / 能运行 / 能改造」描述，含当节验收命令与预期输出。

不要在每节正文重复写 V0–V6 全表、全课 capability 矩阵或完整 package 目录树；这些放在 `outline.md` 或该课最后一节。

文档可以长，但不能为了显得完整堆概念。篇幅应由解释是否充分决定，而不是由行数决定。对全新概念的首篇正文通常应明显厚于普通专题。

**厚度验收（发布前自检）**：

- 至少 **1 段连续业务场景叙事**（建议 300–500 字），让学习者能「跟着走一遍」需求评审助手里的具体一幕，而不是只有抽象 bullet。
- 至少 **3 道自检题**：不看正文能否回答（例如「LLM 和数据库查询有什么不同？」「为什么这里不能只调一次 API？」）。
- 新概念首篇除上述外，还应包含完整的「是什么 / 输入输出 / 与相近概念区别」说明（见 §6.1）。

若自检题答不上来，说明解释层不够，应加厚机制说明与反例，而不是继续堆表格。

**常见误解（不要这样理解规范）**：

- 不堆概念 ≠ 不写当节机制；无关概念才删，核心机制必须讲透。
- 以真实代码为准 ≠ 正文变成源码走读；须内嵌少量关键片段并讲解（§6.6）。
- 一节一主体 ≠ 只写交付、不讲为什么；认知链路与交付链路并存（§7）。
- 表格是辅助，不能替代 prose、反例和排查路径。
- 本节实战 ≠ demo README；正文只保留理解本节所需的入口、命令和观察点，详细参数、文件对照、配置表优先放到对应 README。

**双层验收**：正文须同时满足 **认知链路**（场景叙事、机制 prose、对比、误区、排查或判断路径）与 **交付链路**（少量关键片段、本节实战、运行与观察、自检题、本节沉淀）。去掉「本节实战」后，「基础原理」仍应可读通。

**重心验收**：课程正文的第一责任是教学解释，不是操作手册。若一篇正文中代码块、配置表、文件树和命令说明占据阅读主导，应压缩交付细节或下沉到 demo / package README，并补足连续讲解、反例和判断路径。读者关闭代码块与表格后，仍应能理解「为什么需要这个能力、它如何工作、什么时候会失败」。

**按节类型**：

| 类型 | 适用 | 要求 |
| --- | --- | --- |
| 新概念首篇 | 每门课 `00` | 是什么 / 输入输出 / 与相近概念区别；场景叙事建议 ≥400 字 |
| 普通专题 | `01` 及以后 | 满足上文双层验收与厚度自检 |
| 跨节能力 | 当节不展开 | 写明 defer 到第 X 节 + 当节最小判断（不能只写「见第 X 节」） |

厚度自检另加三条：去掉「本节实战」后「基础原理」仍可读通；去掉代码块和表格后，正文 prose 仍能串起当节机制；「失败分析」至少一条含「表现 → 原因 → 怎么验证」。

**机制递进**（当节存在多种相近手段或层层约束时）：在「基础原理」中写出**从弱到强**的递进链——每步说明解决什么、仍遗留什么、一个反例；不能只写最终项目里的调用路径（例如只写「定义 schema → 跑 demo」）。完整能力若 defer 到后续节，须写清目标节号与**当节最小判断**。

专题正文写作的执行清单见仓库 [skills/course-writing/SKILL.md](../skills/course-writing/SKILL.md)。

### 6.4 文档与代码节奏（一节一交付）

默认节奏：**每节 = 当节文档 + 当节 `packages/` 增量代码 + 完成标准 → Git commit → 下一节**。不要批量写完多节文档再统一补代码。

```text
每门课 00：在 source/packages/ 创建或扩展本课 *_core + 第一个可运行 demo（若该课 00 正文要求）
每节 01、02、…：写当节文档 + 实现当节 package 代码 + 更新相关 README
一节完成：文档可读、代码可跑、完成标准可勾选
```

**目录真源（避免多处维护文件树）**

| 用途 | 真源 |
| --- | --- |
| 规范与策略 | 本文 §6.4、§10 |
| 当前 `source/` 实物清单 | [source/README.md](../source/README.md)（仅顶层，随增删改） |
| 当节路径与运行命令 | `course/.../0N_*.md`「本节实战」 |
| 备课与教学边界 | 各课 `outline.md`（**不**随每次代码 commit 同步目录树） |

**source 目录约定（扁平）**

```text
source/
├── packages/          # *_core 全仓库单实例，按课增量创建（如 llm_core）
├── demos/             # 关键 demo；命名 {课号}_{名称}，非每节必建
├── apps/              # 学习期应用壳，在当节正文落地时创建
└── python_base/       # 已完成 Python 练习，不适用 package 规范

review_assistant/      # 07_projects 起：根目录可部署产品（import source/packages）
```

**禁止预建占位（硬规则）**

- **无用户明确指令**，不得创建 `.gitkeep`、空 package、空 demo、空 app 目录。
- 目录与文件只在**当节文档 + 当节代码**落地时创建。
- `outline.md` 中的里程碑、终态对照是**备课清单**，不是实施前要建好的目录树；**不为对齐目录而频繁改 outline 或其它 `docs/` 规范文档**。

**Demo 策略**（按顺序判断，非每节必建 demo）

1. 已有 demo 加参数能否观测？→ 复用，不新建。
2. package 内 smoke / 测试是否够？→ 不新建 demo。
3. 本节是否需要**新的观测维度**（如流式 token 事件）？→ 才新增或拆 demo。
4. 长链路课 → 优先**串联节**合并验收，而非每节一个脚本。

**00 的特殊职责**

- 建立 `source/packages/{name}_core/`（若尚不存在）、`__init__.py`、`.env.example`（若尚未有）、`source/demos/{课号}_first_*` 第一个可运行 demo。
- **00 代码上限**：目录 + 可 import 的 package 壳 + 一个最小 demo；不预建后续节的模块文件，不搭完整基础设施（如全量入库流水线、完整 Workflow 运行时、完整 app 闭环）。
- 不实现后续节能力，不画全模块架构图。

**01 起的特殊职责**

- 「本节实战」必须对应真实文件路径与运行命令。
- 正文示例**默认引用当节已实现的 `source/` 代码**（片段或文件链接），与代码同步完成。

**无新增代码节（例外）**

少数专题以解释、对照框架为主（如纯 LangGraph 概念对比）。须在完成标准中**显式写明**：「本节无新增代码；运行与观察 = 执行 xxx demo 验证理解」。仍是完整的一节交付，只是代码变更可为空。

**串联节（可选）**

RAG、Agent 等长链路课程中，可在若干专题完成后设一节「串联 demo」：不新讲概念，只把前几节代码接成一条可跑链路。仍是一节一交付、一次 commit，不是批量文档后统一实现。

**期末对照（可选）**

各课 `outline.md` 可保留全课能力对照表，供备课与复盘；**不作为写作顺序约束**，**不维护与 `source/` 一一对应的文件树**（实物以 `source/README.md` 与当节正文为准）。

### 6.5 正文链接策略

正文以**当节自学**为目标，交叉链接从简：

| 允许 | 尽量避免 |
| --- | --- |
| 同课上一节 / 下一节 | 跳到未学课程的 outline 或专题 |
| 当节对应的 `source/...` 文件 | 跳到 `07_projects` 子专题 |
| 本课 `outline.md`（一篇） | 在 00/01 链到 `archive/`、未学课程等 |
| `learning-guide.md` 某节（规范引用） | 多个 outline 互相索引 |

超前能力用**口头一句**交代即可（如「后续 RAG 课负责检索」），不必加 Markdown 链接，除非读者**此刻就要打开**该文件才能继续当节实战。

### 6.6 概念示意（例外）

一节一交付下，正文**以 `source/` 真实代码为准**。文档与当节代码同步完成，不存在「先写伪代码、后补实现」的主路径。

**默认写法**：

- 每篇普通专题通常内嵌 **1–2 个**当节真实代码片段；机制复杂时最多 **3 个**。片段应短而关键，服务「为什么这样设计能 work」，不能为满足数量而贴源码。
- 路径链接作补充，禁止整节只有链接、无片段；也禁止用大量片段替代机制讲解。
- 「最小实现」与「基础原理」用少量片段 + 输入输出示例、数据流或 mermaid 说明机制。配置全集、命令变体、实验参数表、文件名细节优先放到对应 README。

**仅下列情况可用简短示意**（须标注「概念示意，不进入项目」）：

- 明确不实现的机制（如第三方平台内部调度，本仓库不会写）。
- 纯框架对照（「若用 LangGraph 会写成…」），本节不新增项目代码。
- 跨多文件、下节才实现的接口形状预览（宜极简，下节必须用真实代码替换）。

正文中不应保留未标注的伪代码，也不应让示意代码伪装成可运行入口。

## 7. 专题文档模板

后续新增或重写专题正文时，优先使用下面的结构。

**两层结构**：

| 层次 | 回答的问题 | 章节 |
| --- | --- | --- |
| **A. 学习认知链路** | 怎么从问题走到理解、验证、对照框架、知边界 | 真实问题 → 基础原理 → 最小实现 → 主流框架实现 → 失败分析与能力边界 |
| **B. 单节交付约定** | 这一节文档写什么、代码写哪、怎么验收 | 本节实战 → 完成标准 → 本节沉淀 → 相关专题（可选） |

`outline.md` 里可能仍保留「评估观测 / 项目收敛 / 小项目实战」等备课条目；正文按本模板写，不必逐字复制 outline 章节名。

### 真实问题

说明遇到什么问题，为什么需要学习这个能力。新概念首篇按「学习者 / 产品 / 工程」三层展开（§6.2）。聚焦**当节**，不提前展开未学课程的全景。

### 基础原理

讲清楚当节核心机制。新概念首篇必须先解释「是什么、做什么、怎么用、输入输出是什么」（§6.1）。每个核心机制宜含：**机制 prose**（非纯 bullet）、**反例**、**排查或判断路径**（先怀疑什么、再看什么）。存在多层约束或多种相近 API 时，按 §6.3「机制递进」写出从弱到强的理解链，再落到项目最小实现。

### 最小实现

用尽量小的实现验证原理是否成立。**默认**从当节 `source/` 代码摘取：

- 1–2 个关键函数或调用链的短片段（附文件路径）；
- 一次 API / CLI 的输入输出示例；
- 若本节实战在前、最小实现在后写，两者描述同一链路，不重复两套步骤。

**职责**：回答「为什么这样设计能 work」。可运行部分在本节实战中；最小实现侧重解释路径，不是第二份实现指南，也不是源码导览。若需要解释多个文件的协作，优先用数据流和职责边界描述，只摘最关键的代码片段。

### 主流框架实现

说明主流框架如何封装同一件事，例如 OpenAI SDK、LangChain、LangGraph、MCP 等。

**职责**：回答「业界通常怎么包、和自研最小实现差在哪」。篇幅**宜短可长**——LLM 节可短对照，RAG / Agent 节可更长甚至带 demo；**多数节不必为此新增项目代码**。

### 失败分析与能力边界

当节的常见失败、误区、不适用场景。至少一条按「表现 → 原因 → 怎么验证」写清。跨节能力须写明 defer 到第 X 节 + 当节最小判断，不能只写「见第 X 节」。

### 本节实战

**本节代码实现过程**（需求评审助手是唯一主项目，每节在主项目底座上增量交付，不是另起小项目）。把「最小实现」中需要跑通的部分落到 `source/`。

必须包含：

- **目标**：本节结束后项目多了什么能力（一句话）。
- **涉及文件**：`source/` 下本节新增或修改的关键路径；不贴完整文件树，完整清单放 README。
- **实现步骤**：关键步骤与原理、最小实现对照；避免展开成源码走读。
- **运行方式**：一条主命令（根目录 `.venv`、`.env`）；命令变体和完整配置放 README。
- **预期结果**：终端输出或行为描述（可贴示例）。

00 可以是「package 骨架 + first demo」；01 起应对应当节全部核心代码。

本节实战应保持「能开始、能观察、能验收」的粒度。若需要解释 demo 顶部常量、样例全集、参数组合、错误分支或配置矩阵，应在正文给出 README 链接，而不是把 README 搬进课程正文。

### 完成标准

用可验证动词描述，并包含**本节验收**：

- 能解释 / 能判断 / 能运行 / 能改造（按当节选取）。
- **运行与观察**：一条命令 + 应看到什么（代替大部分节的「评估观测」）；无新增代码节见 §6.4 例外说明。
- **自检题**：至少 3 道（§6.3）。

### 本节沉淀

仅 2–3 行，回答：

- 本节新增或修改了哪些文件或模块。
- 对需求评审助手多出来的能力（一句话，不写 V0–V6 全表）。
- （可选）下一节依赖本节什么——可链**同课**下一篇文档。

不要在 00/01 写全课 capability 矩阵、完整 `*_core` 八模块树；全课收敛放在该课 **最后一节** 或 `outline.md`。

### 相关专题（可选，从简）

默认只链：**同课上一节 / 下一节**。不链未学课程、不链 `07_projects` 子专题。

---

**「评估观测 / 项目收敛 / 小项目实战」在正文中的处理**

| outline 常用名 | 正文处理方式 |
| --- | --- |
| 最小实现 | **保留**；可运行部分在本节实战中落地 |
| 主流框架实现 | **保留**；宜短可长，多数节不必新增项目代码 |
| 小项目实战 | 改名为 **本节实战**（主项目增量，不是另起小项目） |
| 评估观测 | 合并进 **完成标准 → 运行与观察**；`05_eval` 专章除外 |
| 项目收敛 | 瘦身为 **本节沉淀**；全课收敛仅末节 / outline |

## 8. 项目文档模板

后续新增项目文档时，优先使用下面的结构。

### 项目定位

说明项目解决什么真实问题，不解决什么问题。

### 面向角色

说明项目服务哪些用户或协作角色。

### 业务主链路

用文本链路描述输入、处理、输出和反馈。

### V0 最小闭环

说明最小可运行版本包含什么、不包含什么，以及如何判断完成。

### V1 能力增强

说明在 V0 基础上新增哪些能力，为什么新增。

### V2 产品化完善

说明前端体验、评估、观测、部署和演示能力如何完善。

### 技术模块

说明各技术模块的职责边界。

### 失败边界

说明项目不能解决什么、依赖什么前提、什么时候需要人工介入。

### 完成标准

说明项目到什么程度算完成，而不是只写“实现某功能”。

## 9. 文档写作规范

- 一节一交付：当节文档与当节代码同步完成，再进入下一节（§6.4）。
- **学习者真实问题先于概念术语**：先建立「为什么要学、业务里发生什么」（§6.2），再在「基础原理」中做新概念入门（§6.1）。
- 概念入门先于架构设计。
- 正文链接从简（§6.5）；专注当节教学，不引导读者跳转未学章节。
- 最小闭环先于工程完整。
- 正文以当节 `source/` 真实代码为准；概念示意仅作例外（§6.6）。
- 不写迁移过程、历史流水账和过渡状态。
- 不维护多个重复真源。
- 不把实时学习进度写入长期文档。
- 不为了完整性堆概念或全栈映射表。
- 不为了每个知识点硬造脚本。

文档至少要回答：

- 为什么学（当节）。
- 这个概念是什么、输入输出是什么、和已有概念有什么区别。
- 学到什么程度。
- 当节代码如何运行、应看到什么。
- 什么时候不该用。
- 本节在主项目里多了什么能力。

## 10. 代码组织规范

- 示例代码服务理解；项目代码服务闭环。
- AI 可以辅助生成代码，但代码必须可解释、可修改、可验证、可调试。
- **每节推进当节代码**，不等整门课正文写完再统一实现（§6.4）。
- 每门课 00 在 `source/packages/` 创建或扩展本课负责的 `*_core`（若尚不存在）；后续节在同一包上增量修改。
- 新代码优先围绕主项目能力组织，**不按章节堆独立脚本**；**不为一节一 demo**。
- 每个 **package** 与**已落地的 demo** 必须有 README、运行入口和当节完成标准。
- 涉及真实 API、密钥、模型配置时必须隔离（`.env`）。
- 涉及 RAG、Agent 时，优先保留可调试输出（在对应章节再展开）。

### 共享 package 策略（单实例、import、逐步加厚）

- 每个能力域全仓库只维护**一个** package 实例，物理位置在 **`source/packages/`**（如 `llm_core`；`rag_core` 等在**对应课当节实施时**再创建），**禁止 copy 多份平行实现**。
- 后续课程、`source/demos/`、学习期 `source/apps/` 与根目录 `review_assistant/` 一律通过 **import** 复用并增量完善，不重复造轮子。
- **深度分阶段**：`03_rag` 等以**最小实现**跑通机制；`06_ai_native` 以**工程深度**整合（FastAPI、PostgreSQL / pgvector、Redis、Docker Compose 等）——仍是同一包渐进加厚。
- **应用根（届时创建）**：
  - `source/apps/` 下学习期应用壳：在当节正文落地时创建；
  - `review_assistant/`（仓库根）：`07_projects` 起**可部署产品真源**；import `source/packages`，不 copy、不另建平行 `*_core`。
- **`source/python_base/`**：已完成 Python 基础练习，保留章节式结构，不适用上述 package/demos 规范。

### Python workspace

- 根目录 `pyproject.toml` 以 editable 方式安装 `source/packages/*` 下各包，避免手写 PYTHONPATH。
- 根 `review_assistant/`（及届时存在的学习期 app）通过 path dependency 依赖上述包。
- 依赖清单仍维护在根 `requirements.txt`。

示例代码可以简单，但必须能运行、能解释。

## 11. AI Coding 下的掌握标准

后续课程和项目可以大量使用 AI Coding，但不能把“AI 生成代码并跑通”视为已经掌握。

一个代码任务完成后，至少应能回答：

- 为什么这样设计。
- 入口在哪里。
- 数据流、状态流和异常流怎么走。
- 如果需求变化，应该改哪些模块。
- 如果报错，应该从哪里定位。
- 如何验证改动没有破坏原有能力。

完整原则见 [ai-coding-mastery.md](ai-coding-mastery.md)。

## 12. 完成标准

一个专题完成，至少满足：

- 能解释真实问题（当节三层）。
- 能解释关键概念是什么、解决什么问题、输入输出是什么。
- 能说明基础原理与常见误区。
- **本节实战代码可运行**（或完成标准中显式声明「本节无新增代码」及验收 demo），且文档中的运行方式与 `source/` 一致。
- 能说明当节失败边界（不展开全栈）。
- 能通过 §6.3 厚度自检（场景叙事 + 自检题）。
- 完成标准中含**运行与观察**（命令 + 预期输出）。
- 已按 §2 约定提交 Git（用户执行时）；阶段 tag 仅用户明确下令时打（如 `course/02_llm`）。

一个项目阶段完成，至少满足：

- 有可运行入口。
- 有业务输入输出。
- 有基础评估或验证方式。
- 能展示来源、过程或结果。
- 能讲清楚价值和边界。
