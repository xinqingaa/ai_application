# Learning Guide

这份文档是当前仓库学习设计、课程设计、项目设计、写作规范和代码组织规范的唯一真源。

## 1. 学习组织方式

当前学习以真实问题和项目闭环组织：

```text
真实问题
-> 基础原理
-> 最小实现
-> 主流框架实现
-> 失败分析与能力边界
-> 小项目实战
```

学习目标不是机械完成章节，而是形成可解释、可运行、可展示、可评估的 AI 应用能力。

在 AI Coding 参与下，课程完成不以“代码由谁敲出”判断，而以是否真正拥有代码所有权判断。具体标准见 [ai-coding-mastery.md](ai-coding-mastery.md)。

## 2. 学习方式与项目驱动原则

本仓库的学习方式，是按**一个确定的业务项目**开展系统性的 AI 应用开发学习，而不是按孤立章节刷脚本。

### 唯一主项目

当前唯一长期主项目是「**需求评审助手**」：从 RAG 知识助手，逐步演进为具备多 Agent 协作、Workflow、人工确认、评估观测和 AI Native 工作台的完整评审系统。

通过这个项目，系统性掌握 LLM、RAG、Agent、Multi-Agent、Workflow、Eval、Observability 和 AI Native 工程能力。项目版本路线见 [strategy.md](strategy.md) 与 [course/07_projects/outline.md](../course/07_projects/outline.md)（V0–V6）。

### 文档 + 代码

- **文档最重要**：定义真实问题、原理、边界、完成标准和项目收敛方式。
- **代码是实现方式**：验证关键机制，沉淀可复用 package，完成 demo 和项目闭环；不为每个知识点硬造脚本。
- 专题文档须包含「评估观测」「项目收敛」与「完成标准」；新增专题遵循本文 §7 模板。

### 真实项目开发模式

以真实项目开发的逻辑边学边做：自己承担产品、前后端和 AI 开发角色，按业务版本递进交付，以**完整业务逻辑闭环**为验收目标，而不是以「章节完成」或「脚本跑通」为终点。

### AI Coding

大量借助 AI 写代码是预期方式。是否掌握，以 [ai-coding-mastery.md](ai-coding-mastery.md) 中的代码所有权标准判断：能解释、能验证、能修改、能调试、能迁移——而不是代码由谁生成。

### 能力分层，而非能力割裂

`02_llm`、`03_rag`、`04_agent` 等目录是教学分层，服务于项目递进；LLM、RAG、Tool Calling、Agent、Workflow、Eval、Frontend 可在项目需要时交叉出现，不强制线性学完一整门再进入下一门。

### 参考开源项目的方式

可参考 `other/` 下 MaxKB、RAGFlow 等项目的**设计思路**（如知识生产层、应用产品对象、Workflow 运行时），吸收机制与边界判断；不复制其目录结构、脚本组织或学习顺序。

### 与 00_archive 的关系

`00_archive` 仅为过渡保留，与新课程无学习路径关系。编写新文档和新代码时默认不依赖 archive；详见 [AGENTS.md](../AGENTS.md)。

## 3. 为什么这样组织

Python 和 LLM API 适合章节化学习，也适合用脚本快速获得反馈。

RAG、Agent 和 Workflow 更接近系统工程。如果仍然要求每个知识点都对应一组脚本，容易把学习变成“完成脚本”，而不是理解问题、设计链路和完成项目。

后续学习更关注：

- 真实问题是什么。
- 为什么需要这个能力。
- 最小实现如何验证原理。
- 主流框架如何封装同一件事。
- 能力边界和失败模式在哪里。
- 如何收敛到一个可运行的小项目。

## 4. 课程编排原则

当前课程按主线和补充区区分：

- `01_python`：已完成的 Python 基础，后续不作为主线扩展。
- `02_llm`：模型交互、Prompt、结构化输出、上下文工程和调用底座。
- `03_rag`：RAG、LangChain 与单 Agent 知识助手。
- `04_agent`：Agent 工作流、工具系统和可控执行。
- `05_eval_observability`：评估、可观测、回归和质量工程。
- `06_ai_native`：AI Native 体验、运行态 UI、交互协议和项目必需后端工程。
- `07_projects`：需求评审助手的项目收敛与作品化。
- `99_foundation`：非主线知识补充区，默认不阅读、不扩展、不维护。

其中 `01_python` 和 `99_foundation` 不作为后续课程扩展重点。后续新内容以当前规范为准。

当前主项目是需求评审助手。课程评估和新内容设计应优先服务这个项目的可运行闭环（V0–V6）；`docs/ai-application-platform.md` 是远期平台化能力地图，不作为当前阶段的完整交付要求。

从 RAG 开始，课程编排不再强制遵循“完整学完一个大阶段才能进入下一个大阶段”的线性方式。

RAG、Tool Calling、Agent、Workflow、Evaluation、Observability 和 Frontend Workbench 可以围绕项目需要交叉出现。

课程目录可以继续讨论和演进，但每个阶段都应该服务 AI 应用主链路，而不是孤立堆知识点。

## 5. RAG 与 Agent 以及后续课程的关系

RAG 和 Agent 不应该被理解成完全割裂的两门课。

更合理的关系是：

- 固定 RAG 是知识注入和依据生成的基础能力。
- Tool Calling 是模型连接外部能力的基础机制。
- Agentic RAG 是在固定 RAG 不够时，引入查询改写、动态检索、多知识源路由和质量判断。
- LangGraph / Workflow 用于复杂状态流转、分支、循环和人机协作。
- Evaluation 和 Observability 贯穿所有 AI 应用阶段。

学习顺序应该服务项目闭环：

```text
先跑通固定链路
-> 再按问题引入工具调用
-> 再按复杂度引入 Agent / Workflow
-> 始终保留评估、观测和前端可视化
```

## 6. 课程正文写作标准

课程正文不是 outline 的展开，也不是项目架构设计笔记。它首先是学习材料，然后才是项目设计输入。新增或重写专题时，必须同时服务两件事：

- 让学习者从当前认知水平理解一个概念。
- 让这个概念进入需求评审助手的项目闭环。

### 6.1 概念先行

每个阶段遇到新的一级概念时，第一篇必须先解释：

- 它是什么。
- 它解决什么问题。
- 它接收什么输入，产生什么输出。
- 它和已有概念有什么区别。
- 它在需求评审助手中处于哪一环。
- 它常见失败和误区是什么。

例如：

- `02_llm/00` 需要先解释 LLM 是什么、和普通程序 / 搜索 / 数据库有什么不同。
- `03_rag/00` 需要先解释 RAG 是什么、为什么不能只靠 Prompt 或长上下文。
- `04_agent/00` 需要先解释 Agent / Chain / Workflow 的差异，而不是直接进入 Tool Runtime。

不要假设学习者已经掌握后续章节的概念。正文不能带着完整系统知识回头总结；它必须让学习者按顺序读得懂。

### 6.2 三层真实问题

专题中的「真实问题」必须按三层写清楚，而不是只写项目架构问题：

| 层级 | 要回答的问题 |
| --- | --- |
| 学习者真实问题 | 我为什么要学这个？它和我已知的东西有什么不同？ |
| 产品真实问题 | 需求评审助手为什么需要这个能力？不用会怎样？ |
| 工程真实问题 | 引入这个能力后会出现什么失败、边界、成本和治理问题？ |

如果一个专题的真实问题只有「系统需要稳定、可观测、可扩展」这类抽象表达，通常说明还不够真实。应补充一个可感知的业务例子，例如：用户上传一份 PRD 后，系统为什么不能直接把全文丢给模型；模型为什么可能引用不存在的条款；前端为什么不能只等最终答案。

### 6.3 正文深度

课程正文要比 outline 更厚。每篇专题至少应包含：

- 概念解释：用学习者能理解的语言解释术语。
- 最小例子：给出一个输入、处理、输出的完整小场景。
- 对比表：说明它和相近概念或传统做法的区别。
- 数据流：说明信息从哪里来、到哪里去。
- 常见误区：列出初学者容易误解的地方。
- 项目映射：说明进入需求评审助手哪个版本、哪个模块。
- 完成标准：用「能解释 / 能判断 / 能运行 / 能改造」描述。

文档可以长，但不能为了显得完整堆概念。篇幅应由解释是否充分决定，而不是由行数决定。对全新概念的首篇正文通常应明显厚于普通专题。

**厚度验收（发布前自检）**：

- 至少 **1 段连续业务场景叙事**（建议 300–500 字），让学习者能「跟着走一遍」需求评审助手里的具体一幕，而不是只有抽象 bullet。
- 至少 **3 道自检题**：不看正文能否回答（例如「LLM 和数据库查询有什么不同？」「为什么这里不能只调一次 API？」）。
- 新概念首篇除上述外，还应包含完整的「是什么 / 输入输出 / 与相近概念区别」说明（见 §6.1）。

若自检题答不上来，说明解释层不够，应加厚机制说明与反例，而不是继续堆表格。

### 6.4 文档与代码节奏

新课程不回到「每个知识点一个脚本」的旧模式，也不建议把全部文档写完后再统一写代码。推荐节奏是：

```text
先写一组概念与设计文档
-> 到达一个能力闭环点
-> 写最小可运行代码
-> 回头修正文档，把设计草图替换为真实代码入口
-> 再进入下一组文档
```

`02_llm` 可按下面的节奏执行（**示例排期**，具体以 [course/02_llm/outline.md](../course/02_llm/outline.md) 中的「代码里程碑」为准，实施时可修订）：

```text
00-03：LLM 问题空间 / Provider / Prompt / Structured Outputs
-> 第一轮代码（M1）：llm_core 最小版本 + structured_review_output demo

04-06：Streaming / Context / Reliability
-> 第二轮代码（M2）：streaming + error handling demo

07-10：Harness / Cost / Function Calling / Project Foundation
-> 第三轮代码（M3）：收敛 llm_core package
```

**代码里程碑完成定义（以 `02_llm` M1 为例，其他课程同理写 2–3 条）**：

| 里程碑 | 最小验收 |
| --- | --- |
| M1 调用 + 结构化 | `LLMClient` 可切换 provider；`structured_review_output` demo 可跑；00–03 正文中设计草图已回链到 `source/` |
| M2 流式与可靠性 | streaming demo 可跑；错误分类与有限重试可观测 |
| M3 package 收敛 | `llm_core` 模块齐全；`03_rag` 可 import 调用底座 |

每轮代码完成后必须：**回链文档**（§6.5）、更新对应 outline 里程碑状态（可选标注 `done` / 日期）。

`03_rag`、`04_agent`、`05_eval_observability`、`06_ai_native` 也应在各自 outline 写 **2–3 个代码里程碑（草案）**，按能力闭环切分，而不是整门课文档全部写完后才动代码。

### 6.5 伪代码规则

文档初稿可以使用伪代码，但必须明确标注为「设计草图」或「伪代码」。伪代码用于解释数据流和接口形状，不应伪装成可运行代码。

当真实代码存在后，应回头更新文档：

- 将伪代码替换为真实代码入口、函数签名或关键片段。
- 链接到 `source/` 中的真实文件。
- 说明运行方式和观察重点。
- 保留必要的设计解释，不把文档降级成代码注释。

长期正文中不应保留大量过期伪代码。若某段代码只是概念演示而不会实现，应明确说明「本段只用于解释，不进入项目代码」。

## 7. 专题文档模板

后续新增专题文档时，优先使用下面的结构。

### 真实问题

说明遇到什么问题，为什么需要学习这个能力。遇到新概念时，优先按「学习者真实问题 / 产品真实问题 / 工程真实问题」三层展开。

### 基础原理

用最少必要概念讲清楚核心机制。新概念首篇必须先解释「是什么、做什么、怎么用、输入输出是什么」。

### 最小实现

用尽量小的实现验证原理。不是每个专题都必须写大量脚本，但如果有代码，必须能说明入口、运行方式和观察重点。

### 主流框架实现

说明主流框架如何封装同一件事，例如 OpenAI SDK、LangChain、LangGraph、MCP 或其他工具。

### 失败分析与能力边界

说明常见失败、误区、不适用场景和需要人工介入的情况。

### 小项目实战

说明这个能力如何进入阶段项目，并和其他能力组成完整链路。

### 评估观测

说明如何记录指标、样例、对比实验或最小 harness，便于判断改动是否变好。即使尚未实现完整 eval 平台，也应写清「观察什么、记录什么」。

### 项目收敛

说明本专题沉淀到哪个 package / 模块、与项目哪个版本（V0–V6）对齐、给下游课程什么接口或约定。

### 完成标准

用可验证动词描述完成标准，例如：

- 能解释
- 能运行
- 能实现
- 能判断
- 能改造
- 能在项目中使用

## 8. 项目文档模板

后续新增项目文档时，优先使用下面的结构。

### 项目定位

说明项目解决什么真实问题，不解决什么问题。

### 面向角色

说明项目服务哪些用户或协作角色。

### 业务主链路

用文本链路描述输入、处理、输出和反馈。

### V0 最小闭环

说明最小可运行版本包含什么、不包含什么，以及如何判断完成。

### V1 能力增强

说明在 V0 基础上新增哪些能力，为什么新增。

### V2 产品化完善

说明前端体验、评估、观测、部署和演示能力如何完善。

### 技术模块

说明各技术模块的职责边界。

### 失败边界

说明项目不能解决什么、依赖什么前提、什么时候需要人工介入。

### 完成标准

说明项目到什么程度算完成，而不是只写“实现某功能”。

## 9. 文档写作规范

- 文档先于代码。
- **学习者真实问题先于概念术语**：先建立「为什么要学、业务里发生什么」（§6.2 第一层、第二层），再在「基础原理」中做新概念入门（§6.1）；不是跳过「是什么」直接列工程故障清单。
- 概念入门先于架构设计。
- 学习者真实问题先于系统设计问题。
- 最小闭环先于工程完整。
- 项目收敛先于章节堆叠。
- 伪代码必须标注为设计草图，并在真实代码出现后回链到 `source/`。
- 不写迁移过程、历史流水账和过渡状态。
- 不维护多个重复真源。
- 不把实时学习进度写入长期文档。
- 不为了完整性堆概念。
- 不为了每个章节都造脚本。

文档至少要回答：

- 为什么学。
- 解决什么真实问题。
- 这个概念是什么、输入输出是什么、和已有概念有什么区别。
- 学到什么程度。
- 最小可验证结果是什么。
- 什么时候不该用。
- 如何进入项目实战。

## 10. 代码组织规范

- 示例代码服务理解。
- 项目代码服务闭环。
- AI 可以辅助生成代码，但代码必须可解释、可修改、可验证、可调试。
- 不强制每个知识点都有脚本。
- 代码按能力闭环推进，不等全部正文写完后才统一实现。
- 新代码优先围绕项目阶段组织。
- 可以按 `V0 / V1 / V2` 多包逐步推进。
- 每个 package 必须有清晰职责、README、运行入口和完成标准。
- 涉及真实 API、密钥、模型配置时必须隔离。
- 涉及 RAG、Agent、评估和观测时，优先保留可调试输出。

示例代码可以简单，但必须能解释。

项目代码可以逐步工程化，但必须能运行。

## 11. AI Coding 下的掌握标准

后续课程和项目可以大量使用 AI Coding，但不能把“AI 生成代码并跑通”视为已经掌握。

一个代码任务完成后，至少应能回答：

- 为什么这样设计。
- 入口在哪里。
- 数据流、状态流和异常流怎么走。
- 如果需求变化，应该改哪些模块。
- 如果报错，应该从哪里定位。
- 如何验证改动没有破坏原有能力。

完整原则见 [ai-coding-mastery.md](ai-coding-mastery.md)。

## 12. 完成标准

一个专题完成，至少满足：

- 能解释真实问题。
- 能解释关键概念是什么、解决什么问题、输入输出是什么。
- 能说明基础原理。
- 有最小实现，或明确说明为什么不需要代码。
- 能说明主流框架如何解决同类问题。
- 能说明失败边界。
- 能说明评估观测与项目收敛（写进正文对应章节）。
- 能通过 §6.3 厚度自检（场景叙事 + 自检题）。
- 能进入项目链路。

一个项目阶段完成，至少满足：

- 有可运行入口。
- 有业务输入输出。
- 有基础评估或验证方式。
- 能展示来源、过程或结果。
- 能讲清楚价值和边界。
