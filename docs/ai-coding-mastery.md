# AI Coding 学习、掌握与代码所有权

这份文档用于沉淀本仓库在 AI Coding 参与下的学习原则、掌握标准和代码所有权要求。

它回答一个核心问题：

> 如果大量代码由 AI 生成，怎样才算真正学会 AI 应用和 Agent 开发？

## 1. 核心担心

后续课程和项目会大量使用 AI Coding。代码可能有 95% 甚至 99% 由 AI 生成，用户主要负责理论、架构、需求判断和代码审查。

这种学习方式有明显优势：

- 可以更快完成项目闭环。
- 可以把精力放在架构、边界、评估和调试上。
- 可以更接近未来真实开发方式。

但也有风险：

- 代码跑起来了，但并不理解为什么这样设计。
- 能让 AI 生成代码，但不能判断代码是否合理。
- 遇到异常报错，只会继续让 AI 猜。
- 需求变化后，不知道应该改哪个模块。
- 公司真实项目中架构、数据、流程都不理想时，无法迁移能力。

这份文档的目标是避免“完成项目”被误认为“真正掌握”。

## 2. 基本判断

AI 生成代码不等于没有学会。

AI 生成代码也不等于已经掌握。

真正的分界线不是“代码是谁敲的”，而是：

> 谁能对系统目标、结构、边界、失败、取舍和改动负责。

如果 AI 写了代码，但用户能解释它、验证它、修改它、调试它、迁移它，那么这仍然是真学习。

如果 AI 写了代码，用户只知道“它跑起来了”，但不知道为什么能跑、为什么会错、需求变了该改哪里，那么这还没有掌握。

## 3. 掌握的判断标准

后续课程不以“代码由谁生成”判断是否学会，而以以下能力判断。

### 能解释

能解释：

- 这个模块解决什么真实问题。
- 为什么需要这个设计。
- 输入是什么，输出是什么。
- 数据流、状态流、异常流分别怎么走。
- 为什么不用更简单或更复杂的方案。

### 能验证

能验证：

- 最小实现是否跑通。
- 关键输入输出是否符合预期。
- RAG 是否命中正确来源。
- Agent 是否调用正确工具。
- Workflow 是否走到正确节点。
- 修改前后质量是否变好。

### 能修改

能在需求变化时判断：

- 应该改 Prompt、schema、retriever、tool、Agent、workflow，还是前端展示。
- 哪些改动影响数据结构。
- 哪些改动需要补评估样本。
- 哪些改动可能破坏已有功能。

### 能调试

能定位问题来自哪里：

- 文档解析。
- chunk。
- metadata。
- embedding。
- retriever。
- prompt。
- model。
- tool schema。
- tool runtime。
- agent decision。
- workflow state。
- frontend state。

### 能迁移

能把方法迁移到其他业务场景：

- 需求评审。
- 客服知识助手。
- 合同审查。
- 内部知识库。
- 测试用例生成。
- 研发提效工具。

迁移不是复制代码，而是迁移问题判断、系统分层、评估方法和调试路径。

### 能取舍

能判断：

- 什么时候不用 Agent。
- 什么时候固定 RAG 就够。
- 什么时候需要单 Agent RAG。
- 什么时候需要 Multi-Agent。
- 什么时候必须引入 Human-in-the-loop。
- 什么时候平台化过早。

## 4. 什么叫真正掌握 Agent 开发

真正掌握 Agent 开发，不是会背 LangChain、LangGraph 或 Tool Calling API，而是能回答这些问题：

- 这个需求为什么需要 Agent，而不是普通 RAG 或固定 Workflow？
- Tool schema 怎么设计，参数怎么校验，失败怎么处理？
- Tool runtime 如何做权限、确认、审计和重试？
- Agent 什么时候该继续，什么时候该停止，什么时候该追问？
- Multi-Agent 是真的有职责拆分，还是只是把一个 Prompt 拆成多个 Prompt？
- Workflow state 里应该放什么，不应该放什么？
- Human-in-the-loop 为什么在这个节点出现，而不是最后补一个按钮？
- 一次结果错了，怎么判断是检索错、Prompt 错、工具错、Agent 决策错，还是数据错？
- 成本和延迟变高了，怎么定位和优化？
- 公司换了业务场景，能否迁移这套方法？

掌握 Agent 开发的核心不是写出复杂 Agent，而是能构建可控、可观测、可评估、可修改的 AI 应用系统。

## 5. AI Coding 下的学习流程

后续课程和项目可以大量使用 AI Coding，但应采用下面的协作方式：

```text
先判断问题
-> 先设计结构
-> 再让 AI 辅助实现
-> 审查代码
-> 运行验证
-> 解释代码
-> 修改需求
-> 复盘失败
```

不推荐的方式是：

```text
把需求直接交给 AI
-> 接受生成结果
-> 跑通
-> 认为已经学会
```

AI 是执行加速器，不是责任替代者。

## 6. 每个代码任务的完成标准

后续课程代码或项目代码，不能只满足“跑通”，还要满足以下标准。

### 结构理解

- 能画出模块结构。
- 能说清入口文件。
- 能说清核心函数或类的职责。
- 能说清依赖关系。

### 流程理解

- 能解释数据流。
- 能解释状态流。
- 能解释异常流。
- 能解释日志和 trace 如何关联。

### 设计理解

- 能说明为什么这样拆模块。
- 能说明为什么使用这个框架或不用框架。
- 能说明哪些地方是可替换的。
- 能说明哪些地方暂时简化了。

### 修改能力

- 能增加一个字段。
- 能增加一个工具。
- 能增加一个 workflow 节点。
- 能调整一个 Agent 角色。
- 能替换一个 retriever 或模型配置。

### 调试能力

- 能根据错误信息定位模块。
- 能根据 trace 找到失败步骤。
- 能构造最小复现样例。
- 能判断修复后是否需要回归测试。

## 7. 防止掌握幻觉的检查清单

每完成一个阶段，应至少选择若干问题自检：

- 关掉 AI，我能否手写一个最小版本？
- 我能否用自己的话解释代码入口和主链路？
- 我能否指出 AI 生成代码的 3 个风险？
- 我能否给这段代码增加一个字段、工具、节点或状态？
- 我能否故意制造一个错误并定位它？
- 我能否把这个能力迁移到另一个业务场景？
- 我能否解释为什么这里不用更复杂的 Agent？
- 我能否说明这段代码如何测试、如何观测、如何回归？

如果这些问题答不上来，说明项目可能完成了，但能力还没有真正沉淀。

## 8. 面对公司真实环境的迁移能力

公司里的真实项目通常不会像课程大纲这样理想化：

- 产品需求会变化。
- 数据会很脏。
- 老系统很难接。
- 权限和流程会复杂。
- 模型效果会不稳定。
- 架构会受历史包袱影响。
- 业务方只关心结果是否可用。
- 很多时候必须做取舍和妥协。

因此，课程真正训练的不是记住某个完美架构，而是：

- 判断问题类型。
- 设计可演进结构。
- 用 AI 加速实现。
- 用评估证明质量。
- 用 trace 定位问题。
- 在约束下做取舍。
- 在需求变化时知道改哪里。

这才是岗位中可迁移的核心能力。

## 9. 后续课程执行约定

以后从零开始推进课程和项目时，AI Agent 应遵守以下约定。

### 代码之前

- 先说明真实问题。
- 先说明最小闭环。
- 先说明模块边界。
- 先说明为什么这样设计。

### 代码之中

- 生成代码要服务理解，不为炫技。
- 关键路径要可读、可调试、可替换。
- 保留必要日志、sources、scores、tool calls、state transitions。
- 不为了章节完整性造无意义脚本。

### 代码之后

- 说明如何运行。
- 说明如何验证。
- 说明如何修改。
- 说明常见错误和定位方式。
- 说明如果需求变化，应该改哪些模块。
- 必要时做 code review，而不是只宣布完成。

## 10. 课程阶段的代码所有权要求

每个课程阶段结束时，应尽量补齐以下材料：

- 架构图或模块说明。
- 主链路说明。
- 关键代码解释。
- 可运行入口。
- 最小验证方式。
- 常见异常与排查路径。
- 一个定制化改造练习。
- 一个 bad case 或失败复盘。

这些材料的目标不是增加文档负担，而是确保用户拥有代码所有权。

## 11. 最终目标

本仓库使用 AI Coding 的目标，不是训练用户手写每一行代码，而是训练用户成为：

> 能用 AI 和工程工具，把一个不确定的业务问题落成可运行、可评估、可迭代 AI 应用的人。

AI 可以生成 99% 的代码。

但系统责任必须 100% 由用户理解和接住。
