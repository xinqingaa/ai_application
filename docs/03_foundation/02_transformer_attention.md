## 2. 为什么会出现 Transformer 📌

### 2.1 在 Transformer 之前，序列模型有什么限制

在 Transformer 之前，RNN / LSTM 是更常见的序列模型思路。

应用开发者不需要掌握它们的实现细节，但要知道它们在大规模场景下有几个明显限制：

1. 难并行
2. 长距离依赖处理困难
3. 序列越长，信息越容易衰减

如果把一句很长的话、一个长段落、甚至一整篇文档交给模型处理，这些限制都会放大。

### 2.2 Transformer 为什么更适合大模型

Transformer 的核心意义不在于“结构更复杂”，而在于它更适合解决两类问题：

1. 更高效地训练大规模模型
2. 更灵活地处理长距离上下文关系

这两点正好对应今天 LLM 的两个现实需求：

- 规模要足够大
- 上下文要足够长

所以今天主流大模型基本都建立在 Transformer 系列结构上，并不是偶然。

### 2.3 对应用开发者最重要的结论

你不需要记 Transformer 的模块列表，但要记住：

- 现代 LLM 天生是围绕“如何利用上下文”来工作的

这就是后面为什么：

- Prompt 很关键
- RAG 很关键
- 上下文压缩很关键
- 记忆管理很关键

因为这些本质上都在优化模型的“阅读材料”。

## 3. Attention 的最小直觉 📌

### 3.1 模型不是平均地看所有上下文

这是这一章最重要的一句判断。

当模型生成当前 token 时，它并不是：

- 把前面所有内容一视同仁地平均处理

而是会根据当前需求，对上下文中的某些位置、某些词、某些模式给予更高关注。

这就是 Attention 的最小直觉。

### 3.2 为什么这个直觉很重要

因为它直接解释了很多你在工程里看到的现象：

- 同一份材料，换个顺序结果就变了
- 把关键规则放前面比埋在后面更稳
- 无关上下文会污染结果
- 文档越多不一定越好

换句话说，模型不是“读完整份材料以后统一总结”，而是在生成过程中不断重新判断：

- 当前最值得参考的上下文是什么

### 3.3 一个直观例子

比如这句话：

- “小王把合同交给小李，因为他是法务。”

如果模型要理解“他”指的是谁，就必须在上下文里判断：

- 当前这个代词更可能关联谁

这类“相关性选择”就是 Attention 在做的事。  
虽然真实模型比这个复杂很多，但工程层面的直觉已经足够了。

## 4. Query / Key / Value 怎么直觉理解 📌

### 4.1 不看公式时的理解方式

在应用开发者视角里，可以把它们简单理解成：

- Query：当前 token 现在需要什么信息
- Key：上下文中每个位置大概在声明“我这里是什么类型的信息”
- Value：如果匹配成功，真正被带入后续计算的内容

这不是严格数学定义，但足够支持你做工程判断。

### 4.2 为什么这种理解对工程有帮助

因为它会让你意识到：

- 模型不是简单扫描文本
- 模型是在根据当前需求做相关性选择

这意味着你写 Prompt 时要思考的不是：

- “我有没有把话写进去”

而是：

- “我有没有把最关键的信息放到更容易被关注的位置”

### 4.3 对 RAG 的直接启发

RAG 中常见错误不是“检索不到”，而是：

- 检索到了，但组织方式差

例如：

- 片段顺序混乱
- 标题缺失
- 关键元信息没带
- 无关内容和关键内容混在一起

这会让模型在注意力分配时更难稳定聚焦到正确依据。

## 5. 这和 Prompt / RAG / 长上下文到底有什么关系 📌

### 5.1 Prompt：不是把话说出来就够了

如果模型的上下文处理依赖相关性选择，那 Prompt 的作用就不只是“表达任务”，还包括：

- 组织任务结构
- 组织阅读顺序
- 提供显式约束
- 突出关键条件

这就是为什么这些写法通常更稳：

- 先写任务目标
- 再写限制条件
- 再写输出格式
- 最后给输入材料

而不是把一切混成一段自然语言。

### 5.2 RAG：检索只是第一步

很多人学 RAG 时，容易把重点都放在“怎么检索”。

但在模型看来，真正进入推理的是：

- 检索后的上下文

所以完整 RAG 至少包含两步：

1. 找到相关材料
2. 组织这些材料进入上下文

如果第二步做得差，即使检索本身没有问题，效果也可能不好。

### 5.3 长上下文：解决的是容量，不是组织

长上下文模型的价值在于：

- 你能放进更多材料

但它不自动解决：

- 哪些材料最相关
- 哪些内容该优先关注
- 哪些噪声会干扰结果

所以长上下文不是 RAG 的简单替代品。

更准确的理解应该是：

- 长上下文提供容量
- RAG 提供选择
- 上下文组织提供可读性

### 5.4 上下文压缩：不是节省字数，而是提高可用性

压缩上下文的目的不只是省 token，而是：

- 去掉噪声
- 提高相关密度
- 让模型更容易聚焦

所以后面你做摘要记忆、检索结果拼接、聊天历史保留策略时，都要用这个视角。

## 6. 常见误区与应用开发者的正确理解 📌

### 6.1 误区一：上下文越多越好

现实中更常见的情况是：

- 上下文越多，噪声越多
- 结果越不稳定
- 成本越高

### 6.2 误区二：长上下文出来以后 RAG 就没用了

错。

长上下文解决的是“放得下”，RAG 解决的是“挑得对”。

这两个经常是互补关系，而不是替代关系。

### 6.3 误区三：只要把检索结果拼进去，模型就会自己理解

模型的确很强，但它不会自动替你完成：

- 信息排序
- 关键信号突出
- 依据区分
- 噪声清洗

这些都是系统设计的一部分。

### 6.4 本章真正要建立的判断

你应该形成这种稳定认知：

> 对大模型来说，上下文不是附件，而是推理现场。上下文怎么组织，往往就决定了结果会不会稳定。

## 8. 本章小结

这一章你真正应该带走的是：

1. Transformer 更适合处理大规模上下文关系
2. Attention 的核心直觉是“模型不会平均地看所有上下文”

5. 长上下文不是万能替代，RAG 和重组仍然有价值
