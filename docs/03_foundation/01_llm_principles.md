## 2. LLM 的本质：预测下一个 Token 📌

### 2.1 为什么一句话就能概括很多能力

LLM 最核心的任务可以概括为：

- 在给定上下文的前提下，预测下一个最可能出现的 Token

这句话表面上很简单，但它解释了很多现象：

- 为什么模型能聊天
- 为什么模型能总结
- 为什么模型能翻译
- 为什么模型能写代码
- 为什么模型有时也会胡说

真正需要理解的是：

> 这些能力并不是模型内部存在很多“专门功能按钮”，而是模型在大量语料和任务模式上学会了如何继续生成更符合上下文的 token 序列。

### 2.2 它不是数据库查询

这点必须尽早建立正确认知。

当你问模型一个问题时，模型不是：

- 去数据库里按主键查一条标准答案

模型真正做的是：

- 根据你给的上下文
- 根据它在训练中学到的语言模式和知识模式
- 继续生成一串更“合理”的 token

这意味着它天然具备：

- 泛化能力
- 表达能力
- 风格迁移能力

也意味着它天然存在：

- 不确定性
- 误差
- 幻觉
- 格式漂移

如果这一点没有想明白，后面很容易出现两个极端：

1. 过度神化模型，什么都想直接问
2. 一旦模型答错，就觉得“它不可靠所以没法用”

更成熟的理解应该是：

- 模型很强
- 但模型能力本身不等于产品能力
- 产品能力来自“模型能力 + 工程约束 + 外部知识 + 工作流设计”

### 2.3 为什么预测 token 还能表现出任务能力

很多初学者的困惑在于：

- 如果只是预测下一个 token，为什么还能做分类、抽取、总结、问答？

原因是模型在训练数据中见过大量模式，例如：

- 问答模式
- 列表模式
- JSON 模式
- 角色对话模式
- 推理解释模式

所以当你写：

- “请总结下面内容”
- “请输出 JSON”
- “请提取字段”
- “请扮演客服”

你其实是在给模型构造一种高概率模式，让它沿着这种模式继续生成。

这也解释了为什么：

- Prompt 有效
- Few-shot 有效
- 角色设定有效
- 输出格式约束有效

但它们也永远不可能做到完全 deterministic。

### 2.4 作为应用开发者需要理解到什么程度

你现在不需要做到：

- 推导训练目标
- 深入模型架构细节
- 复现论文

你必须做到：

- 知道模型不是数据库
- 知道模型是概率生成系统
- 知道 Prompt 和上下文在推理阶段就是任务注入手段
- 知道不稳定性不是偶发 bug，而是系统特性

这会直接影响你后面怎么设计：

- Prompt
- 检索
- 记忆
- 审核
- 工具调用
- 人工接管

## 3. 从输入到输出：一次模型调用到底发生了什么 📌

### 3.1 最小数据流

从应用开发者视角，可以把一次调用简化成下面这条链：

```plain
用户文本 -> tokenizer -> token ids -> model -> next token prediction -> decode -> 文本输出
```

这条链虽然简化，但已经足够解释大部分工程问题。

### 3.2 Token 化意味着什么

模型不是直接处理原始自然语言，而是先把输入切成 token。

这在工程上直接带来三个现实影响：

1. 相同字数不代表相同成本
2. 中文、英文、代码的 token 比例可能差很多
3. “看起来很短”的输入，实际 token 消耗未必低

所以你后面做：

- 成本估算
- 上下文裁剪
- 预算控制

都必须基于 token 视角，而不是字符视角。

### 3.3 输出为什么天然适合流式

模型不是先想完再一次性返回全文，而是一步一步生成 token。

所以流式响应并不是某种额外外挂能力，而是生成过程本身很自然的暴露方式。

这也解释了：

- 为什么流式体验对聊天产品很合适
- 为什么流式输出有时会先吐出不完整句子
- 为什么中途中断是合理能力

### 3.4 为什么上下文越长越贵

每次调用模型时，上下文都要被重新纳入计算。

这意味着：

- 上下文越长，费用越高
- 上下文越长，时延越高
- 上下文越长，不相关信息越容易污染输出

所以“把所有历史都带进去”通常不是好策略。

这正是后面这些能力存在的原因：

- RAG
- 上下文压缩
- 记忆分层
- 检索而不是全量拼接

### 3.5 为什么模型会跑偏

因为生成是逐步进行的，一旦前面几步进入了错误轨迹，后面就可能沿着错误方向继续展开。

这就是为什么工程上要强调：

- 明确任务
- 限定格式
- 缩小自由度
- 增加校验
- 必要时人工复核

## 4. 为什么 Prompt 会影响这么大 📌

### 4.1 Prompt 不是“备注”，而是输入的一部分

对模型来说，Prompt 不是开发者写在旁边的说明，而是推理时真正参与计算的上下文。

所以 Prompt 一变，模型对“下一个 token 应该是什么”的判断也会变。

这就是为什么：

- 同一个问题，换一种说法，结果可能差很多
- 加一个 JSON 格式要求，输出就会更收敛
- 加 few-shot 示例后，风格和格式会明显稳定

### 4.2 Prompt 的作用不是让模型“更聪明”

更准确地说，Prompt 的作用是：

- 明确任务边界
- 收缩输出空间
- 提供格式先验
- 注入必要上下文

所以你后面写 Prompt 时要形成这种意识：

- Prompt 不是文案
- Prompt 是程序的一部分

这也是为什么在工程里它应该：

- 独立管理
- 版本化
- 模块化

### 4.3 Prompt 能解决什么，不能解决什么

Prompt 很适合解决：

- 任务描述不清
- 输出风格不稳
- 格式要求不明确
- 角色语气不统一

Prompt 不适合独立解决：

- 外部知识缺失
- 高频知识更新
- 复杂动作执行
- 长流程状态管理

这就是后面需要引入：

- RAG
- Tool
- Agent / Workflow

的原因。

## 5. 为什么结构化输出、检索、工具调用会更稳定 📌

### 5.1 结构化输出：减少自由表达空间

自由文本的优点是灵活，但缺点是：

- 难以消费
- 难以校验
- 难以稳定

结构化输出通过：

- 字段约束
- 类型约束
- 格式约束

把模型输出拉回到更可控的空间里。

所以你在工程里经常会看到：

- JSON Schema
- Pydantic
- Structured Outputs

它们的本质都不是“更高级的玩法”，而是：

- 为了让概率系统更适合程序消费

### 5.2 检索：不是让模型更聪明，而是让上下文更对

模型本身有通用知识，但它不知道你当前任务最需要哪部分外部信息。

RAG 做的事情不是“增强大脑”，而是：

- 先找到更相关的外部知识
- 再把它放进本轮上下文

所以检索的真正价值是：

- 提高相关性
- 提供依据
- 降低知识过期风险

### 5.3 工具调用：把精确执行交给确定性系统

模型适合：

- 理解问题
- 组织语言
- 解释结果

模型不适合独立完成：

- 精确计算
- 访问外部系统
- 执行数据库查询
- 处理需要确定性的动作

这时候最合理的方式不是逼模型自己算，而是：

- 让模型决定“是否需要工具”
- 让工具做精确执行
- 再由模型组织最终输出

这就是为什么 Tool Calling 往往比“纯聊天”更可靠。

## 6. 常见误区与应用开发者的正确理解 📌

### 6.1 误区一：模型错了，所以不能用于产品

更准确的理解是：

- 模型本来就不是确定性知识引擎
- 产品设计应该围绕这种不确定性来构建兜底

### 6.2 误区二：上下文越多越好

实际上常见情况是：

- 上下文越多，噪声越多
- 成本更高
- 结果反而更差

### 6.3 误区三：Prompt 足够强就能解决一切

Prompt 很重要，但它不能代替：

- 检索
- 工具
- 工作流
- 审核机制

### 6.4 误区四：模型输出不稳定说明接口有问题

接口问题当然可能存在，但更常见的是：

- 任务本身缺少约束
- 输出格式缺少约束
- 上下文组织不合理
- 外部知识没有正确注入

### 6.5 这一章真正要建立的判断

你要形成的稳定认知是：

> 模型是概率生成器，所以 AI 产品必须通过 Prompt、结构化输出、检索、工具、工作流和人工兜底来把它收束成可用系统。
