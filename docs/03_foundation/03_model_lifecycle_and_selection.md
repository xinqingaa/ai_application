## 2. 模型生命周期：从训练到推理 📌

### 2.1 预训练：通用能力从哪里来

预训练阶段，模型在大规模语料上学习：

- 语言模式
- 知识模式
- 表达结构
- 推理痕迹

这一阶段决定的是模型的通用能力上限。

你要理解的是：

- 模型之所以“好像什么都懂一点”，主要来自预训练

但预训练并不保证：

- 会按你的任务格式输出
- 会像助手一样听指令
- 会遵守你的业务边界

### 2.2 指令微调：为什么聊天模型更像助手

指令微调会把模型从“擅长续写文本”，变成更像“听任务做事”的助手。

这一步通常会强化模型在这些方面的表现：

- 对话形式
- 任务跟随
- 问答风格
- 结构化任务模式

所以你平时调用的“聊天模型”，往往已经不是纯基座模型，而是做过这一步的版本。

### 2.3 对齐与偏好优化：为什么不同模型行为差异很大

这一阶段主要解决：

- 更符合人类偏好
- 更安全
- 更听话

你不需要掌握 RLHF / DPO 的算法，但必须知道：

- 为什么不同模型在“是否保守”“是否容易自由发挥”“是否容易越界”上差异明显

这也是后面做模型选择时必须考虑的因素。

### 2.4 推理：你当前真正工作的地方

对于 AI 应用开发者来说，最直接相关的是推理阶段。

推理阶段你能控制的主要是：

- Prompt
- 消息组织
- 上下文裁剪
- 结构化输出
- 检索
- 工具调用
- 工作流

这意味着你现在绝大多数真正能产生项目价值的工作，不在训练侧，而在推理侧。

这也是为什么当前路线是：

- 先会调模型
- 再会设计上下文
- 再会做 RAG 和 Agent
- 最后才考虑微调和私有化

## 3. 为什么很多任务优先靠 Prompt 就够了 📌

今天的大模型已经具备很强的通用能力和对齐能力，因此很多任务里，最先该做的不是“训练”，而是：

- 把任务描述清楚
- 把输出格式约束清楚
- 把上下文组织清楚

所以工程上的常见顺序通常是：

1. 先改 Prompt
2. 再加结构化输出
3. 再引入检索
4. 再做工作流拆解
5. 最后才考虑微调

这个顺序很重要，因为它决定了成本和迭代速度。

如果一开始就想着训练或私有化，往往是在高成本阶段解决一个原本可以低成本解决的问题。

## 4. Prompt、RAG、长上下文、微调怎么选 📌

### 4.1 Prompt 适合解决什么

Prompt 更适合解决：

- 任务目标不清
- 输出风格不稳
- 角色边界不清
- 结构和格式要求不明确

它不适合优先解决：

- 高频变化的外部知识
- 大规模私域知识注入

### 4.2 RAG 适合解决什么

RAG 更适合解决：

- 外部知识需要动态注入
- 知识更新频繁
- 需要依据引用
- 需要结果可追溯

所以企业知识问答、规则问答、文档问答通常优先考虑 RAG。

### 4.3 长上下文适合解决什么

长上下文更适合：

- 单次材料很长
- 低频高价值分析任务
- 临时材料理解

例如：

- 一次性分析 50 页材料
- 对单份财报做重点提炼
- 临时比较多份说明文档

它不适合当万能方案，因为：

- 成本高
- 时延高
- 噪声也会被一起放大

### 4.4 微调适合解决什么

微调更适合：

- 固定风格强化
- 稳定任务模式
- 特定格式学习
- 领域化表达风格迁移

它不适合优先解决：

- 知识实时更新
- 检索缺失
- 工作流拆解不足

### 4.5 一个实用判断表

| 问题类型 | 优先方案 |
|---------|---------|
| 输出格式不稳 | Prompt + Structured Output |
| 私域知识问答 | RAG |
| 超长材料一次性分析 | 长上下文 |
| 固定风格、固定格式长期输出 | 微调候选 |
| 需要多步动作和工具 | Workflow / Agent |

## 5. 企业里最常见的误判 📌

### 5.1 误判一：知识更新问题用微调解决

如果问题本质是：

- 文档常变
- 规则常变
- 知识需要经常更新

那优先考虑的通常应该是：

- RAG

而不是微调。

### 5.2 误判二：行为不稳就去做 RAG

如果问题本质是：

- 输出格式不稳定
- 角色风格不稳定
- 安全边界不稳定

那优先该改的是：

- Prompt
- Schema
- 工作流

而不是先补知识。

### 5.3 误判三：长上下文可以替代一切

长上下文很强，但它不能替代：

- 检索
- 依据管理
- 工作流控制
- 人工接管

### 5.4 误判四：模型不够强就一切都没救

模型能力当然重要，但很多项目中的实际问题，优先不是模型本身，而是：

- 任务没拆清
- 上下文没组织好
- 输出没约束
- 系统没兜底

## 6. 开源模型、本地推理、私有化需要理解到什么程度 📌

### 6.1 API 模型适合什么阶段

对你现在来说，API 模型更适合：

- 学习
- 快速验证
- 低成本迭代
- 快速对比模型差异

这也是你当前优先用百炼这类平台跑通工作流的原因。

### 6.2 本地推理适合什么

本地推理更适合：

- 体验开源模型
- 初步理解私有化可行性
- 做敏感数据和成本边界的早期讨论

但不应该在当前阶段成为主线投入点。

### 6.3 真正私有化适合什么

真正私有化通常适合：

- 数据合规要求非常高
- 成本模型和并发需求可预测
- 团队有平台和运维能力

这不是你当前阶段最优先突破的事情。

### 6.4 你当前阶段应该掌握的边界

你不需要现在就会：

- 部署 vLLM 集群
- 训练 LoRA 模型
- 调优显存和吞吐

你需要现在会判断：

- 什么场景值得考虑私有化
- 什么阶段还应该优先 API 调用
- 什么问题不是“换部署方式”能解决的

## 7. LoRA / QLoRA 为什么要知道但现在不用做

LoRA / QLoRA 的价值是：

- 降低微调成本
- 让特定任务和风格强化更可行

你现在知道它们就够了，因为后续在阅读项目、面试、选型讨论时会经常见到。

但对你当前路线来说，更优先的是：

- Prompt
- RAG
- Agent / Workflow
- 评测和可观测性

如果把顺序搞反，很容易在错误的阶段做了成本最高的事。

这一章你真正应该带走的是：

1. 模型生命周期大致可以理解为：预训练 -> 指令微调 -> 对齐 -> 推理

6. 选型错误，往往比实现不熟更伤项目
