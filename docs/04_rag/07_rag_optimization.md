## 2. 为什么评估必须前置 📌

### 2.1 为什么不能只凭感觉调 RAG

RAG 里能改的变量非常多：

- `chunk_size`
- `chunk_overlap`
- Embedding 模型
- 检索参数
- Prompt
- Rerank

如果没有固定基线，你很难回答：

- 到底有没有变好
- 变好的是检索还是生成
- 成本是不是也一起变高了

### 2.2 最小 Golden Set 应该长什么样

一个最小 Golden Set 不需要一开始就很大，但至少应该包含：

- `question`
- `expected_answer`
- `expected_sources`

更进一步，还可以标记：

- 多跳问题
- 模糊问题
- 无答案问题
- 需拒答问题

## 3. 第七章要建立哪些评估意识 📌

### 3.1 检索评估

检索评估主要关心：

- 正确来源是否被召回
- 排名是否合理
- 召回是否稳定

常见指标包括：

- Recall
- MRR
- 来源命中率

### 3.2 生成评估

生成评估主要关心：

- 答案是否准确
- 是否完整
- 是否正确引用
- 无答案时是否保守

### 3.3 端到端评估

端到端评估更关心业务体验：

- 用户问题是否被真正解决
- 响应是否足够快
- 成本是否可接受

所以第七章的重点不是“选一个万能指标”，而是：

> 先建立一套最小可重复的实验尺子。

---

## 4. 坏案例为什么比好案例更重要 📌

好案例只能说明系统“有时候能答对”。

坏案例才能推动你回答：

- 到底是哪类问题失败
- 失败发生在检索、生成还是知识源本身
- 下一轮应该改哪一层

坏案例通常应该沉淀成：

- 固定问题
- 当前错误表现
- 正确来源或正确答案
- 失败类型判断

## 5. 第七章实现后应该怎么学

等第七章真实代码快照落地后，建议按这个顺序：

1. 先看 Golden Set 数据结构
2. 再跑基础评估器
3. 再做一轮配置对比
4. 最后记录坏案例并归类

本章最终完成标准应该是：

- 能用固定样本重复评估
- 能说明某次优化是否真的变好
- 能记录坏案例并驱动下一轮修改
