这份文档不是课程正文，而是当前 `04_rag` 课程的修订总计划。

1. 明确当前版本到底出了什么问题
2. 明确后续修复时要遵守的边界、顺序和目标结构

后续新开对话时，可以直接以这份文档为上下文入口，不必重新解释本轮审查结论。

## 3. 当前现状判断

当前 `01-04` 不是“坏掉了”，而是已经演化成另一套课程产品形态：

- 章节独立闭环
- 每章最小实现
- 每章自带 README 和测试
- 尽量避免跨章节工程耦合

这条路线本身是合理的，符合当前学习目标：

- 不希望工程化过早占用注意力
- 不希望前一章目录结构持续吞噬下一章上下文
- 不希望为了铺垫未来章节而削弱当前章节知识点

问题不在于“独立章节思路错误”，而在于：

- 当前大纲仍然描述的是“完整版渐进主线”
- 现版文档和代码已经切成“独立极简主线”
- 两者没有被重新对齐

因此当前最核心的问题是：

> 课程规格没有被重新定义，导致大纲、文档、代码各自代表不同版本的课程。

## 5. 当前存在的问题

### 5.1 课程规格冲突

当前同时存在三套不同层次的“真相”：

1. `outline.md`
   - 描述的是覆盖更完整、工具更真实、从评估到 Chroma 再到 LangChain 的主线课程
2. `docs/04_rag/01-04`
   - 描述的是独立闭环、最小实现、强边界控制的学习版课程
3. `source/04_rag/01-04`
   - 实现的是与学习版课程相匹配的 toy 或最小实现

结果是：

- 用户看大纲会期待某些内容
- 打开章节文档和代码后却发现实际不是那套东西
- 最终形成“文档承诺”和“课程交付”不一致

### 5.2 第 1 章缺少真正的课程级评估前置

大纲在课程开头要求：

- 先建立最小评估集
- 记录实验配置
- 用同一套样本回归后续调整

当前现状：

- 第 1 章有 mini golden set
- 但它仍然是章节内部回归
- 没有真正成为整个 `01-04` 的共享评估基线

影响：

- 章节独立是成立的
- 但课程主线缺少统一实验尺子

### 5.3 第 2 章覆盖范围偏窄

当前第 2 章主要覆盖：

- 本地文件发现
- 文本规范化
- chunk 切分
- metadata
- stable id

但相对大纲，仍缺少：

- PDF / 更真实 Loader 认知
- `DirectoryLoader` 级别的批量加载思路
- `MarkdownHeaderTextSplitter` 级别的结构感知切分
- 通用 `DocumentPipeline` 交付视角
- 生命周期治理与知识库运营意识的更强落点

说明：

- 当前第 2 章基础主线是对的
- 但需要补“扩展输入层”和“面向后续工程的最小治理意识”

### 5.4 第 3 章停留在 toy embedding 层

当前第 3 章已经很好地讲清楚了：

- `SourceChunk -> EmbeddedChunk`
- provider 契约
- query/document 两个入口
- 同一向量空间约束

但相对大纲，仍缺少：

- 真实 Embedding API 接入
- OpenAI / HuggingFace 类接口认知
- 模型选择与维度差异的最小实战感知
- 从 toy provider 过渡到真实 provider 的桥接

说明：

- 第 3 章的原理层没问题
- 缺的是“真实接入桥”

### 5.5 第 4 章与大纲偏差最大

当前第 4 章主要实现了：

- 本地 JSON 持久化
- `upsert()`
- `replace_document()`
- `similarity_search()`
- `delete_by_document_id()`
- `filename` 过滤

这对讲清“存储层职责”是有价值的。

但相对大纲，缺口很明显：

- 没有 `Chroma`
- 没有 `LangChain VectorStore`
- 没有 `from_documents / similarity_search / as_retriever`
- 没有“主流向量数据库差异”落到代码层
- 没有真正兑现“向量数据库”章节名在大纲中的承诺

这部分是必须补的，不然第四章的课程名和交付内容不匹配。

### 5.6 文档与代码之间的绑定还不够强

虽然当前已有 README 和脚本顺序，但仍有几个问题：

- 不是每个文档小节都严格映射到脚本
- 某些知识点只出现在文档，没有对应实验
- 某些代码脚本存在，但文档中的教学顺序还不够细
- 当前章节之间的桥接更多靠说明文字，不够靠“具体运行闭环”

目标应该是：

> 学员阅读某一小节后，立刻知道该跑哪个脚本、观察什么现象、为什么它属于本章边界。

### 5.7 存在维护质量问题

目前还存在一些次级但明确的问题：

- 文档中有硬编码到旧绝对路径 `/Users/linruiqiang/...`
- 路径引用与当前机器工作区不一致
- 这会影响本地跳转和后续维护一致性

这类问题优先级不如课程结构，但在修订过程中应一起清理，改为相对路径。

## 7. 结构决策

### 7.1 文档结构必须线性

最终文档结构保持线性，如果内容过多，可以选择新建文件夹，但是必须保持01到09这类的递增：

```text
docs/04_rag/
├── outline.md
├── 01_rag_basics.md
├── 02_document_processing.md
├── 03_embeddings.md
└── 04_vector_databases.md
```

说明：

- 不新增 `00_xxx.md`
- 不使用 `02b / 03b / 04a / 04b` 这种编号分叉
- 如果某章内容变多，也优先在同一文件中做“主线 + 扩展小节”
- 只有当某章真的无法在一个文件里维持可读性时，才考虑该章建立子目录

当前阶段默认不拆子目录。

### 7.2 代码结构必须镜像文档结构

最终代码主线保持：

```text
source/04_rag/
├── 01_rag_basics/
├── 02_document_processing/
├── 03_embeddings/
├── 04_vector_databases/
└── labs/
```

说明：

- `labs/` 保留但降级
- 01-04 才是主学习目录
- 每章代码目录内部按照学习顺序递增命名

### 7.3 每章内部统一采用顺序脚本

每章建议采用：

- `01_*.py`
- `02_*.py`
- `03_*.py`
- `04_*.py`
- `05_*.py`

是否需要到 `05`，由该章实际复杂度决定。

原则：

- 文件名按学习顺序递增
- 不使用 `phase_x`
- 不建立脱离文档顺序的技术文件堆

### 7.4 文档与代码必须强绑定

每章必须同时存在：

- 主文档：完整教学说明
- README：本章运行入口
- 顺序脚本：跟着文档一步步运行
- tests：锁定本章最小闭环
- data：本章最小样例输入

## 10. 代码模板规范

每章 README 统一采用以下结构：

1. 本章核心原则
2. 项目结构
3. 先跑哪个
4. 第 1 步
5. 第 2 步
6. 第 3 步
7. 第 4 步（如需要）
8. 最后看测试锁定什么
9. 建议学习顺序
10. 当前真实进度和下一章

每个脚本文件建议满足：

- 文件头注释说明“对应文档哪一节”
- 只承担一个清晰教学目标
- 输出可读、可观察
- 不混入下一章职责

每章测试建议锁定：

- 最小 happy path
- 至少一个边界失败案例
- 至少一个 mini golden set 或稳定样例

### 10.1 教学复制说明规范

本课程允许为了教学目的复制代码，但复制必须是显式、可解释、可追踪的。

这里反对的不是“复制”，而是：

- 复制后不说明来源
- 复制后不说明原因
- 复制后不说明新增改动
- 复制后逐渐分叉，最终课程自己失去一致性

如果某一章复制了前一章的代码、对象定义或最小实现，必须同时写清楚四件事：

1. `copied_from`
   - 复制自哪个章节、哪个文件
2. `copy_reason`
   - 为什么本章不直接依赖前一章实现，而要保留独立副本
3. `chapter_delta`
   - 本章在复制后新增、修改或强调了什么
4. `extension_target`
   - 这份副本是为了支撑本章哪个知识点或哪段扩展

落地要求：

- 主文档要说明为什么这一章重新保留一份实现
- README 要说明这份副本和前一章的关系
- 关键脚本文件头建议注明复制来源和本章增量
- 测试要体现“本章比前一章多验证了什么”

原则上：

- 可以不共享运行时代码
- 但必须共享概念定义、字段契约和课程叙述标准
- 学员必须看得出“为什么这里重新写一份”

## 16. 风险点

### 16.1 重新补内容后再次失控

风险：

- 为了补大纲，章节重新变成“大而杂”

控制方式：

- 每章都坚持“主线 + 明确边界”
- 只补最关键的桥接内容，不补全所有扩展可能性，但是需要在文档说明清楚哪些是“核心内容”，哪些是“扩展内容”，并且扩展内容不破坏主线的独立性。

### 16.2 第四章真实工具接入导致复杂度骤增

风险：

- 一引入 Chroma / LangChain，第四章又走向工程化过载，

控制方式：

- 保留原理脚本作为前半段
- 真实工具脚本只演示小范文闭环，完整实例工程需要在文档说明，指明在后续章节完整实现，对比当下和完整的差异，预留并指导扩展空间和方向。

### 16.3 教学复制与实现漂移之间的边界不清

风险：

- 为了保持章节独立，章节之间会复制部分代码、数据结构或最小实现
- 复制本身并不是问题，真正的风险是复制后没有说明来源、目的和改动点，最终导致实现漂移
- 学员如果看不清“这一章为什么重新写一份”，就会误以为课程结构混乱，或者误以为这些代码应该天然共用

控制方式：

- 允许为教学目的复制代码，但必须是显式复制，不能是隐式分叉
- 每次复制都要说明：
  - 从哪里复制
  - 为什么这一章要保留独立副本
  - 这一章在复制后新增或修改了什么
  - 这份副本服务于本章哪一个知识点
- 如果两章都使用相似对象或流程，要共享“概念定义”和“字段契约”，不强求共享运行时代码
- README 和主文档都要明确告诉学员：
  - 这份代码是否来自前一章的简化迁移
  - 当前这份代码的教学边界是什么
  - 如果以后继续扩展，应该从哪一份开始，而不是默认把所有副本同时维护
