## 2. 什么情况下固定 RAG 已经不够 📌

固定 `2-step RAG` 很适合：

- 单跳问答
- 文档问答
- FAQ、帮助中心、规则库
- 来源可追溯的标准知识问答

但当你遇到这些问题时，可能需要升级：

- 问题需要多步分解
- 需要动态决定是否继续检索
- 需要在多个知识源之间路由
- 需要图结构推理

这时候才值得考虑更复杂方案。

---

## 3. GraphRAG 和 Agentic RAG 分别在解决什么 📌

### 3.1 GraphRAG

GraphRAG 更适合：

- 实体关系复杂
- 多跳路径重要
- 仅靠相似度检索不够表达关系网络

它关注的是：

- 图结构
- 关系推理
- 多跳知识连接

### 3.2 Agentic RAG

Agentic RAG 更适合：

- 查询需要改写
- 需要动态多次检索
- 需要决定调用哪个知识源
- 需要把问题拆成子任务

它关注的是：

- 动态决策
- 工具调用
- 状态管理
- 工作流控制

所以两者都比固定 RAG 更复杂，但复杂点并不一样。

---

## 4. 为什么不要过早上复杂方案 📌

过早上复杂方案的常见问题是：

- 还没把基础检索做稳，就引入更多不确定性
- 问题定位更困难
- 成本和延迟一起上升
- 团队维护门槛显著提高

很多看起来像“该上 Agent 了”的问题，实际根因可能只是：

- chunk 切得不好
- metadata 不够
- 检索策略没调好
- Prompt 没把来源和拒答边界写清楚

所以第八章最重要的判断不是“复杂方案是否存在”，而是：

> 先证明基础 RAG 确实不够，再升级。
