# 08. 进阶 RAG 方向

> 本章目标：建立对 `GraphRAG`、`Agentic RAG` 和更复杂知识系统的边界判断，明确什么时候固定 RAG 够用，什么时候才值得升级复杂方案。

---

## 1. 概述

### 学习目标

- 理解为什么第八章真正新增的是“架构升级判断”，不是“再讲几个高级名词”
- 理解为什么第八章必须建立在第七章的评估闭环之上，而不是凭感觉判断“固定 RAG 不够了”
- 掌握“固定 RAG / GraphRAG / Agentic RAG”三类方案分别在解决什么问题
- 能区分“真的需要升级复杂方案”和“其实只是基础链路还没调好”
- 能说明复杂方案带来的收益、代价和常见误判
- 能把第八章自然衔接到 [05_agent/outline.md](../05_agent/outline.md)

| 检索效果不稳定 | 先判断是基础链路问题，还是架构层级问题 |
| 多跳复杂问答 | 先判断是不是关系推理问题，而不是简单召回问题 |

| 团队想上 Agent | 先证明固定链路真的不够，再谈动态决策 |

| 想进入 Agent 课程 | 先把固定 RAG 的边界认知立住，避免把检索问题和编排问题混在一起 |

## 2. 为什么第八章不是“再加两个高级名词” 📌

### 2.1 真正难的不是知道名词，而是知道何时升级

很多人学到这里，很容易把第八章理解成：

- 传统 RAG 学完了
- 接下来要学更高级的 `GraphRAG`
- 再接着学更高级的 `Agentic RAG`

这种理解很危险，因为它默认了一件事：

> 复杂方案天然比固定方案更好。

但真实工程里，复杂方案从来不是“更高阶版本的默认升级”，而是：

- 只有当固定链路已经证明不够时才值得引入
- 只有当问题类型真的变化了，复杂性才是合理成本
- 只有当团队能评估、能调试、能维护时，复杂架构才有意义

所以第八章真正要解决的问题不是：

- “复杂方案有哪些”

而是：

- “什么时候复杂方案才是必要的”
- “什么时候你以为需要复杂方案，实际上只是基础链路还有缺口”

### 2.2 为什么第八章一定建立在第七章之后

如果没有第七章的评估闭环，第八章的所有升级判断都会很虚。

因为你一旦说“固定 RAG 不够了”，至少应该能回答下面这些问题：

1. 你的 `Golden Set` 里，到底哪些 case 失败了？
2. 失败主要发生在 retrieval、context selection、citation、refusal 还是 answer quality？
3. 是少数极端 case 失败，还是一个稳定的大类问题持续失败？
4. 你已经试过哪些基础改动？结果怎样？
5. 当前失败是“证据没召回”，还是“证据召回了但系统不会用”？

如果这些问题答不上来，那么“固定 RAG 不够了”大概率只是感受，不是结论。

换句话说：

> 第七章提供的是“升级判断的证据基础”，第八章提供的是“升级判断的框架”。

### 2.3 升级讨论的前提是什么

进入第八章之前，默认你已经尽量完成了下面这些事情：

| 检查项 | 至少应该做到什么 |
|--------|------------------|
| 文档处理 | chunk、metadata、document_id / chunk_id 基本稳定 |
| 检索策略 | 已经比较过 `top_k / threshold / MMR / hybrid / rerank` 的差异 |
| 生成链路 | 已经把 `answer + sources + refusal` 做成稳定输出 |
| 评估机制 | 已经有最小 `Golden Set` 和固定实验配置 |
| 失败分析 | 已经能收集坏案例，而不是靠零散截图讨论 |
| 真实验证 | 至少在真实 LLM 路径上做过最小回归 |

如果这些前提都还没满足，那第八章更合理的动作通常不是升级，而是回到前几章继续补基础。

### 2.4 第八章的默认决策顺序

本课程建议把升级判断固定成下面这条顺序：

```text
先确认文档处理和 metadata 没问题
-> 再确认 retrieval strategy 已经尽量调稳
-> 再确认 context selection / citation / refusal 已经稳定
-> 再用 Golden Set 证明失败是“结构性问题”
-> 再判断是否需要 GraphRAG
-> 最后才判断是否需要 Agentic RAG
```

这个顺序很重要，因为它能避免两类常见错误：

1. 基础链路没打稳，就把问题升级成“需要 Agent”
2. 关系表达不清的问题，还没分清是不是图结构问题，就先上动态工作流

## 3. 固定 RAG 到底什么时候够用，什么时候不够 📌

### 3.1 固定 RAG 的甜蜜区是什么

固定 `2-step RAG` 很适合下面这类任务：

- 单跳问答
- FAQ、帮助中心、制度库、产品说明书
- 以“给出依据 + 回答问题”为主的文档问答
- 单一知识源、结构相对稳定的企业知识库
- 问题大多可以用“召回证据 -> 拼上下文 -> 生成答案”解决的场景

这些场景的共同特点是：

1. 问题通常不需要多轮分解
2. 知识来源相对稳定
3. 证据通常可以从少量 chunk 中直接找到
4. 系统更需要“稳定、可控、可评估”，而不是“高度动态”

所以对大多数业务知识问答来说，固定 RAG 并不是“初级版本”，而是：

> 最常见、最应该优先做稳、也最容易落地成生产系统的主线方案。

### 3.2 固定 RAG 常见失败，很多并不等于“需要升级”

下面这些现象很常见，但它们本身并不能自动推出“该上复杂方案”：

| 现象 | 更可能的根因 | 第一反应应该是什么 |
|------|--------------|--------------------|
| 总是答非所问 | chunk、metadata、retriever 配置有问题 | 先检查召回证据 |
| 引用不稳定 | context selection 或 prompt 约束不清 | 先看 citation 设计 |
| 有些编号、术语总查不准 | 关键词和语义检索没结合好 | 先试 `hybrid` |
| 答案重复、上下文噪声大 | 候选去重和排序不稳 | 先试 `MMR / rerank` |
| 无答案问题乱答 | refusal 设计不足 | 先补拒答评估 |
| 某类问题一直漏关键点 | `top_k`、候选池和 prompt 截断不合理 | 先看 context coverage |

这些问题之所以重要，是因为它们很容易造成一种错觉：

> “系统不够聪明，所以应该升级成更复杂的系统。”

但现实里，很多时候不是系统“不够聪明”，而是固定链路里最基础的证据流动还没打通。

### 3.3 什么才算“固定 RAG 真的不够”

只有当下面这类问题持续存在，而且已经经过前几章方法验证后仍无法有效解决，才更像是真正的升级信号。

| 升级信号 | 说明 | 更可能指向什么 |
|----------|------|----------------|
| 问题天然需要多跳关系推理 | 证据分散在多个实体和关系上，单纯相似度召回难以表达路径 | `GraphRAG` |
| 需要先拆问题再分别求证 | 用户问题过大、过混合，必须分成多个子问题处理 | `Agentic RAG` |
| 是否检索本身需要动态判断 | 有些问题应直接回答，有些需要检索，有些需要拒答 | `Agentic RAG` |
| 多知识源之间需要动态路由 | 同一问题可能落到 FAQ、数据库、文档库、工单系统等不同源 | `Agentic RAG` |
| 需要持续追踪实体关系网络 | 核心价值在“关系结构”而不是单文档片段 | `GraphRAG` |
| 固定 top-k 证据拼接始终覆盖不了问题结构 | 不是召回量问题，而是证据组织方式不对 | 视情况指向 `GraphRAG` 或 `Agentic RAG` |

这里最重要的不是背下表格，而是理解：

- `GraphRAG` 主要在补“关系表达和多跳连接”
- `Agentic RAG` 主要在补“动态决策和分步执行”

它们不是同一件事，也不是先后必经的等级体系。

### 3.4 升级假信号：最容易误判的情况

第八章最有价值的一部分，不是“升级信号”，而是“假信号”。

因为很多团队并不是不会升级，而是升级得太早。

| 看起来像升级信号 | 实际上更可能是什么 | 正确动作 |
|------------------|--------------------|----------|
| “模型有时答错” | Prompt、context、citation 边界不清 | 先回第六章和第七章 |
| “检索偶尔漏掉答案” | `top_k / threshold / hybrid / rerank` 还没调够 | 先回第五章 |
| “不同问题需要不同提示词” | prompt 变体管理问题 | 先做固定配置对比 |
| “有些问题很复杂” | 只是问题长，不一定需要多步决策 | 先做问题类型拆分 |
| “想支持更多知识源” | 可能只是 metadata filter 和 source priority 不够 | 先补检索治理 |
| “想让系统更聪明” | 目标描述模糊，没有失败归因 | 先补评估和坏案例分析 |

很多时候，真正的问题不是“该不该上 Agent”，而是：

- 你还不知道失败到底发生在哪一层
- 你还没有把固定系统的边界压到足够清楚

## 4. GraphRAG 在解决什么问题 📌

### 4.1 先说结论：GraphRAG 不是“更高级的检索器”

很多人第一次听到 `GraphRAG`，会把它理解成：

- 向量检索的升级版
- 比 `hybrid`、`rerank` 更强的一层
- 只要问题难一点就该上图

这种理解不准确。

`GraphRAG` 真正在解决的是：

> 当知识的核心价值不只是“文本片段相似”，而是“实体之间的关系结构”时，普通 chunk 级检索表达能力不够的问题。

也就是说，`GraphRAG` 的重点不是把召回做得“更多”，而是把知识组织方式从：

- 文本片段集合

部分升级成：

- 实体、关系、路径、社区、摘要等结构化知识表示

### 4.2 普通固定 RAG 为什么会在这类问题上吃亏

固定 RAG 最擅长的是：

- 找到和当前问题相近的局部文本证据
- 把这些文本片段放进上下文
- 让模型根据片段回答

但如果问题本身依赖关系路径，局部相似度就可能不够。

典型表现包括：

1. 每一段文本单看都不完整
2. 关键证据分散在多个实体和多个文档里
3. 问题需要的不只是“有没有提到”，而是“实体之间怎么连接”
4. 召回的 chunk 都相关，但模型仍然很难自己稳定拼出关系图

举例来说：

- “某个项目负责人所在部门的上级负责人是谁？”
- “一家公司收购的子公司里，哪些产品线和当前故障服务有关？”
- “某项政策影响了哪些流程、角色和审批节点？”

这类问题的共同点是：

> 你真正要找的不是一个答案片段，而是一条关系路径。

### 4.3 GraphRAG 的原理到底是什么

如果用非常克制的方式概括，`GraphRAG` 主要做三件事：

1. 把文本里的实体抽出来
2. 把实体之间的关系显式表示出来
3. 在回答时不只看相似文本，还会利用关系结构做连接、汇总和推理

所以 `GraphRAG` 更像是在补：

- 关系表达能力
- 多跳连接能力
- 全局结构视角

而不是单纯补：

- top-k 不够
- 关键词没命中
- rerank 不够强

把它展开一点，可以理解成下面这条链路：

```text
原始文档
-> 抽取实体（人、组织、系统、政策、产品、事件）
-> 抽取关系（属于、依赖、影响、负责、上下游、引用）
-> 形成图结构或图索引
-> 围绕问题找到相关实体和关系子图
-> 再把子图信息、相关摘要、原始证据一起交给模型生成答案
```

这里的关键变化在于：

- 固定 RAG 的基本检索对象是 `chunk`
- `GraphRAG` 的核心检索对象开始包含 `entity / relation / path / community summary`

所以它不是只把“文本找出来”，而是开始把“文本背后的关系网络”也纳入检索对象。

### 4.4 GraphRAG 在实际使用时通常怎么用

这里不展开具体实现代码，但你需要知道它的典型使用形态。

最常见的落地方式不是“完全抛弃原来的 RAG”，而是：

1. 先保留原有文档和 chunk
2. 在离线处理阶段额外做实体抽取和关系抽取
3. 构建一个图视角的索引或知识表示
4. 查询时先找相关实体、关系路径或社区摘要
5. 再把这些结构化结果和原始文本证据一起送给 LLM

也就是说，很多 `GraphRAG` 更像：

```text
document chunks + graph structure + graph summaries
-> 一起服务回答
```

而不是：

```text
只剩图，不看原文
```

这点很重要，因为真实回答通常仍然需要：

- 原始文本证据
- 可引用来源
- 对关系推理结果做文本落地

### 4.5 如果用了 GraphRAG，系统会多出什么能力

`GraphRAG` 最直接的好处，不是“模型更聪明”这种抽象表述，而是下面这些更具体的变化：

| 固定 RAG 容易吃亏的地方 | GraphRAG 可能带来的改善 |
|-------------------------|-------------------------|
| 证据分散在多个文档，单个 chunk 都不完整 | 可以先通过实体和关系把分散证据连起来 |
| 问题依赖多跳关系，模型靠临时拼接不稳定 | 可以显式给模型一条关系路径或子图 |
| 同一问题需要全局视角，不只是局部片段 | 可以先做社区摘要、全局摘要，再回答 |
| 很难解释“为什么这个对象和那个对象有关” | 可以把关系链条显式展示出来 |
| 大量文本里概念关系复杂，普通相似度容易漂移 | 可以先按图结构收缩搜索范围 |

换句话说，`GraphRAG` 的收益通常体现在：

- 关系型问题回答更稳定
- 多跳问题更容易找到完整证据链
- 全局综述类问题更容易先聚合再回答
- 回答过程更容易围绕“对象关系”组织，而不是零散 chunk

### 4.6 什么场景更适合 GraphRAG

| 场景 | 为什么更适合 GraphRAG |
|------|------------------------|
| 组织架构、角色职责、汇报链查询 | 本质是实体和关系网络 |
| 产业链、供应链、投资关系分析 | 需要跨实体做多跳连接 |
| 复杂法规、制度、流程依赖 | 问题依赖多个对象之间的约束关系 |
| 科研文献综述、概念关系图 | 需要从大量文本里抽关系再看全局 |
| 安全事件、系统依赖、故障传播分析 | 关注的是节点关系和影响路径 |

这些场景并不意味着一定要做很重的知识图谱工程，但至少说明：

> 当问题核心开始转向“关系网络”，普通 chunk 检索就不是唯一合理表示了。

### 4.7 什么场景通常不值得上 GraphRAG

下面这些场景即使听起来“知识很复杂”，也不一定值得立刻上图：

| 场景 | 为什么通常不值得先上 GraphRAG |
|------|-------------------------------|
| FAQ、客服知识库 | 大多数是局部证据问答，固定 RAG 足够 |
| 产品文档、操作手册 | 问题通常能由少量 chunk 直接回答 |
| 规则库、公告库、标准制度查询 | 更多是检索覆盖和引用准确性问题 |
| 刚起步的知识库项目 | 连基础 metadata、评估和回归都未稳定 |
| 只是想做一个展示型 Demo | 图构建和维护成本往往高于实际收益 |

很多团队在这里的误判是：

- 问题看起来复杂
- 文档很多
- 所以应该做图

但文档很多不等于图结构必要；文本量大和关系复杂是两件不同的事情。

### 4.8 GraphRAG 的主要代价是什么

讨论 `GraphRAG` 时，不能只说价值，不说成本。

它常见的代价包括：

- 实体抽取和关系抽取本身就需要额外链路
- 图构建质量会直接影响最终结果
- 图更新和删除一致性比普通向量索引更复杂
- 调试路径更长，失败归因更难
- 评估标准也更复杂，不再只是“是否召回 chunk”

所以第八章对 `GraphRAG` 的基本态度是：

> 只有当“关系表达能力”本身成为核心瓶颈时，图结构的复杂度才是合理代价。

### 4.9 关于 GraphRAG 的两个常见误区

常见误区 1：

> “多跳问题 = 一定要上 GraphRAG”

这并不总成立。

因为有些多跳问题其实只是：

- 需要更好的 query rewrite
- 需要更好的 multi-query
- 需要更大的候选池和更稳的 rerank
- 需要更清楚的上下文组织

只有当你发现：

- 问题反复依赖稳定的实体关系网络
- 文本 chunk 的表达始终无法稳定承载这种关系

这时 `GraphRAG` 才更像对症方案。

常见误区 2：

> “GraphRAG = 一定要先建设完整知识图谱平台”

这也不准确。

很多实际系统更常见的做法是：

- 先从小范围实体抽取开始
- 先为某一类高价值关系问题补图结构
- 先把图作为固定 RAG 的增强层，而不是全量替代层

也就是说，`GraphRAG` 完全可以是渐进式增强，而不是一步到位的大工程。

## 5. Agentic RAG 在解决什么问题 📌

### 5.1 先说结论：Agentic RAG 不是“让 RAG 自己跑一跑”

很多人会把 `Agentic RAG` 粗糙地理解成：

- RAG + 工具调用
- RAG + 多轮检索
- 更聪明的 RAG

这些说法有一点方向，但还不够精确。

`Agentic RAG` 真正在补的是：

> 当“是否检索、检索几次、改不改写问题、走哪个知识源、是否拆子问题”都不能提前固定写死时，固定链路不再够用的问题。

所以它的核心不是“多一步检索”，而是：

- 决策变成动态的
- 流程不再是单一路径
- 系统要根据当前状态选择下一步动作

### 5.2 Agentic RAG 的原理到底是什么

固定 RAG 的默认形态通常是：

```text
question
-> retrieve
-> pack context
-> answer
```

而 `Agentic RAG` 更像是：

```text
question
-> 判断是否需要检索
-> 选择知识源
-> 必要时改写查询
-> 检索一次或多次
-> 必要时拆成子问题
-> 汇总证据
-> 回答或拒答
```

也就是说，`Agentic RAG` 真正引入的是：

1. 动态路由
2. 多步决策
3. 状态管理
4. 工具选择
5. 中间观察后的再决策

这和固定 RAG 的复杂度不是一个量级。

把它再讲白一点，`Agentic RAG` 的原理不是“让模型更会背知识”，而是：

- 让模型先判断任务类型
- 再决定该调用哪类检索能力
- 在中间拿到观察结果后继续决定下一步
- 必要时把复杂问题拆开求解

所以它的底层逻辑是：

> 把“检索”从固定步骤，升级成“可被系统动态调度的一种工具能力”。

### 5.3 Agentic RAG 在实际使用时通常怎么用

这里也不展开具体框架代码，但你需要知道它的典型使用形态。

最常见的方式不是“让 Agent 无限自由发挥”，而是：

1. 先定义系统可用的知识源和工具
2. 让模型先判断当前问题属于哪一类
3. 按规则或模型判断去选择检索路径
4. 中间拿到结果后再决定是否继续检索、是否改写、是否拆问题
5. 最后把证据汇总成答案，并保留来源或轨迹

一个很常见的最小形态其实是：

```text
问题分类
-> 选择数据源
-> 一次检索
-> 如果证据不足则改写再检索
-> 汇总答案
```

再复杂一点的形态会变成：

```text
问题拆分
-> 每个子问题分别检索
-> 对比不同来源
-> 汇总中间结论
-> 输出最终答案
```

所以很多 `Agentic RAG` 的真实用法，不是为了“炫技”，而是为了处理下面这些固定流程很难写死的情况：

- 同一个入口面对多种问题类型
- 不同问题需要不同知识源
- 有些问题一次检索就够，有些不够
- 有些问题必须先改写再查，或先拆再查

### 5.4 如果用了 Agentic RAG，系统会多出什么能力

`Agentic RAG` 带来的收益，通常体现在“过程能力”上，而不只是最终答案表面更长。

| 固定 RAG 容易吃亏的地方 | Agentic RAG 可能带来的改善 |
|-------------------------|-----------------------------|
| 所有问题都被迫走同一条链路 | 可以先判断问题类型，再选路径 |
| 多知识源问题很难统一处理 | 可以按问题动态路由到不同知识源 |
| 一次检索不够，但固定流程不会补救 | 可以做二次检索、查询改写或补查 |
| 大问题过于混合，单次回答容易漏 | 可以先拆子问题再分别求证 |
| 需要先查知识再查结构化数据 | 可以把不同工具串成步骤化过程 |

所以 `Agentic RAG` 的主要好处通常是：

- 系统对复杂问题更有弹性
- 多知识源接入更自然
- 中间可以做补救动作，而不是一次失败就结束
- 更适合研究型、分析型、流程型任务

但要注意，这些收益不是白送的，它们一定伴随更高的治理成本。

### 5.5 什么场景更适合 Agentic RAG

| 场景 | 为什么更适合 Agentic RAG |
|------|---------------------------|
| 同一入口接多个知识源 | 需要动态判断该查 FAQ、文档、数据库还是工单系统 |
| 研究型问题 | 往往需要多轮检索、改写问题、对比不同来源 |
| 复杂业务助手 | 既可能查知识，又可能查结构化数据，还可能调业务工具 |
| 大问题需要拆成多个子问题 | 固定单次召回很难覆盖整个求解过程 |
| 用户问题质量不稳定 | 需要先澄清、改写或逐步缩小检索范围 |

这些场景的共同特点是：

- 不是所有问题都走同一条路径
- 不是一次检索就能解决所有情况
- 决策本身已经成为系统能力的一部分

### 5.6 什么场景通常不值得上 Agentic RAG

| 场景 | 为什么通常不值得先上 Agentic RAG |
|------|-----------------------------------|
| 标准 FAQ 问答 | 固定链路已经足够稳定且更易控 |
| 单知识源文档问答 | 路由和多步决策收益有限 |
| 团队尚未建立最小评估和回归 | 动态系统会显著放大不可控性 |
| 只是检索效果偶尔不理想 | 很可能是 retriever 和 prompt 问题 |
| 需要强可解释、强可审计的简单系统 | 动态决策反而增加解释成本 |

很多团队会在这里犯一个典型错误：

- 看到某些问题复杂
- 就想让系统“自己多查几次”

但“多查几次”本身并不等于“结果更可靠”，它同时也意味着：

- 成本上升
- 时延上升
- 失败路径变多
- 调试难度上升

### 5.7 Agentic RAG 的主要代价是什么

`Agentic RAG` 常见的代价包括：

- 路由错误会导致整条链路偏航
- 查询改写可能把问题改坏
- 多步检索会带来成本和时延放大
- 子问题拆分可能制造新的错误传播
- 工具调用和权限边界会让系统治理更难
- 评估不再只是最终答案，还要看中间轨迹是否合理

所以第八章对 `Agentic RAG` 的基本态度是：

> 只有当“固定流程本身不够表达任务过程”时，动态决策才值得引入。

### 5.8 关于 Agentic RAG 的三个常见误区

常见误区 1：

> “复杂问题都应该让 Agent 自己拆。”

实际风险是：

- 拆得不对
- 查得更多但更散
- 中间步骤没有评估和约束

常见误区 2：

> “只要加上路由和多轮检索，效果就会自然变好。”

现实里更常见的情况是：

- 有些问题确实变好了
- 但另一些问题因为流程更长反而更不稳定

常见误区 3：

> “Agentic RAG = 一定要做完整 Agent 平台”

这也不准确。

很多实际系统的第一步只是：

- 先做一个简单路由器
- 先支持两三个知识源的选择
- 先支持一次补检索和一次查询改写

也就是说，`Agentic RAG` 也完全可以渐进式增加，而不是一开始就做成复杂编排系统。

所以 `Agentic RAG` 不是“默认更优”，而是“在特定问题类型上更有必要”。

## 6. 固定 RAG / GraphRAG / Agentic RAG 对比 📌

### 6.1 先建立一个最小对比框架

如果只记一张表，本章建议记下面这张。

| 维度 | 固定 RAG | GraphRAG | Agentic RAG |
|------|----------|-----------|-------------|
| 主要解决什么 | 召回相关证据并回答 | 表达实体关系和多跳路径 | 处理动态决策和多步检索 |
| 默认知识表示 | chunk / document | entity / relation / path / summary | 多知识源 + 工具 + 状态 |
| 默认流程 | 固定链路 | 图结构辅助检索与推理 | 动态路由、改写、拆分、反复检索 |
| 最适合的问题 | 单跳文档问答、FAQ、制度库 | 关系网络、多跳连接、结构依赖 | 多知识源、多步骤、过程不固定 |
| 最常见收益 | 稳定、简单、可控、易评估 | 关系表达更强 | 决策灵活、适应复杂任务 |
| 最常见代价 | 关系表达有限 | 图构建和维护成本高 | 成本高、时延高、调试难 |
| 常见误判 | 以为它太“基础” | 以为多跳都必须上图 | 以为复杂问题都该上 Agent |
| 默认优先级 | 最高，主线方案 | 只在关系结构成为瓶颈时考虑 | 只在动态过程成为瓶颈时考虑 |

### 6.2 一句判断法

如果你只想记一句快速判断：

- 证据就在文档里，问题主要是“找出来并回答”时，优先固定 RAG
- 问题核心是“实体之间怎么连”时，考虑 `GraphRAG`
- 问题核心是“系统下一步该怎么做”时，考虑 `Agentic RAG`

## 7. 为什么不要过早上复杂方案 📌

### 7.1 复杂方案最容易带来的不是收益，而是混乱

很多团队过早升级后遇到的第一个问题，不是效果立刻提升，而是：

- 失败归因开始变得混乱
- 任何问题都不知道是哪个环节造成的
- 回归成本上升
- 维护者开始害怕改动

比如：

1. 原本只是 `hybrid` 没调好，却上了多知识源路由
2. 原本只是 citation 不稳，却开始做查询改写和多轮检索
3. 原本只是某类关系问题难答，却直接引入完整 Agent 工作流

这些路径的问题在于：

> 复杂系统会放大基础问题，而不是自动掩盖基础问题。

### 7.2 升级前应该先问自己的问题

在决定升级前，建议先用下面这组问题做一次自检：

| 自检问题 | 如果答案是否定的，更合理的动作 |
|----------|-------------------------------|
| 我已经能稳定复现失败 case 吗？ | 先补 `Golden Set` |
| 我知道失败发生在 retrieval 还是 generation 吗？ | 先补分层评估 |
| 我已经试过固定链路里的主要优化手段了吗？ | 先回第五章和第六章 |
| 我能明确说出“为什么固定流程不够”吗？ | 先补失败归因 |
| 我能接受更高的时延和成本吗？ | 先评估业务约束 |
| 团队有能力维护更复杂的链路吗？ | 先保持固定方案 |

如果这里有两三项都答不上来，通常不应该急着升级。

### 7.3 推荐的默认升级顺序

本课程推荐的默认顺序是：

```text
固定 RAG 做稳
-> 建立评估闭环
-> 证明失败具有结构性
-> 判断是“关系表达不足”还是“动态决策不足”
-> 前者再看 GraphRAG
-> 后者再看 Agentic RAG
```

这个顺序的价值在于：

- 先缩小问题空间
- 再引入必要复杂度
- 避免为了“高级感”而过度设计

### 7.4 第八章最该避免的一种思维

最该避免的思维是：

> “固定 RAG 学完了，下一步自然就是更复杂的 RAG。”

更准确的说法应该是：

> 固定 RAG 是默认主线；复杂方案是边界外的有条件升级，不是线性等级提升。

## 8. 场景判断：怎样做第八章的真实决策练习 📌

### 8.1 场景一：企业 FAQ 与制度问答

问题特点：

- 问题大多是单跳
- 证据通常在单份文档或少量 chunk 中
- 重点是引用正确、拒答正确、更新及时

更合理的方案：

- 优先固定 RAG

理由：

- 主要矛盾在检索覆盖、来源对齐和回答可控
- 动态路由和图结构通常不是主要瓶颈

不应优先做的事：

- 不应因为知识库文档多，就直接上 `GraphRAG`
- 不应因为有少量复杂问题，就默认引入 `Agentic RAG`

### 8.2 场景二：组织架构与职责关系查询

问题特点：

- 核心在“谁向谁汇报”“谁负责什么”“哪个角色依赖哪个流程”
- 关系链条比局部文本片段更关键
- 问题往往依赖多跳连接

更合理的方案：

- 更可能考虑 `GraphRAG`

理由：

- 问题核心不是“某段文本写了什么”，而是“关系如何连接”
- 如果只靠 chunk 相似度，容易召回相关片段但难稳定表达路径

需要警惕的误判：

- 如果问题其实只是固定模板问答，不要为了“组织架构”四个字就默认上图

### 8.3 场景三：研究型助手或复杂分析助手

问题特点：

- 用户问题不稳定
- 同一问题可能需要多个知识源
- 需要改写问题、拆分子问题、逐步求证

更合理的方案：

- 更可能考虑 `Agentic RAG`

理由：

- 难点不只是找证据，而是决定下一步该查什么、去哪里查、查几次

需要警惕的误判：

- 如果只是单一文档库里的复杂问答，不一定要立刻做 Agent
- 先确认是不是固定链路上的检索和上下文组织还不够稳

### 8.4 场景四：产品想快速验证价值

问题特点：

- 时间紧
- 目标是先验证用户是否愿意使用
- 工程资源有限

更合理的方案：

- 先固定 RAG，甚至先做更轻的 Hosted 检索方案

理由：

- 当前最重要的是验证需求，不是追求架构完整度

第八章在这里想强调的是：

> 复杂架构常常适合第二阶段，不适合零到一验证阶段。

### 8.5 一张场景判断表

| 如果你的主要问题是... | 更应该先考虑什么 |
|------------------------|------------------|
| 证据召回不稳 | 固定 RAG 调优 |
| 引用和拒答不稳 | 固定生成链路调优 |
| 问题依赖关系网络 | `GraphRAG` |
| 流程依赖动态路由和多步决策 | `Agentic RAG` |
| 只是想快速验证产品 | 固定 RAG 或更轻方案 |
| 团队还没有评估与回归能力 | 暂缓升级复杂方案 |

## 综合案例：什么时候才值得从固定 RAG 升级

请用本章框架判断下面四类场景：

1. 企业 FAQ 问答，重点是来源正确、拒答正确、知识更新及时
