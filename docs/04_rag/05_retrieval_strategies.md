## 2. Retriever 在系统里到底负责什么 📌

### 2.1 为什么不能把 Vector Store 直接当 Retriever

Vector Store 更偏基础设施层，负责：

- 存
- 查
- 删
- 过滤

Retriever 更偏应用层，负责：

- 接收用户问题
- 决定查询参数
- 决定返回多少结果
- 决定是否做多样性控制、阈值过滤或增强策略
- 把结果整理成后续 Chain 能消费的统一接口

所以 Retriever 的本质不是“再包一层类”，而是：

> 把底层查询能力收束成稳定的应用侧检索接口。

### 2.2 为什么检索质量是 RAG 的核心变量

如果召回结果本身不对，后面 Prompt 写得再好，模型也只能在错误上下文里回答。

所以第五章要建立一个很重要的习惯：

> 先看召回结果，再看生成结果。

## 3. 常见检索策略分别解决什么问题 📌

### 3.1 `top_k`

`top_k` 解决的是：

- 给模型多少条候选上下文

它太小，可能召回不全。

它太大，可能混入过多噪声，还会增加上下文成本。

### 3.2 `score_threshold`

阈值过滤解决的是：

- 不让明显不相关的结果也进入上下文

它的价值主要在于：

- 控制噪声
- 更好地处理“无答案”场景

### 3.3 MMR

MMR 的目标不是单纯找“最像”的结果，而是：

> 在相关性和多样性之间做平衡。

当知识库里有很多相似 chunk 时，单纯的 similarity 很容易把高度重复的内容全召回来。

MMR 更适合：

- 重复内容较多
- 希望上下文覆盖更多角度

### 3.4 metadata filter

metadata filter 适合：

- 只查某类文档
- 只查某个知识域
- 只查某个版本或来源

它不是“高级技巧”，而是很多真实业务里非常常见的基本需求。

### 3.5 混合检索、Rerank、查询变换

这些增强策略分别更适合：

- 混合检索
  术语、编号、产品名很多的场景
- Rerank
  初召回够广，但排序不够准的场景
- 查询变换 / HyDE
  用户问题表达模糊，或需要改写查询才更容易召回的场景

重点不是“越多越高级”，而是：

> 先把基础 similarity 做稳，再基于坏案例判断是否需要升级。

## 4. 检索效果差时先看什么 📌

很多人遇到回答不好，会先改 Prompt。

但更高效的排查顺序通常是：

1. 用户问题本身是否清晰
2. query embedding 是否合理
3. Top-K 结果里有没有正确来源
4. metadata filter 是否过严或过松
5. 召回结果是否高度重复
6. 最后才看 Prompt 和生成

这也是为什么第五章要开始记录坏案例。

没有坏案例，你很难回答：

- 现在到底是哪种失败
- 加一个策略到底有没有变好
