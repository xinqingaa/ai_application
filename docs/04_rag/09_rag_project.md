## 2. 为什么最终项目不该一开始就给出来 📌

如果一开始就直接把完整项目摊开，常见问题是：

- 学习入口不清晰
- 目录很多，但不知道为什么有这些层
- 文档和代码容易脱节

这也是为什么 `04_rag` 采用的是：

```text
章节正文
+ 独立章节实现
+ 最终 rag_lab
```

先按章学习，再在最后收口成完整项目。

---

## 3. 第九章要收口哪些能力 📌

一个完整的 RAG 项目，至少应该把下面几类能力收回来：

1. 文档处理
2. 向量化
3. 向量存储
4. 检索
5. 生成
6. 评估

此外还需要把这些“项目级能力”补清楚：

- 如何运行
- 如何评估
- 当前支持什么
- 当前不支持什么

这就是第九章的重点：

> 不是继续加知识点，而是把系统真正整理成可交付形态。

---

## 4. 第九章整合工作和 `rag_lab` 的关系 📌

这里的两个概念角色不一样：

| 项目 | 角色 |
|------|------|
| 第九章整合工作 | 把前面章节能力收束成统一项目的过程 |
| `rag_lab/` | 整合完成后的最终参考目录 |

也就是说：

- 第九章首先是一次整合动作，而不是单独维护一个旧式 phase 目录
- `rag_lab` 才是收口后的完整参考实现

但即使有了 `rag_lab`，学习入口也不应该倒退成“直接看最终项目”。

---

## 5. 第九章实现后应该怎么学

等第九章真实代码快照落地后，建议按这个顺序：

1. 先读本章正文
2. 再看 `rag_lab` 的目录和 README
3. 再看统一运行入口
4. 最后看统一评估入口

本章最终完成标准应该是：

- 最终 `rag_lab` 能作为完整参考项目存在
- 学习入口仍然是九章正文和前面已经落地的独立章节目录
- 项目 README 能说明如何运行、如何评估、当前不支持什么
