# 07. 综合项目

> 本节目标：把前六章的能力真正收束成一个可运行、可调试、可扩展的 LLM 小项目，掌握为什么课程最后一定要做综合项目、为什么采用 CLI + 服务层 + API 这套结构，以及这些模块如何一起形成 AI 应用的最小工程骨架

---

## 1. 概述

### 学习目标

- 理解为什么学完前六章后，必须用一个综合项目把所有知识点真正串起来
- 理解为什么综合项目不是“再写一个 demo”，而是把能力从脚本阶段推进到项目阶段
- 掌握 CLI、服务层、API 层、Prompt 文件、会话状态、导出与统计之间的职责分层
- 掌握多轮会话 CLI 的命令设计思路，而不是只会做最简单的一问一答
- 理解 JSON 模式、流式输出、provider 切换、错误处理、成本统计、安全检查是怎样在同一项目里协同工作的
- 能写出一个既能跑命令行，也能暴露 FastAPI 接口的最小 LLM 应用骨架

### 预计学习时间

- 为什么要做综合项目：45 分钟
- 项目分层与服务层设计：1-1.5 小时
- CLI 命令设计与状态管理：1-2 小时
- JSON 模式、Prompt 文件与导出：1 小时
- API 复用与流式输出：1-1.5 小时
- 项目回顾与练习：1 小时

### 本节在 AI 应用中的重要性

| 场景 | 相关知识 |
|------|---------|
| 本地快速验证产品思路 | CLI、多轮会话、Prompt 文件 |
| 抽离业务内核 | 服务层、配置层、状态管理 |
| 统一接入不同前端形态 | CLI 与 API 共用一套服务 |
| 项目级可观测性 | usage、成本统计、导出、会话回放 |
| 继续扩展到后续课程 | 为 RAG、Agent、前端 UI 做骨架准备 |

> **如果前六章解决的是“一个个能力点怎么实现”，第七章解决的就是“这些能力点怎样组合成一个真正能继续长大的项目”。**

### 本章与前六章的关系

第一章解决的是：

1. 怎么发起一次最小聊天请求
2. 怎么理解消息结构
3. 怎么理解响应对象和参数

第二章解决的是：

1. 怎么切换不同 provider
2. 怎么把平台差异收敛到配置层

第三章解决的是：

1. Prompt 为什么不只是聊天措辞
2. Prompt 怎样从字符串变成可复用资产

第四章解决的是：

1. 结构化输出为什么重要
2. JSON、Schema、校验、重试怎样一起工作

第五章解决的是：

1. 流式输出为什么能改善体验
2. CLI / 服务端 / SSE 是怎样连接起来的

第六章解决的是：

1. 错误怎么分类和重试
2. 成本怎么统计
3. 缓存、配额、安全怎么进入主链路

第七章要做的是：

1. 把这些能力真正组合起来
2. 把“章节脚本”推进到“项目骨架”
3. 让学习结果不只是“我知道这些点”，而是“我已经把它们装进一个真实工程”

所以第七章本质上是：

> 从“会做章节练习”过渡到“会组织一个最小 AI 应用项目”。

### 本章的学习边界

这一章重点解决：

1. 综合项目分层
2. CLI 命令系统
3. 统一服务层
4. Prompt 文件和 JSON 模式
5. 流式输出在项目层的接入
6. 导出、统计和 API 复用

这一章不展开：

- 浏览器 UI
- 数据库持久化
- RAG 检索链路
- Agent 多工具编排
- WebSocket 双向协议
- 多副本部署和生产运维

这些内容会在后续课程或项目深化阶段继续展开。

### 平台学习策略

第七章仍然延续前面的策略：

- 教学理解主线：`OpenAI / Claude / Gemini`
- 实际练习优先：`百炼 / 通义、DeepSeek、GLM`

原因没有变：

1. 成本更友好
2. OpenAI-compatible 调用路径更统一
3. 适合高频试验

但在项目章里要额外强调一点：

> 你真正要抽象的不是“某个平台的某个字段”，而是“项目内部稳定的调用接口”。

也就是说：

- 外部 provider 可以换
- 项目内部服务层接口最好不要频繁变

这就是项目章开始强调“内部边界”和“模块职责”的原因。

## 2. 为什么课程最后一定要做综合项目 📌

### 2.1 只学章节知识，很容易形成“碎片熟悉”

很多人在学一门技术课的时候，会出现一种错觉：

> 这一章我看懂了，那个脚本我也跑通了，所以我已经会了。

但一旦让他自己从头写一个小项目，就会暴露很多断层：

- 不知道代码该怎么分文件
- 不知道状态放哪
- 不知道 CLI 和 API 该怎么共用
- 不知道哪些能力该放服务层
- 不知道统计、导出、缓存、安全到底接到哪一层

这说明什么？

说明“单点理解”还没有变成“项目级理解”。

### 2.2 为什么 AI 应用特别需要综合项目收束

因为 AI 应用天然比普通脚本更依赖横切能力。

普通脚本常常只解决一个点：

- 调一个接口
- 打印一下结果

但 AI 应用通常同时涉及：

1. 消息状态
2. Prompt 资产
3. 流式输出
4. 成本统计
5. 错误恢复
6. 多平台切换
7. 安全防护
8. 导出和回放

如果这些能力一直散落在独立脚本里，你会越来越难判断：

- 哪些东西是实验代码
- 哪些东西是项目内核

### 2.3 综合项目不是“终极复杂系统”，而是“最小完整系统”

很多人一听“项目章”，会误以为应该一下子做成：

- 前后端齐全
- 数据库齐全
- 用户系统齐全
- 权限齐全

这其实是错误方向。

课程最后应该做的，不是把项目做得越来越重，而是做出一个：

> 足够小，但结构完整的系统。

所谓“结构完整”是指：

1. 有项目入口
2. 有状态管理
3. 有统一服务层
4. 有普通输出和流式输出
5. 有统计与导出
6. 有基本的错误处理与安全能力

这套东西一旦立起来，后面要往上加：

- Web 前端
- RAG
- Agent
- 持久化

都比较自然。

### 2.4 为什么第七章选择 CLI 作为核心项目

CLI 有几个很强的教学优势。

#### 优势 1：去掉前端噪音

如果一开始就做浏览器 UI，学员很容易被：

- 页面交互
- 样式
- 网络调试
- 框架细节

分散注意力。

而 CLI 可以让你把注意力集中在：

- 消息状态
- 命令设计
- 服务层组织
- 模型调用链路

#### 优势 2：特别适合练多轮状态管理

CLI 本身就是一个天然的“逐轮输入 -> 逐轮输出”环境。

这和聊天应用的核心交互非常契合。

#### 优势 3：特别适合练命令系统

像这些命令：

- `/provider`
- `/model`
- `/json`
- `/stream`
- `/stats`
- `/export`

在 CLI 里非常自然。

你可以直接把运行时配置显式化，而不是把一切都藏在 UI 按钮后面。

#### 优势 4：特别适合做调试和回归

CLI 比前端页面更容易：

- 快速试错
- 录制命令流
- 做回归验证

### 2.5 为什么 CLI 还不够，必须再补服务层和 API

如果项目只有一个 `chat_cli.py`，那它很容易变成“单文件大杂烩”。

比如：

- 命令解析写在一起
- 聊天逻辑写在一起
- 导出写在一起
- provider 配置也写在一起

这种写法短期能跑，长期很难扩展。

所以项目章必须再往前迈一步：

1. CLI 是入口
2. 服务层是内核
3. API 是另一种入口

这样你才真正进入“项目组织”的语境。

### 2.6 为什么 API 要出现在项目章里

因为它能证明一件关键的事：

> 你的服务层不是只为命令行写的，而是能支撑不同调用入口。

这很重要。

因为真实项目里，今天可能是 CLI，明天可能是：

- FastAPI
- Web UI
- 定时任务
- Agent 调用链

如果你的服务层是稳定的，这些入口都能复用同一套内核。

如果你的服务层不存在，每换一个入口就要重写一套逻辑。

### 2.7 这一章真正想培养的能力是什么

不是“会用几条命令”。

而是下面这些能力：

1. 看见项目边界
2. 看见职责分层
3. 看见哪些逻辑应该沉淀为通用能力
4. 把章节知识变成项目结构

这也是为什么第七章不是附录，而是整门课的收束点。

## 3. 综合项目到底要收回前六章哪些能力

### 3.1 第一章能力：基础聊天闭环

如果没有第一章的基础能力，综合项目甚至没法开始。

项目章仍然要依赖：

- 基础聊天请求
- `messages`
- 环境变量
- 响应对象

区别在于：

第一章关心的是“能不能跑”。

第七章关心的是“能不能收进项目结构里”。

### 3.2 第二章能力：provider 切换

综合项目里，provider 切换不再是某个教学脚本里的演示，而是一个真正的运行时能力。

比如在 CLI 里，你希望能这样用：

```text
/provider deepseek
/provider glm
```

这时 provider 就不是一段静态配置，而是：

- 会影响后续请求
- 会影响默认 model
- 会影响成本和行为观察

### 3.3 第三章能力：Prompt 资产化

项目章里，Prompt 不能只靠代码里的长字符串。

你需要让它具备：

1. 能写在文件里
2. 能在运行时加载
3. 能和 JSON 模式联动
4. 能和 system prompt 区分开

这就是 `/prompt <file>` 命令的意义。

### 3.4 第四章能力：结构化输出

项目章不一定要把完整 Schema 校验全塞进 CLI，但至少要体现出：

- 项目支持“结构化结果导向”的模式
- JSON 输出不是只在章节里讲讲，而是进入了实际项目

所以第七章把 `json_mode` 做成了运行时开关。

### 3.5 第五章能力：流式输出

流式输出在项目章里会同时出现在：

- CLI 里逐段打印
- API 里 SSE 返回

这比第五章更进一步。

第五章重点是理解“流式是什么”。

第七章重点是理解“流式怎么被项目入口复用”。

### 3.6 第六章能力：可靠性、成本和安全

项目章如果没有这些能力，就很容易退回“漂亮 demo”。

所以第七章要把这些能力真正接进来：

- usage 统计
- cache
- quota
- safety
- export

虽然这里仍然是教学级别的最小实现，但思路上已经是项目级的。

### 3.7 为什么第七章不是“新知识最多”，但却很重要

因为它不靠继续堆更多新概念取胜。

它的重要性在于：

> 让你第一次把零散能力装进一个真实结构里。

这一步如果没有迈过去，前六章知识很容易停留在“我看过、我跑过、但我自己很难组织起来”的阶段。

## 4. 为什么是 CLI + 服务层 + API 这套结构 📌

### 4.1 先看最容易写崩的版本

很多人第一版综合项目会直接写成：

```python
while True:
    user_text = input("> ")
    # 这里直接拼 messages
    # 这里直接调用模型
    # 这里直接 print
    # 这里顺便做导出
```

看起来很快。

但随着功能增加，它会迅速变得混乱。

比如你想加：

- `/json`
- `/stream`
- `/provider`
- `/export`
- `/stats`

就会发现：

1. 命令解析和业务逻辑混在一起
2. 会话状态无处可放
3. API 复用几乎不可能

### 4.2 CLI 层的职责应该是什么

CLI 层应该只做：

1. 接收用户输入
2. 识别命令
3. 调用服务层
4. 把结果打印给用户

CLI 不应该承担：

- provider 细节管理
- 成本计算核心逻辑
- 模型调用细节
- SSE 编码
- 导出底层实现

一旦 CLI 承担这些事，它就很难再被其他入口复用。

### 4.3 服务层的职责应该是什么

服务层是综合项目最关键的一层。

它负责：

1. 组织调用前后的完整流程
2. 维护会话状态
3. 构造请求
4. 调用模型
5. 处理错误
6. 记录 usage
7. 统计成本
8. 更新会话
9. 导出会话

你可以把它理解成：

> 项目内部真正稳定的业务接口。

### 4.4 API 层的职责应该是什么

API 层不应该重新发明一套逻辑。

它更像一个“协议适配层”。

它主要做：

1. 接收 HTTP 请求
2. 做请求体验证
3. 把参数映射到服务层
4. 把服务层结果返回为 JSON 或 SSE

如果 API 层写太重，你会发现：

- CLI 一套逻辑
- API 一套逻辑

这就破坏了项目章真正想建立的结构。

### 4.5 为什么这套结构对 AI 应用特别适合

因为 AI 应用里最不稳定的是外部调用环境。

可能会变的有：

- provider
- model
- Prompt
- UI 入口
- 调用方式

而最应该稳定的，是项目内部的调用语义。

比如：

- “发送一轮聊天请求”
- “流式返回一轮聊天结果”
- “导出当前会话”
- “查看当前统计”

这就是服务层要稳定下来的地方。

### 4.6 如果将来接前端页面，这套结构怎么继续用

这其实就是项目章的一个重要价值。

当你未来想接前端页面时：

- CLI 不一定再是主入口
- 但服务层仍然能用
- API 层甚至也可能继续用

也就是说：

第七章做的不是一个“最终形态项目”，而是一个：

> 能继续长到下一阶段的内核原型。

### 4.7 第七章目录结构怎么理解

本章代码目录大致是：

```text
07_chat_cli/
├── schemas.py
├── llm_service.py
├── chat_cli.py
├── chat_api.py
└── prompts/
```

你可以这样理解：

- `schemas.py`：项目内部共识的数据结构
- `llm_service.py`：项目内核
- `chat_cli.py`：命令行入口
- `chat_api.py`：HTTP 入口
- `prompts/`：Prompt 资产

这已经比单纯脚本目录更接近真实项目形态。

## 5. 项目分层之后，状态该放在哪里

### 5.1 综合项目里，状态不再只是 `messages`

第一章你关注的状态主要是：

- `messages`

到第七章，状态明显变多了。

至少包括：

- `session_id`
- `provider`
- `model`
- `system_prompt`
- `json_mode`
- `stream_mode`
- `temperature`
- `max_tokens`
- `keep_last_messages`
- `messages`

这说明什么？

说明聊天项目真正的状态，从来都不只是会话文本。

### 5.2 为什么这些状态不能只放在全局变量里

当然，你也可以这么写：

```python
provider = "bailian"
json_mode = False
stream_mode = False
messages = []
```

短期能用。

但一旦你想支持：

- 多会话
- API 重用
- 导出
- 更复杂的统计

你就会发现这种写法很脆弱。

因为状态之间缺少结构。

### 5.3 为什么要用 `ProjectSession`

项目章把这些运行时状态收到了 `ProjectSession` 里。

这么做的价值在于：

1. 状态有明确归属
2. 一个会话就是一个对象
3. CLI 和 API 都能围绕同一套状态结构工作
4. 导出时更容易完整落盘

### 5.4 会话对象里为什么要保留累计统计

比如：

- `accumulated_prompt_tokens`
- `accumulated_completion_tokens`
- `accumulated_total_tokens`
- `accumulated_estimated_cost`
- `turn_count`

这些字段不是为了“看起来更完整”。

它们的真实价值是：

1. 支持 `/stats`
2. 支持导出
3. 支持后续观察成本变化
4. 让会话不只是聊天文本，而是一个可观测对象

### 5.5 为什么要再定义 `TurnResult`

因为一次调用返回的结果，和整个会话状态，不是一个概念。

`ProjectSession` 关注的是：

- 会话累积状态

`TurnResult` 关注的是：

- 这一轮请求发生了什么

例如这一轮里，你会关心：

- provider
- model
- reply
- usage
- cost
- retries
- safety
- request_preview
- from_cache

这就是为什么第七章引入了单轮结果对象。

### 5.6 为什么这和前端组件的状态管理很像

如果你有前端经验，会发现这其实很熟悉。

你可以类比：

- `ProjectSession` 像页面级状态
- `TurnResult` 像一次 action 的结果
- `CLIState` 像运行时配置视图

这种类比很有帮助。

因为 AI 应用很多时候本质上就是：

> 一套围绕模型调用构建的状态管理系统。

### 5.7 状态对象不是越多越好，而是边界要清楚

项目章不是鼓励你把所有东西都拆成很多类。

真正重要的是：

1. 会话状态归会话状态
2. 单轮结果归单轮结果
3. 入口层不要自己维护太多隐式状态

只要这三件事清楚，项目就会稳很多。

## 6. 多轮对话 CLI 到底该怎么设计

### 6.1 为什么 CLI 不能只有“输入一句 -> 输出一句”

如果 CLI 只有下面这种形态：

```text
> 你好
AI: 你好
```

那它更像一个最小聊天样例。

项目章需要的是：

> 一个可操作、可切换、可调试的运行环境。

也就是说，CLI 不只是聊天窗口，它还是：

- 配置面板
- 调试入口
- 状态观察器
- Prompt 试验台

### 6.2 为什么用斜杠命令

斜杠命令有几个好处：

1. 用户输入和控制命令天然区分
2. 容易扩展
3. 对话语义清晰

比如：

- 普通文本：表示“这是业务输入”
- `/provider xxx`：表示“这是运行时指令”

这种设计在聊天产品、开发者工具里都很常见。

### 6.3 哪些命令是最关键的

本章项目里最关键的命令有：

#### `/provider`

切换模型平台。

它体现的是第二章的能力。

#### `/model`

手动覆盖模型名。

它让 provider 和 model 两个层次分开。

#### `/system`

动态改 system prompt。

它体现的是第三章 Prompt 工程能力进入运行时。

#### `/json`

切换结构化输出导向。

它体现的是第四章能力进入项目层。

#### `/stream`

切换流式输出。

它体现的是第五章能力进入项目层。

#### `/stats`

查看累计 token、成本、模式状态。

它体现的是第六章可观测性进入项目层。

#### `/export`

导出会话。

它体现的是项目回放、调试和留档能力。

#### `/prompt`

从文件加载 Prompt。

它体现的是 Prompt 资产化进入项目层。

### 6.4 为什么 `/json` 和 `/stream` 用“模式切换”而不是每次都手写参数

因为这更贴近真实项目中的“运行时状态”。

你当然也可以每次发消息时都手动指定：

- `json_mode=True`
- `stream_mode=True`

但在 CLI 里，更自然的方式是：

1. 先切状态
2. 再连续发多轮请求

这和真实用户在 UI 里切开关再继续使用，是同一种思路。

### 6.5 为什么 `/prompt` 设计成“读取并立即发送”

因为这样更贴近实际工作流。

很多时候你不是想“先把文本读进来存着”，而是想：

1. 从文件里拿一份任务模板
2. 直接把它发送给模型
3. 观察当前模式下的结果

这就是为什么 `/prompt` 在第七章里是一个真正的动作命令。

### 6.6 `/stats` 应该显示什么

如果 `/stats` 只显示“当前 provider”，那意义不大。

更有价值的是显示：

- session_id
- provider
- model
- json_mode
- stream_mode
- turn_count
- accumulated_total_tokens
- accumulated_estimated_cost
- estimated_tokens_current_history
- cache_size

这些字段能帮助你理解：

- 你当前在什么运行态
- 这次实验已经花了多少
- 历史是否膨胀
- 当前项目有没有缓存命中空间

### 6.7 CLI 最容易犯的一个错误：越写越像后端逻辑

很多人写 CLI，最后会把：

- provider 解析
- 模型调用
- 缓存逻辑
- usage 统计

全写进 `chat_cli.py`。

这是最危险的倾向之一。

CLI 层应该尽量薄。

如果一个功能未来既想给 CLI 用，也想给 API 用，那它大概率就不该写死在 CLI 层。

### 6.8 本章 CLI 的学习重点不是“命令多”，而是“命令背后的边界”

真正该理解的是：

- 哪些命令只改状态
- 哪些命令会触发调用
- 哪些命令只做观察
- 哪些命令做导出

这就是项目章开始出现“命令语义设计”的原因。

## 7. 为什么综合项目还要继续讲 Prompt 文件和 JSON 模式

### 7.1 因为项目最怕把前几章能力忘回去

很多课程到了项目章，会发生一种很常见的问题：

> 前面讲过的知识点，到项目里几乎看不见了。

这样项目章就失去了“综合”的意义。

所以第七章必须显式把这些能力接进来：

- Prompt 文件
- JSON 模式

### 7.2 Prompt 文件为什么是项目化的重要一步

Prompt 文件的价值，前几章已经讲过。

到了项目里，它会变得更明显。

因为你很快会遇到：

- 长 Prompt 难读
- 长 Prompt 难复用
- 长 Prompt 难比较版本差异

把 Prompt 放到 `prompts/` 目录里，至少带来这些好处：

1. 文件边界清晰
2. 更适合版本管理
3. 更适合共享和调试
4. CLI 可以直接加载

### 7.3 为什么 `/prompt <path>` 比“把文本粘进去”更有教学价值

因为它让学员第一次感受到：

> Prompt 不是一次性字符串，而是可以被工具链消费的项目资源。

这会直接影响后续：

- 做评测
- 做模板管理
- 做版本对比

### 7.4 第七章里的 JSON 模式，到底解决什么

项目里有些请求，明显是给人看的。

比如：

- 聊天问答
- 一段说明

但也有些请求更偏向给程序消费。

比如：

- 总结成结构化对象
- 输出一个简单的分析结果

所以项目章需要显式体现：

> 同一个项目，不同轮次可能需要不同输出取向。

这就是 `json_mode` 的意义。

### 7.5 为什么第七章里不把 JSON 模式做成“绝对严格保证”

因为项目章的重点不是再重复第四章的所有校验细节。

项目章更关注：

- 项目运行时怎样切换结构化输出导向
- CLI 和 API 如何感知这种模式

所以在这一章里，JSON 模式更像一个项目级接口语义：

- 用户打开这个模式
- 服务层按 JSON 输出导向组织请求
- 结果变得更适合程序读取

如果未来你要更强约束，就可以继续接回第四章的：

- Pydantic
- Schema 校验
- 自动修复与重试

### 7.6 为什么 JSON 模式和流式模式放在一起特别有意思

这是项目章里很值得观察的一点。

如果你：

1. 开了 `json_mode`
2. 又开了 `stream_mode`

你会发现：

- 从体验上看，它像普通聊天一样在逐段输出
- 从结果语义上看，它又更像结构化接口

这能帮助你意识到：

> 模式不是彼此独立的，而是可以组合的。

这也是项目章比单章练习更接近真实应用的地方。

## 8. 统一服务层到底收了什么逻辑 📌

### 8.1 如果没有服务层，项目会立刻散掉

你可以想象一下没有服务层的情况。

CLI 想做的事：

- 发送消息
- 切换模式
- 看统计

API 想做的事：

- 接收请求
- 返回 JSON
- 返回 SSE

如果没有服务层：

- CLI 自己拼 `messages`
- API 自己拼 `messages`
- CLI 自己做缓存
- API 自己做缓存
- CLI 自己做统计
- API 自己做统计

这会导致大量重复。

### 8.2 第七章服务层至少负责哪些事情

本章 `ProjectLLMService` 至少负责：

1. 解析 provider 配置
2. 获取或创建 session
3. 组织消息历史
4. 根据模式构造请求
5. 普通聊天调用
6. 流式聊天调用
7. usage 和 cost 计算
8. cache 和 quota
9. 导出当前会话
10. 生成会话快照

这其实已经是一套最小项目内核了。

### 8.3 为什么服务层要知道 session，而不是只接收 messages

因为项目级逻辑已经不仅仅是“这次请求要发什么”。

它还要知道：

- 当前 provider 是谁
- 当前 model 是谁
- 是否开了 JSON 模式
- 是否开了 stream 模式
- 历史要保留多少

这些都属于运行时状态。

而 session 正是这些状态的容器。

### 8.4 为什么服务层要返回结构化 `TurnResult`

这和第六章的思路一致。

项目层不能只满足于：

```python
return "一段回答"
```

因为调用结果不只是回答本身。

你还需要知道：

- 这一轮是否成功
- 是否来自缓存
- 是否是 mock
- 使用了哪个 provider / model
- usage 是多少
- cost 是多少
- 有没有重试
- 有无安全风险

这就是结构化返回值的价值。

### 8.5 为什么服务层里还保留 `request_preview`

很多人会觉得这个字段“很调试化”。

但它在教学和项目早期都非常有价值。

因为它能帮助你直接看到：

- 当前模式下到底发了什么请求
- provider 切换后模型名有没有变
- JSON 模式是否真的影响了请求结构
- 流式模式是否打开了

这对于学习和排查都很有帮助。

### 8.6 服务层里为什么还要继续做缓存和配额

因为项目章不是把第六章扔掉，而是要把第六章收进来。

如果综合项目里没有缓存和配额，那第六章就像和项目脱节了。

而一旦它们进入服务层，你就能看到：

- 第六章不是孤立知识
- 它真的会变成项目内核的一部分

### 8.7 为什么服务层做导出，而不是让 CLI 自己写文件

因为导出的对象其实是“会话”。

而会话的完整结构，服务层最清楚。

如果 CLI 自己负责导出，会很容易只导出：

- 一点消息文本

但更合理的导出内容通常包括：

- 会话状态
- 当前模式
- 统计信息
- 请求预览

这些都更适合由服务层统一组织。

### 8.8 第七章服务层不是最终答案，但方向是对的

要强调一点：

本章服务层是教学级最小实现。

它不是完整生产框架。

但它的方向是正确的：

- 能抽象
- 能复用
- 能扩展
- 能继续长

这对课程来说已经非常关键。

## 9. 普通输出、流式输出、JSON 模式为什么都要进同一个项目

### 9.1 因为真实项目不会只做一种调用形态

真实项目里，你经常会同时需要：

- 普通聊天
- 流式聊天
- 结构化输出

如果这些能力一直分散在不同章节脚本里，学员就很容易形成错误印象：

> 它们好像是互不相关的三种专题。

但项目章要让你看到：

> 它们其实都是同一个项目运行时里的不同模式。

### 9.2 普通输出在项目里仍然有价值

不是所有请求都适合流式。

也不是所有请求都需要 JSON。

所以普通输出仍然是最基础、最稳定的模式。

项目里保留普通输出的意义在于：

- 它是最清晰的基线
- 它更适合短回答
- 它更适合某些结构化任务

### 9.3 流式输出进入项目层之后，重点不再只是“看起来很酷”

第五章里你理解了流式为什么能提升体验。

第七章里你要再往前看一步：

流式进入项目层后，意味着：

- 会影响 CLI 的打印方式
- 会影响 API 的协议返回方式
- 会影响错误处理方式
- 会影响是否能中途做重试

也就是说：

> 流式不是一个“显示效果”，而是项目层的运行模式。

### 9.4 JSON 模式进入项目层之后，重点不再只是“返回像 JSON”

第四章里你重点关注：

- JSON 是否能解析
- Schema 是否能验证

第七章里你要再看一步：

JSON 模式成为项目状态之后，会影响：

- CLI 的输出期待
- Prompt 文件写法
- API 的消费方式
- 未来的下游扩展能力

### 9.5 三种模式如何共存

你可以把项目里的输出模式理解为三条维度：

1. 是否普通 / 流式
2. 是否人类可读 / 更偏结构化
3. 是否只用于展示 / 还要继续给程序消费

项目章的价值之一，就是让你开始用“模式组合”而不是“单独专题”的视角去理解这些能力。

### 9.6 为什么第七章里还要保留 mock 模式

因为项目章的重点不只是对真实平台发请求。

更重要的是：

1. 理解结构
2. 理解状态
3. 理解命令
4. 理解导出
5. 理解模式切换

这些能力即使在没有真实 Key 的情况下也应该可练习。

这也是为什么项目章继续保留 mock 的原因。

## 10. 为什么项目里还要保留成本统计、导出和会话回放

### 10.1 很多 demo 最大的问题：结果一闪而过

很多聊天 demo 的典型问题是：

- 你问了一句
- 模型回了一句
- 然后这轮结果就只存在于终端滚动记录里

这对展示还行，对项目成长很不利。

因为你无法：

- 回放
- 对比
- 留档
- 排查

### 10.2 导出能力为什么在项目章非常重要

导出是项目迈向“可回顾”的关键一步。

一旦你能导出会话，就意味着你可以做这些事：

1. 回看本次实验
2. 对比不同 provider 的结果
3. 观察同一 Prompt 在不同模式下的输出差异
4. 做最小级别的回归检查

### 10.3 成本统计为什么也必须跟着导出一起看

因为真实项目里，你不只关心：

- 输出好不好

你还关心：

- 花了多少
- 历史会不会越来越贵
- 哪种模式更消耗 token

所以项目导出和统计放在一起，才更完整。

### 10.4 为什么 `/stats` 和 `/export` 是项目级命令，不是“调试彩蛋”

这两个命令不是为了好玩。

它们其实对应了真实项目里很核心的两类需求：

#### `/stats`

对应的是：

- 实时运行态观察

#### `/export`

对应的是：

- 离线留档与回看

这就是为什么第七章把它们提到和 `/provider`、`/json` 同一层级。

### 10.5 会话回放为什么对 Prompt 工程也有价值

因为很多 Prompt 优化不是看单次结果，而是看多轮过程。

例如你想回看：

- 这个 system prompt 是怎么影响后续几轮的
- 开启 JSON 模式后结构有没有变稳
- provider 切换前后风格差异如何

这些都离不开会话记录。

### 10.6 这为后续课程埋下了什么能力

到了 RAG 和 Agent，你会更需要：

- 记录完整上下文
- 记录中间步骤
- 导出链路结果
- 做回归和排查

所以第七章的导出和统计，不只是为了当前这一个项目。

它其实是在为后续课程做工程意识铺垫。

## 11. 为什么同一套服务层一定要同时支撑 CLI 和 API 📌

### 11.1 这不是“多做一点”，而是验证结构是否真的成立

如果一个项目只能在 CLI 里工作，说明它可能还没有真正抽象好。

因为：

- 输入方式写死了
- 输出方式写死了
- 协议边界不清楚

而一旦你让它同时支撑 CLI 和 API，就会逼你把这些边界整理清楚。

### 11.2 API 层应该尽量薄

第七章里 `chat_api.py` 的重点不是写很多业务逻辑。

而是体现一种结构：

1. 请求进来
2. 映射到 session 设置
3. 调服务层
4. 返回结果

如果 API 层代码越来越厚，通常说明服务层抽象得还不够好。

### 11.3 普通聊天接口和流式接口为什么都要有

因为它们代表两种不同消费方式：

#### 普通接口

- 适合短请求
- 适合调试
- 适合更标准的 JSON 返回

#### 流式接口

- 适合长回答
- 适合聊天体验
- 适合前端逐段渲染

项目章把两者都放进来，能帮助你建立完整的入口视角。

### 11.4 为什么 API 层仍然需要 session_id

因为多轮会话在 API 层一样存在。

很多人会误以为：

- CLI 才是多轮
- API 就是单次请求

其实不是。

只要是聊天系统，API 一样会需要：

- `session_id`
- 历史管理
- 模式状态

### 11.5 为什么这套结构未来接 Web UI 很自然

如果将来要接前端页面，前端做的事无非是：

1. 发送用户消息
2. 接收普通 JSON 或 SSE
3. 展示会话历史

而这些事，本章已经用 API 骨架体现出来了。

这就是为什么项目章虽然没做 UI，却仍然很有项目感。

### 11.6 为什么这一章不直接做浏览器 UI

不是因为 UI 不重要。

而是因为课程当前目标更重要的是：

> 先把项目内核做对。

如果内核没有建立好，前端 UI 很容易只是在包一个不稳定的 demo。

### 11.7 CLI 和 API 同时存在，对你学习上最大的帮助是什么

它会强迫你从“入口视角”切换到“内核视角”。

这一步非常关键。

因为真正成熟的工程思维，从来不是：

> 这个界面怎么做。

而是：

> 不同界面怎样共用同一套能力。

## 12. 第七章代码结构逐文件解释

### 12.1 `schemas.py`

这个文件的作用不是“为了面向对象而面向对象”。

这是第七章最核心的文件。

它不是辅助工具，而是真正的项目内核。

它里面承担的职责包括：

- provider 配置读取

- 重试

- quota
- 普通聊天
- 流式聊天
- 会话导出

如果你只看一个文件来理解第七章，优先看这个。

### 12.3 `chat_cli.py`

- 命令解析

它体现的是项目入口层的职责边界。

- Prompt 资产化

### 12.6 `.env.example`

这个文件继续保持前几章的 provider 配置风格。

这样做的意义是：

- 学员不会在第七章突然换一套配置体系
- 前六章的配置经验可以直接延续

## 13. 综合项目最容易出现的几个设计误区

### 误区 1：把所有逻辑都堆进 `chat_cli.py`

这是最常见的错误。

一开始看着很快，后面会非常难扩展。

### 误区 2：项目章只做一个能跑的聊天循环就算完成

不够。

项目章的关键是：

- 状态
- 分层
- 导出
- 统计
- 模式切换

### 误区 3：CLI 和 API 各写一套逻辑

这会直接破坏综合项目的教学价值。

因为你没有真正建立统一服务层。

### 误区 4：把 JSON 模式、流式输出、provider 切换看成完全独立功能

项目里它们其实都是“运行时模式”的一部分。

### 误区 5：项目章就应该立刻做完整前端

不一定。

项目章最重要的是内核正确，而不是表面更复杂。

### 误区 6：只关注“能回答”，忽略统计和导出

这样做出来的项目很容易停留在演示层，难以继续演进。

### 误区 7：有 mock 就等于代码不真实

不是。

mock 在课程里承担的是：

- 降低试验门槛
- 保证可学习性
- 让结构先站住

只要真实路径也存在，mock 就是合理的教学工具。

第七章最重要的收获，不是“我又写了一个聊天工具”。

1. 你第一次把前六章能力真正装进了一个项目结构里
