# 04. 结构化输出

> 本节目标：掌握把 LLM 自由文本输出收敛成可解析、可验证、可重试、可复用结构化结果的方法，理解 Prompt JSON、JSON Mode、Pydantic Schema、校验重试与通用提取器之间的关系

## 1. 概述

### 学习目标

- 理解为什么 AI 应用开发不能长期依赖“看起来差不多对”的自由文本输出

- 掌握从模型输出中提取、解析和校验 JSON 的基本方法
- 掌握用 Pydantic 定义输出 Schema、做字段类型校验和统一输出对象
- 掌握结构化输出失败后的重试与修复思路
- 能写出一个根据 Schema 自动生成 Prompt、批量处理文本并导出报告的最小提取器

## 2. 为什么结构化输出是 AI 应用开发必修课 📌

### 2.1 从“模型回答得不错”到“程序真的能用”

很多人在接入 LLM 的早期，关注点主要是：

- 回答像不像人
- 内容是否有帮助
- 风格是否自然

这在纯聊天产品里没有问题。

但一旦进入应用开发，你会很快遇到另一个问题：

> 你不只是想把回答展示给人，还想把结果继续交给程序。

比如：

- 把工单分类结果写入数据库
- 把简历提取结果填进表单
- 把客户信息同步到 CRM
- 把模型提取的字段继续传给下一步服务

这时“回答得挺像那么回事”已经不够了。

你真正关心的是：

1. 字段是否齐
2. 类型是否对
3. 结果是否能稳定解析
4. 下游逻辑能不能直接消费

### 2.2 自由文本为什么天然不适合做程序接口

看一个例子。

用户给模型一句话：

```text
张三，男，28岁，北京人，软件工程师。
```

如果你只让模型“提取信息”，它可能这样回答：

```text
这段文字描述的是一位叫张三的男性，28 岁，来自北京，从事软件工程师工作。
```

从人类角度看，这段话完全没问题。

但从程序角度看，它有很多问题：

- 没有字段名
- 年龄是文本还是数字不明确
- 下游代码需要自己再做字符串解析
- 稍微换种说法就可能崩

也就是说：

> 自由文本对人类友好，但对程序脆弱。

### 2.3 结构化输出到底解决什么

结构化输出解决的是：

1. **让结果有固定字段**
2. **让字段有固定类型**
3. **让下游逻辑不再依赖自然语言猜测**
4. **让错误更容易定位**

还是同样的输入，如果模型返回：

```json
{
  "name": "张三",
  "gender": "男",
  "age": 28,
  "city": "北京",
  "job": "软件工程师"
}
```

它就立刻具备了程序价值：

- 可以直接存数据库
- 可以映射到表单
- 可以进一步校验
- 可以作为 API 返回值

### 2.4 哪些场景最需要结构化输出

结构化输出不是所有任务都必须上，但以下场景基本都需要：

#### 场景 1：信息提取

例如：

- 从简历中提取姓名、年龄、技能
- 从销售记录中提取公司、预算、意向等级
- 从工单中提取问题类型、优先级、是否需要人工介入

#### 场景 2：分类与标签化

例如：

- 情感分类
- 风险级别分类
- 工单类型分类
- 是否违规判断

#### 场景 3：程序链路衔接

例如：

- 模型先提取 JSON
- 程序拿 JSON 调内部 API
- 再把结果喂给下一步模型或服务

#### 场景 4：数据入库与报表

例如：

- 批量提取线索并入 CRM
- 批量整理访谈纪要
- 批量解析用户反馈

### 2.5 第三章的 Prompt 工程，为什么还不够

你可能会问：

> 第三章不是已经讲了输出格式吗？为什么第四章还要单独讲结构化输出？

因为这两者解决的问题不一样。

第三章更多解决的是：

- 如何让任务更清楚
- 如何让回答更稳定
- 如何让格式不那么飘

但第四章要解决的是更严格的问题：

- 程序能否稳定解析
- 字段能否做类型校验
- 枚举值是否合法
- 失败后如何修复

也就是说：

> “格式比较整齐”不等于“已经是可靠的结构化输出”。

### 2.6 结构化输出常见误区

#### 误区 1：只要看起来像 JSON 就行

例如模型返回：

```text
当然可以，以下是结果：

{
  "name": "张三"
}
```

人类能接受，但程序如果直接 `json.loads()` 整段文本就会失败。

#### 误区 2：只要能 `json.loads()` 就行

即使下面这个 JSON 能解析：

```json
{
  "name": "张三",
  "age": "二十八",
  "priority": "urgent"
}
```

它仍然可能不符合你的业务规则：

- `age` 应该是整数
- `priority` 只允许 `高 / 中 / 低`

#### 误区 3：失败了就再试一次

很多人遇到结构化输出错误后的第一反应是：

> 再跑一次看看。

但如果你不告诉模型“哪里错了”，那重试往往只是换一种错法。

#### 误区 4：平台原生能力永远可依赖

现实里，不同平台对结构化输出的支持：

- 能力不同
- 命名不同
- 稳定性不同
- 兼容程度不同

所以不能把课程设计成只适用于某一个平台的特性。

### 2.7 第四章真正要建立的能力链

这章最重要的不是单个技巧，而是一条完整链路：

```text
原始文本
-> Prompt 指定结构
-> 模型输出 JSON
-> 程序解析 JSON
-> Pydantic 校验 Schema
-> 失败则反馈错误重试
-> 成功后得到统一输出对象
-> 批量处理 / 入库 / 导出
```

你后面看到的 4 个实践脚本，其实就是沿着这条链逐步展开的。

## 3. 从 Prompt JSON 到 JSON Mode 📌

### 3.1 最基础的结构化输出方式：在 Prompt 里明确写 JSON

第四章最小可行起点不是平台原生能力，而是：

> 在 Prompt 里明确要求模型只输出 JSON。

例如：

```text
请从下面文本中提取信息，并以 JSON 对象输出。

文本：
张三，男，28岁，北京人，软件工程师。

输出要求：
1. 只输出 JSON，不要额外解释。
2. 字段必须包含：
   - name: string
   - gender: string
   - age: integer
   - city: string
   - job: string
3. 如果文本里没有对应信息，填 null。
```

这种方式的优点是：

1. 所有平台都能用
2. 容易理解
3. 适合作为最低可行方案

### 3.2 为什么 Prompt JSON 已经比“自由文本”强很多

因为它至少明确了三层约束：

1. **输出类型**：JSON 对象
2. **字段集合**：`name / gender / age / city / job`
3. **字段类型**：string / integer / null

这会显著减少模型的自由发挥空间。

### 3.3 但 Prompt JSON 仍然不是绝对可靠

即使你写得很清楚，模型仍然可能输出：

#### 情况 1：带解释文字

```text
以下是提取结果：
{
  "name": "张三"
}
```

#### 情况 2：放进 Markdown 代码块

```text
```json
{
  "name": "张三"
}
```
```

#### 情况 3：字段类型错

```json
{
  "name": "张三",
  "age": "28"
}
```

#### 情况 4：字段漏了

```json
{
  "name": "张三",
  "age": 28
}
```

这就是为什么 Prompt JSON 只是第一层，而不是全部。

### 3.4 为什么“只输出 JSON”这种约束仍然非常重要

即使后面你要用 JSON Mode 或原生 Structured Outputs，这句要求仍然经常有价值。

因为它会帮助模型形成更明确的任务预期：

- 当前重点是结构而不是解释
- 当前结果要给程序而不是给人
- 当前多余自然语言是噪声

这可以理解成一种 Prompt 层的“前置收缩”。

### 3.5 JSON Mode 是什么

在 OpenAI 风格接口里，常见的一个能力是：

```python
response_format={"type": "json_object"}
```

它的思路不是只靠 Prompt 约束，而是：

> 在接口层告诉模型，本次结果应该是 JSON 对象。

也就是说，JSON Mode 比 Prompt JSON 更像一种：

- 请求级约束
- 接口级能力
- 平台能力位

### 3.6 Prompt JSON 和 JSON Mode 的差异

可以把它们简单对比成：

| 方式 | 约束层 | 优点 | 风险 |
|------|--------|------|------|
| Prompt JSON | Prompt 层 | 平台通用、最低可行 | 更依赖模型遵守 |
| JSON Mode | 接口层 | 结构更稳 | 平台支持差异大 |

非常关键的一点是：

> JSON Mode 也不等于“业务字段绝对合法”。

它更多解决的是“输出形状更像合法 JSON”，并不自动保证：

- 字段一定存在
- 类型一定符合你的业务 Schema
- 枚举值一定合法

这就是为什么第四章后面还要讲 Pydantic。

### 3.7 原生 Structured Outputs 又是什么

比 JSON Mode 更进一步的，是某些平台提供的原生 Structured Outputs 能力。

简单理解：

- 你不仅要求它返回 JSON
- 还希望它更贴近你定义的 Schema

这通常比普通 JSON Mode 更强。

但在课程里你要保持一个非常现实的认知：

1. 不同平台能力差异明显
2. OpenAI-compatible 不等于“完全兼容原生 Structured Outputs”
3. 真正工程里，不能把所有可靠性都赌在平台原生能力上

所以本章的教学顺序是：

1. 先会 Prompt JSON
2. 再理解 JSON Mode
3. 再理解原生 Structured Outputs 的价值
4. 但最终仍然回到 Schema 校验和重试

### 3.8 为什么课程强调“可回退方案”

你的实际练习平台更多是：

- 百炼 / 通义
- DeepSeek
- GLM

这些平台很多支持 OpenAI-compatible 协议，但你必须保留一个工程边界意识：

> 兼容聊天调用，不代表对所有结构化高级能力的支持都完全一致。

所以一个成熟应用不能写成：

```text
如果平台没原生结构化输出，我的链路就废了
```

正确思路应该是：

```text
能用原生能力时优先用
不能用时退回 Prompt JSON + 解析 + Pydantic + 重试
```

这才是平台无关的工程方案。

### 3.9 自由文本、Prompt JSON、JSON Mode 的工程分层

你可以先建立这张表：

| 层级 | 适合什么场景 | 可靠性 |
|------|-------------|-------|
| 自由文本 | 展示给人看 | 低 |
| Prompt JSON | 基础结构提取 | 中 |
| JSON Mode | 更稳定的 JSON 形状 | 中高 |
| 原生 Structured Outputs + Schema 验证 | 更成熟的生产链路 | 更高 |

但无论你在哪一层，都不要跳过“程序侧校验”。

### 3.10 从文本里提取 JSON 时，程序要怎么做

程序侧最基本的解析动作通常是：

1. 去掉首尾空白
2. 去掉代码块
3. 尝试找到 JSON 片段
4. `json.loads()`
5. 再做 Schema 验证

第四章公共工具里实现了对应逻辑：

- `strip_code_fence()`
- `find_json_candidate()`
- `parse_json_output()`

这些函数看起来不复杂，但很重要。

因为真实项目里，你经常拿到的不是“完美 JSON”，而是：

- 被 Markdown 包了一层
- 前后带解释
- 只有中间一段是对象

### 3.11 第 1 个实践脚本要看什么

对应代码在：

- `source/02_llm/04_structured_output/01_json_output.py`
- `source/02_llm/04_structured_output/structured_utils.py`

脚本会对比：

1. 自由文本 Prompt
2. Prompt 指定 JSON
3. JSON Mode 请求预览

你要重点观察：

1. 为什么自由文本对人类好，但对程序不好
2. 为什么 Prompt JSON 是最低可行方案
3. 为什么 JSON Mode 有价值，但仍不足以替代 Schema 校验

### 3.12 一个工程判断标准

如果你的下游代码要直接用模型结果做逻辑判断、数据入库、接口返回，那默认就应该问自己：

> 这一段输出，能否被稳定解析和校验？

如果答案是否定的，那你就还停留在“看起来能用”，没有进入“程序真的能用”。

## 4. 为什么 JSON 还不够，必须有 Schema 📌

### 4.1 JSON 是格式，不是业务约束

这是第四章最重要的一句话之一：

> JSON 只是一种数据表达格式，不是业务规则本身。

例如下面这个结果：

```json
{
  "name": "王敏",
  "budget": "5万",
  "intent_level": "很高"
}
```

它是不是合法 JSON？

是。

但它是不是一定符合你的程序要求？

不一定。

因为你的程序可能需要的是：

- `budget` 是整数，单位元
- `intent_level` 只能是 `high / medium / low`

这时就需要一层更严格的结构定义。

### 4.2 Schema 的作用是什么

Schema 可以理解成：

> 结构化输出的“类型系统”。

它至少规定了：

1. 有哪些字段
2. 每个字段是什么类型
3. 哪些字段必填
4. 哪些字段可选
5. 某些字段的值域约束

在 Python 里，Pydantic 是非常适合做这件事的工具。

### 4.3 为什么第四章选 Pydantic

原因很现实：

1. Python AI 项目里非常常见
2. 语法相对直观
3. 和 FastAPI、后端 API、数据模型有天然衔接
4. 既能定义结构，又能做验证，又能导出 Schema

这意味着它不只是“为了 LLM 特地学一个库”，而是：

> 后续做真实 AI 应用时，本来就会频繁用到。

### 4.4 一个最简单的 Pydantic 模型

例如：

```python
from pydantic import BaseModel, Field


class Person(BaseModel):
    name: str = Field(description="姓名")
    age: int = Field(ge=0, le=150, description="年龄")
    city: str = Field(description="所在城市")
```

这个模型表达了：

- `name` 是字符串
- `age` 是整数，并且有取值范围
- `city` 是字符串

和纯 JSON 相比，差别非常大。

### 4.5 为什么 `Field()` 很重要

`Field()` 不只是让代码更好看，它承载了很多关键信息：

#### 作用 1：补描述

```python
name: str = Field(description="客户姓名")
```

这个描述后面可以用于：

- 生成 JSON Schema
- 自动生成 Prompt 描述
- 给开发者做文档说明

#### 作用 2：补约束

```python
budget: int | None = Field(default=None, ge=0)
```

它明确告诉程序：

- 预算是整数
- 允许为空
- 不能小于 0

#### 作用 3：统一业务规则

一旦 Schema 定义好，下游很多地方都可以复用：

- LLM 输出校验
- API 返回值
- 数据清洗
- 单元测试

### 4.6 必填字段、可选字段和默认值

这是结构化输出里非常容易忽略的一层。

例如：

```python
email: str | None = None
```

这表达的是：

- 这个字段可以为空
- 如果模型提不出来，不要硬编

而：

```python
name: str
```

表达的是：

- 这是关键字段
- 没有它就不应该算合法结果

这对结构化提取特别重要。

因为真实文本经常有信息缺失。

Schema 的职责不是强迫模型编造，而是：

> 明确哪些地方允许缺失，哪些地方绝不能乱填。

### 4.7 枚举值为什么比自由字符串更重要

很多业务字段其实不该随便输出字符串。

例如工单优先级，如果模型可以自由写：

- 高
- 重要
- 紧急
- urgent
- P1

那下游逻辑会变得非常混乱。

更好的方式是：

```python
priority: Literal["高", "中", "低"]
```

这样你就能强制它收敛到固定集合。

这和第三章 Few-shot 里“标签集合必须固定”是同一类问题，只不过这里更工程化。

### 4.8 嵌套对象为什么值得提前掌握

很多真实业务对象不是平的。

例如销售线索，除了主信息，还可能包含联系方式对象：

```python
class ContactChannel(BaseModel):
    email: str | None = None
    wecom: bool
```

然后：

```python
class LeadRecord(BaseModel):
    name: str
    company: str
    contact: ContactChannel
```

这时你就会得到一个更接近真实业务结构的输出对象。

如果你只会扁平字段，后面接复杂接口时很容易返工。

### 4.9 Schema 不只是给程序看的，也可以反向喂给模型

这是第四章很重要的一点。

一旦你有了 Schema，就可以把它转换成更清楚的 Prompt 描述。

比如自动生成：

```text
- name: string, required, 客户姓名
- company: string, required, 客户所属公司
- budget: integer, optional, 预算金额，单位元
- intent_level: string, required, allowed=['high', 'medium', 'low']
```

这比手写一堆字段说明更不容易漂。

因为它来自同一份 Schema 定义。

也就是说：

> Schema 可以同时服务于“程序校验”和“Prompt 约束”。

### 4.10 Schema 驱动 Prompt 的价值

如果没有 Schema 驱动，你很容易出现这种情况：

- 代码里要求 `budget` 是整数
- Prompt 里却还写“预算可写成 5万”

或者：

- 代码里已经增加了 `need_demo`
- Prompt 里却忘了更新

这种不一致是结构化输出链路里非常常见的 bug 来源。

而 Schema 驱动的好处是：

1. 结构定义单点维护
2. Prompt 可从结构定义推导
3. 校验也基于同一份定义

### 4.11 第 2 个实践脚本做了什么

对应代码：

- `source/02_llm/04_structured_output/02_pydantic_schema.py`

你会在里面看到：

1. `LeadRecord`
2. `ContactChannel`
3. `get_model_json_schema()`
4. `schema_to_prompt_description()`
5. `model_validate()`

脚本做的事情是：

1. 定义一个真实业务对象
2. 导出 JSON Schema
3. 把 Schema 转成 Prompt 描述
4. 发起结构化提取
5. 再用 Pydantic 验证模型结果

### 4.12 为什么说 Schema 是“单一事实来源”

如果这章只记住一个工程概念，建议记住这个：

> Schema 应该成为结构化输出链路里的单一事实来源。

因为一旦结构定义散落在：

- Prompt 一份
- Python 字典一份
- API 文档一份
- 前端类型一份

后面几乎一定会出现不一致。

第四章做的，就是先把 Python 侧这一层稳定下来。

## 5. 输出校验、错误反馈与重试 📌

### 5.1 结构化输出最常见的错误分哪两类

你最好从一开始就把错误分成两层：

#### 第一层：解析错误

例如：

- 不是合法 JSON
- 前后夹着解释文字
- Markdown 代码块没有去掉
- 花括号不完整

这类错误发生在：

> `json.loads()` 之前或过程中

#### 第二层：验证错误

例如：

- 缺字段
- 类型错
- 枚举值非法
- 长度或范围不符合约束

这类错误发生在：

> JSON 已经能解析，但还不符合业务 Schema

把这两类错误分清楚非常重要。

因为它们的修复策略不一样。

### 5.2 为什么不能把 parse 错误和 validate 错误混为一谈

例如下面这个输出：

```text
结果如下：
{
  "priority": "高"
}
```

如果你的程序直接拿整段文本去 `json.loads()`，它会先报解析错误。

但如果你把 JSON 片段提出来之后，再发现：

```json
{
  "priority": "urgent"
}
```

这就不是解析错误，而是验证错误。

如果你不区分，就很难知道：

- 是 Prompt 没强调“只输出 JSON”
- 还是 Schema 约束没传达清楚

### 5.3 很多人重试失败，是因为“重试内容没变”

常见错误写法：

```python
for _ in range(3):
    result = call_llm(prompt)
    try:
        return validate(result)
    except:
        continue
```

这里最大的问题是：

> 失败后你没有告诉模型哪里错了。

也就是说，你不是在“修复”，而是在“重复碰运气”。

### 5.4 正确重试的核心：把错误显式反馈给模型

更合理的修复 Prompt 应该长这样：

```text
上一次结构化输出校验失败，请只返回修正后的 JSON。

错误信息：
priority: 输入值必须是 高 / 中 / 低 之一
need_human_follow_up: 应为布尔值

上一次输出：
{
  "priority": "urgent",
  "need_human_follow_up": "yes"
}

目标 Schema：
- priority: string, required, allowed=['高', '中', '低']
- need_human_follow_up: boolean, required

要求：
1. 只输出一个合法 JSON 对象。
2. 保留已有正确字段，修复缺失字段、类型错误和非法取值。
3. 如果信息不足，用 null 或空数组，不要编造新事实。
```

这个修复 Prompt 的价值是：

- 错误显式
- 目标清楚
- 输出范围更小
- 保留正确部分

### 5.5 为什么错误反馈比“再多加几句原始要求”更有效

因为修复阶段面对的是一个更窄的问题：

- 不是让模型重新理解整个任务
- 而是让模型修正已经明确识别出的错误

这时错误提示比重新讲任务更有针对性。

### 5.6 重试次数应该怎么设

一般来说：

- 1 次：适合极简单任务
- 2-3 次：大多数场景够用
- 超过 3 次：通常说明问题不是偶发，而是系统性问题

超过 3 次还不稳定，通常要回头检查：

1. Prompt 是否写清楚了
2. Schema 是否太复杂
3. 当前模型是否适合该任务
4. 是否应该改用原生 Structured Outputs 或 Tool Calling

### 5.7 重试时参数应该怎么设

结构化输出场景里，通常比普通聊天更偏向：

- `temperature=0.0`
- 更明确的格式约束
- 更短的输出空间

原因很简单：

你追求的是：

- 稳定
- 一致
- 可校验

而不是创造性。

### 5.8 什么时候应该停止自动重试

这件事非常重要。

如果一个结果连续修复仍然失败，继续自动重试可能带来：

- 成本增加
- 延迟增加
- 错误更隐蔽

这时更合理的策略可能是：

1. 标记失败
2. 进入人工检查队列
3. 记录失败样本用于后续 Prompt 迭代

工程成熟度的体现之一，不是“永远自动成功”，而是：

> 明确知道什么时候该自动，什么时候该停止自动。

### 5.9 为什么失败样本值得保留

很多团队只保留成功结果，不保留失败样本。

这会浪费非常有价值的调优材料。

失败样本至少能帮助你：

1. 看出哪些字段最容易错
2. 看出哪类文本最容易失败
3. 看出是解析层问题还是 Schema 层问题
4. 为下一轮 Prompt 迭代准备回归集

### 5.10 第 3 个实践脚本做了什么

对应代码：

- `source/02_llm/04_structured_output/03_retry_validation.py`

脚本里实现了一个最小 `RobustStructuredClient`，核心逻辑是：

1. 先发起结构化提取
2. 先做 JSON 解析
3. 再做 Pydantic 验证
4. 如果失败，就生成修复 Prompt
5. 再次调用模型
6. 成功或达到最大重试次数为止

这个脚本的重点不是“重试次数怎么写”，而是让你建立：

> 解析错误、验证错误、修复 Prompt、重试边界，这四层是如何串起来的。

### 5.11 一个很实用的工程习惯

如果你要做生产可用的结构化输出，建议每次至少记录：

1. 原始 Prompt
2. 原始输出
3. 解析错误或验证错误
4. 修复 Prompt
5. 最终结果

这样一旦线上出问题，你能快速回答：

- 到底错在哪一步
- 模型第一次输出长什么样
- 修复策略有没有效果

而不是只剩下一句“这次抽取失败了”。

## 6. 从单次提取走向通用提取器 📌

### 6.1 为什么只会“提一条”还不够

前面几节你已经会了：

- 让模型返回 JSON
- 用 Pydantic 校验
- 失败时修复重试

但真实项目里，你最终要解决的通常不是：

> 提取这一条文本

而是：

> 提取这一类文本，并且能批量运行、导出结果、统计成功率。

这就是为什么第四章最后要落一个通用提取器。

### 6.2 通用提取器到底抽象了什么

一个最小通用提取器通常要抽象 4 件事：

1. 输入文本
2. 目标 Schema
3. 提取 Prompt 生成逻辑
4. 解析 / 校验 / 报告流程

也就是说，真正复用的不是某条固定 Prompt，而是：

> 一条由 Schema 驱动的结构化提取流程。

### 6.3 为什么 Schema 驱动提取器特别重要

如果没有 Schema 驱动，你做批量提取时很容易出现：

- 每个任务都有自己一套字符串 Prompt
- 每个任务都有自己一套解析逻辑
- 每个任务都有自己一套字段校验

最后系统会越来越散。

而 Schema 驱动的方式更像这样：

```text
输入文本
-> 读取 Schema
-> 根据 Schema 生成 Prompt 描述
-> 调模型
-> 解析 JSON
-> Pydantic 验证
-> 统一输出或报错
```

这条链路具备更强的可扩展性。

### 6.4 批量处理时，最该关注什么

很多人做批量提取时，只看最终有多少条成功。

这还不够。

至少应该关心：

1. 成功多少条
2. 失败多少条
3. 失败在 parse 还是 validate 阶段
4. 哪些文本最容易失败
5. 哪些字段最容易错

只有这些信息都有，你才知道下一步该优化 Prompt、Schema，还是该换模型。

### 6.5 为什么报告导出非常重要

批量提取跑完后，如果只在终端看一眼结果，很快就会丢掉。

更合理的做法是导出：

- 成功结果
- 失败结果
- 错误阶段
- 原始输出

这样你后面可以：

1. 做回归测试
2. 做 Prompt 迭代
3. 做人工复核
4. 做效果复盘

### 6.6 一个真实工程里很常见的形态

假设你要处理一批销售记录：

```text
王敏来自杭州的华星零售，准备给 30 人团队采购 AI 知识库产品。
预算大概 5 万，希望下周安排线上演示。
```

目标不是让模型写一段总结，而是提取：

- name
- company
- city
- intent_level
- budget
- next_action

然后：

- 成功的写入 CRM
- 失败的进人工复核队列

这就是结构化输出真正的业务语境。

### 6.7 通用提取器为什么很适合做章节收尾案例

因为它把第四章前面所有能力都串起来了：

1. Prompt JSON
2. Schema 描述
3. JSON 解析
4. Pydantic 校验
5. 失败分层
6. 批量处理
7. 导出结果

当这几个点连起来，你就不再只是“懂几个知识点”，而是已经能做一个真实的结构化链路。

### 6.8 第 4 个实践脚本做了什么

对应代码：

- `source/02_llm/04_structured_output/04_extractor.py`
- `source/02_llm/04_structured_output/data/customer_notes.json`

脚本里定义了：

1. `SalesLead` Schema
2. `StructuredExtractor`
3. `extract_one()`
4. `extract_batch()`

它做的事情是：

1. 读取一批销售跟进记录
2. 根据 Schema 自动生成 Prompt
3. 逐条调用模型
4. 解析并校验
5. 统计成功 / 失败
6. 导出报告到 `exports/`

### 6.9 通用提取器的边界也要清楚

第四章的提取器是“最小可用结构化提取器”，不是完整生产系统。

它还没有展开：

- 并发控制
- 异步批处理
- 限流与熔断
- 数据库存储
- 审计日志
- 人工复核后台

但它已经足够帮你建立：

> 结构化输出不是一个单独 API 参数，而是一条完整工程链路。

### 6.10 什么时候你已经具备第四章能力

如果你能独立做下面这件事，就说明第四章已经学到位：

1. 定义一个业务 Schema
2. 让模型按这个 Schema 输出 JSON
3. 解析结果
4. 用 Pydantic 校验
5. 出错时反馈错误修复
6. 批量跑一组数据并输出报告

做到这一步，你就已经具备了很多 AI 应用开发中最基础、最实用的“把模型结果接进程序”的能力。

## 8. 小结

第四章真正建立的是一条“模型结果进入程序”的能力链，而不是单个 API 技巧。

你要记住的主线有 6 条：

1. 自由文本适合展示给人，不适合直接给程序做逻辑判断
2. Prompt 指定 JSON 是最低可行结构化方案
3. JSON Mode 和原生 Structured Outputs 有价值，但不能取代程序侧校验
4. JSON 只是格式，Schema 才是业务规则
5. 结构化输出失败后，要把错误显式反馈给模型，而不是只重跑
6. 真正可落地的工程形态，是一个由 Schema 驱动的提取、校验、重试和导出链路

这章之后，你已经具备了做很多实际 AI 功能的基础：

- 自动提取表单字段
- 自动分类和打标签
- 自动整理工单
- 批量解析销售线索

而这些能力会继续成为后面章节的基础：

- 第五章的流式输出解决“怎么把结果更好地实时返回给前端”
- 第六章会继续讨论可靠性、成本和安全
- 最终项目章会把这些能力整合起来
