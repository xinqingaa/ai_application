# 03. Prompt Engineering

> 本节目标：把 Prompt 从“随手写一句话”升级成“可设计、可调试、可评估、可复用”的任务接口，掌握稳定 Prompt 的结构、Few-shot 设计、任务拆解与模板工程化方法

---

## 1. 概述

### 学习目标

- 理解 Prompt 在应用开发里不是聊天措辞，而是任务接口
- 理解角色、任务、上下文、约束、输出格式、负向限制的作用边界
- 能识别常见 Prompt 失败模式，而不是只会说“模型不稳定”
- 掌握 Zero-shot、One-shot、Few-shot 的适用场景与设计方法
- 学会把复杂任务拆成多个稳定步骤，而不是不断堆长 Prompt
- 学会把 Prompt 做成模板文件、变量输入和可校验的工程资产
- 建立一套最小可行的 Prompt 调试与回归方法

### 预计学习时间

- Prompt 本质与结构：1 小时
- 失败模式与调试方法：1-1.5 小时
- Few-shot 设计：1 小时
- 任务拆解：1 小时
- 模板工程化与实践：1-2 小时

### 本节在 AI 应用中的重要性

| 场景 | 相关知识 |
|------|---------|
| 聊天产品回答质量 | 角色、任务、上下文、约束 |
| 分类与提取任务 | Few-shot、标签对齐、格式固定 |
| 客服 / 审核 / 运营工具 | 任务拆解、中间结果显式化 |
| Prompt 复用与协作 | 模板文件、变量渲染、版本管理 |
| 效果迭代 | 小样本评估、对照实验、回归检查 |

> **Prompt Engineering 的核心不是“写出更像 AI 的句子”，而是让模型在不确定输出空间里，更稳定地完成你定义的任务。**

## 2. Prompt 不是聊天话术，而是任务接口 📌

### 2.1 初学者最常见的误解

很多人把 Prompt 理解成：

> 和模型“说一句话”

这在随便试玩聊天产品时没有问题，但一旦进入应用开发，就会立刻暴露两个问题：

1. 同一个需求，不同人写 Prompt，输出差异巨大
2. Prompt 一旦进入代码，就没人知道它到底在要求什么

因为在工程里，Prompt 不只是“人对模型说的话”，而是：

> 应用把任务说明交给模型执行的一份运行时接口协议

这个接口协议至少要回答 5 个问题：

1. 你是什么角色
2. 你要做什么任务
3. 你有哪些输入上下文
4. 你必须遵守哪些限制
5. 你应该按什么格式返回结果

如果这些问题没有被说清楚，模型就会自己补全。

而一旦模型开始自己补全，你的输出就不再稳定。

### 2.2 从“聊天内容”切换到“任务说明书”的视角

看一个很常见的 Prompt：

```text
帮我整理一下这个需求
```

它有什么问题？

- 不知道整理给谁看
- 不知道整理成什么形式
- 不知道应该保留什么、忽略什么
- 不知道是否允许补充常识
- 不知道长度要求

这类 Prompt 的问题不在于“写得不礼貌”，而在于：

> 任务边界不存在。

模型拿到这种 Prompt 后，只能靠自己的默认倾向去猜：

- 也许它会写成产品文档
- 也许它会写成技术方案
- 也许它会写成长文总结
- 也许它会顺手补商业分析

这就是为什么很多人觉得模型“有时候很聪明，有时候很离谱”。

本质不是模型今天心情不同，而是你给它的任务接口不稳定。

### 2.3 一条 Prompt 至少包含哪些层

在应用开发里，一个稳定 Prompt 往往由这些层组成：

| 层 | 作用 |
|----|------|
| 角色 | 限制回答身份、风格、视角 |
| 任务 | 告诉模型这次到底要完成什么 |
| 上下文 | 给出原始材料、输入、背景 |
| 约束 | 限制边界、长度、语气、禁止事项 |
| 输出格式 | 让结果便于检查、展示、解析 |

你可以把它理解成前端组件的 `props interface`：

- 没有类型定义，就容易乱传
- 没有必填字段，就容易出现隐式默认值
- 没有输出约束，就容易让下游逻辑崩掉

Prompt 也是同样的道理。

### 2.4 Prompt 和 `messages` 的关系

在第一章我们已经看到，聊天模型真正接收的是 `messages`：

```python
messages = [
    {"role": "system", "content": "你是一个严谨的助手"},
    {"role": "user", "content": "请根据下面需求整理前端说明..."}
]
```

所以“Prompt”在工程里并不一定只是一个字符串。

更准确地说，它通常落在两层：

1. **消息层 Prompt**
   - `system`
   - `user`
   - 有时还会包含 few-shot 示例消息

2. **模板层 Prompt**
   - 可复用的文本模板
   - 动态变量
   - 不同任务版本

你真正需要设计的是：

> 这次任务的信息应该如何分布在 `system`、`user`、模板变量和示例里。

### 2.5 为什么 Prompt 不是越长越好

很多人意识到 Prompt 要更清楚之后，会进入另一个极端：

> 把所有想到的约束全都塞进去。

这也会出问题。

因为 Prompt 太长时，可能带来：

1. 成本上升
2. 噪声变多
3. 主次信息不清
4. 不同要求之间互相冲突
5. 维护困难

所以 Prompt 工程的目标不是：

> 写得越多越保险

而是：

> 用尽量清楚、尽量稳定、尽量少歧义的方式，把任务接口定义完整。

### 2.6 一个更接近工程形态的 Prompt

比如同样是整理需求，下面这个写法明显更接近“任务接口”：

```text
你是一名资深 AI 产品需求分析助手，主要服务对象是前端开发者。

任务：
请根据下面的原始需求，整理一份「前端实现说明」。

输入材料：
我们要做一个 AI 聊天页面，左侧有会话列表，中间支持多轮对话和流式输出，
还希望用户能看到本轮大概消耗多少 Token。

约束：
1. 只整理和前端实现直接相关的信息。
2. 如果原始需求没有提到，就明确标注“待确认”，不要自行编造。
3. 用简洁中文输出，控制在 350 字以内。

输出格式：
## 页面目标
- ...

## 核心功能
- ...

## 待确认问题
- ...
```

它的价值不是“显得更专业”，而是：

- 角色清楚
- 任务清楚
- 材料清楚
- 约束清楚
- 格式清楚

换句话说：

> 它降低了模型靠猜来补全任务的空间。

### 2.7 Prompt 工程真正要追求什么

在应用开发里，Prompt 设计首先追求的不是“惊艳”，而是下面这 4 件事：

1. **清楚**
   - 模型知道自己要做什么

2. **稳定**
   - 同类输入尽量输出同类结果

3. **可调试**
   - 出问题时知道该改哪一层

4. **可复用**
   - 不是每次都重新手写一遍

如果这 4 件事做不到，Prompt 很难成为工程资产。

## 3. 常见 Prompt 失败模式 📌

### 3.1 为什么必须先学失败模式

很多人学 Prompt Engineering 时，只看“好 Prompt 长什么样”，但这还不够。

因为真实工作里你更多面对的是：

- 结果不稳定
- 结果不完整
- 结果格式飘
- 结果看起来像对的，其实逻辑错了

如果没有失败模式意识，你只会反复做两件事：

1. 继续加更多话
2. 继续碰运气重跑

这两件事都不是工程方法。

### 3.2 失败模式一：任务过于模糊

示例：

```text
帮我写一下这个需求
```

问题：

- 写成文档还是方案？
- 给谁看？
- 要多长？
- 是否允许补充信息？
- 成功标准是什么？

这种问题的典型症状是：

- 输出内容看起来“也不是完全不对”
- 但每次侧重点都不一样
- 很难让结果固定下来

修复方式：

1. 明确目标读者
2. 明确任务产物
3. 明确输出格式
4. 明确边界

### 3.3 失败模式二：多个目标混在一起

示例：

```text
请整理下面需求，顺便给出技术方案、商业价值、竞品分析和页面文案，尽量详细但不要太长。
```

这个 Prompt 的问题不是信息少，而是目标互相打架：

- 需求整理和技术方案不是一回事
- 商业价值和前端说明不是一个读者
- “尽量详细”和“不要太长”互相冲突

这种 Prompt 的症状是：

- 模型会随机偏向某一个目标
- 结果看起来内容很多，但没有重点
- 调整一处后，另一处又坏掉

修复方式：

1. 拆任务
2. 排优先级
3. 只保留本轮必须完成的目标

### 3.4 失败模式三：缺少上下文

示例：

```text
把这个产品需求总结成前端开发说明
```

如果后面没有真实需求材料，模型只能用常识补。

这时看起来好像能输出很多内容，但问题在于：

> 它写的不是你的需求，而是它猜测的“一个大概相似的需求”。

这种问题在以下任务里最常见：

- 需求整理
- 用户反馈总结
- 工单分类
- 合同审核
- 医疗或法律信息整理

修复方式：

1. 给真实输入
2. 明确输入在哪里开始、在哪里结束
3. 缺信息时要求模型标记“待确认”，而不是自行补全

### 3.5 失败模式四：没有边界，模型开始自由发挥

示例：

```text
请根据用户反馈给出优化建议
```

如果没有边界，模型可能：

- 输出过长
- 提出超出业务阶段的建议
- 自行假设系统能力
- 顺手补一堆“最佳实践”

这种 Prompt 常见的表面现象是：

- 结果很像一篇顾问报告
- 但并不适合马上落地

修复方式：

1. 限制任务目标
2. 限制业务范围
3. 限制输出长度
4. 明确“不允许编造 / 不要扩展到未提及内容”

### 3.6 失败模式五：输出格式不固定

示例：

```text
请把下面评论做情感分类
```

第一次可能输出：

```text
正面
```

第二次可能输出：

```text
我认为这条评论偏正面，因为用户表达了满意。
```

第三次可能输出：

```json
{"label": "正面"}
```

如果你后面要接程序，这种波动会直接制造脆弱性。

修复方式：

1. 明确只允许哪些标签
2. 明确“只输出标签，不要解释”
3. 如果需要结构化，明确字段格式

### 3.7 失败模式六：示例和规则互相冲突

这在 Few-shot 里特别常见。

例如规则里写：

```text
只能输出：正面 / 负面 / 中性
```

但示例里却写成：

```text
输入：服务态度很差
输出：差评
```

模型会看到什么？

不是“你的一条规则写错了”，而是：

> 任务接口内部存在两个版本的标签体系。

这时模型会随机学其中一个。

修复方式：

1. 规则和示例标签保持完全一致
2. 示例格式保持完全一致
3. 更新规则时同步更新示例

### 3.8 失败模式七：把太多噪声塞进上下文

很多人知道上下文重要，于是开始往 Prompt 里塞：

- 旧版本需求
- 会议纪要
- 历史讨论
- 参考资料
- 长段示例
- 额外注意事项

最后真正的问题只占 10%，噪声占 90%。

这种情况下模型不一定“做不到”，但它更容易：

- 忽略关键要求
- 抓错重点
- 受到过时材料干扰

修复方式：

1. 只给当前任务必要的材料
2. 将原始材料与规则分区
3. 先做摘要，再做下游任务
4. 过于复杂时改为多步 Prompt

### 3.9 失败模式八：没有负向限制

很多 Prompt 只写“希望它做什么”，不写“不希望它做什么”。

例如：

```text
请根据下面工单写一段客服回复
```

如果没有负向限制，模型很可能：

- 乱承诺修复时间
- 自行判定问题已解决
- 直接替用户下结论
- 说出业务不允许的措辞

修复方式：

```text
不要承诺恢复时间，除非输入中已明确给出。
不要编造“问题已修复”“已经退款”等事实。
```

负向限制的作用不是让 Prompt 变苛刻，而是：

> 收缩模型自由发挥的范围。

### 3.10 失败模式九：缺少评判标准

示例：

```text
请优化这段文案
```

优化到什么程度算好？

- 更短？
- 更正式？
- 更适合社群？
- 更适合官网？

如果没有标准，模型即使改了，也不一定符合你的目标。

修复方式：

1. 指明面向对象
2. 指明语气
3. 指明保留和删除的重点
4. 指明成功标准

### 3.11 失败模式十：结果一错就继续“堆字”

这是最危险的习惯。

结果不好之后，很多人第一反应是：

> 再补两句说明试试。

但如果你不知道问题属于哪一层，继续堆字只会让问题更隐蔽。

正确思路应该是：

1. 先判断问题属于哪一类
2. 一次只改一个变量
3. 再看结果有没有改善

这才是 Prompt 调试，而不是 Prompt 碰运气。

### 3.12 一个最小排查顺序

当输出不稳定时，先按这个顺序排查：

1. 任务是否明确
2. 上下文是否足够
3. 约束是否清楚
4. 输出格式是否固定
5. 是否需要 Few-shot
6. 是否应该拆解任务
7. 是否需要改参数而不是改 Prompt

请注意最后一点。

有时候问题不是 Prompt 本身，而是：

- `temperature` 太高
- 上下文太长
- `max_tokens` 太小

所以 Prompt 调试不能脱离第一章讲过的调用参数。

## 4. 稳定 Prompt 的结构 📌

### 4.1 一个推荐的基础骨架

在大多数任务里，你可以优先从下面这个骨架开始：

```text
你是一名{role}。

任务：
{task}

输入材料：
{context}

约束：
1. ...
2. ...
3. ...

输出格式：
{format}
```

这不是唯一正确写法，但它有一个很大的工程价值：

> 不同任务可以共享同一个结构思路。

### 4.2 角色：不是为了“更像真人”，而是为了限制视角

角色最常见的误用是：

```text
你是全世界最厉害、最聪明、最专业的超级专家...
```

这种写法通常没有太大工程价值。

角色真正的作用是：

1. 限定视角
2. 限定语言风格
3. 限定关注重点

比如：

- `你是一名前端需求整理助手`
- `你是一名客服回复助手`
- `你是一名评论分类助手`

它们的作用不是“提升智商”，而是减少角色漂移。

### 4.3 任务：必须具体到“这一轮到底产出什么”

任务描述常见错误是写成：

```text
请帮我处理这个内容
```

“处理”不是任务，是占位词。

更好的任务描述应该能回答：

- 整理？
- 分类？
- 提取？
- 改写？
- 生成回复？

例如：

```text
请根据下面的原始需求，整理一份面向前端开发者的功能说明。
```

这个版本至少说明了：

1. 是“整理”
2. 不是“写方案”
3. 读者是前端开发者
4. 产物是功能说明

### 4.4 上下文：不要让模型用常识替代真实输入

上下文的核心不是“信息越多越好”，而是：

> 给当前任务必要且正确的输入材料。

例如：

```text
输入材料：
我们要做一个 AI 聊天页面，支持历史会话、新建会话、流式输出、本轮 Token 显示。
```

如果你没有给这段材料，模型只能按自己的经验写“一个它以为常见的聊天页面需求”。

### 4.5 约束：让任务边界变硬

约束用来回答“哪些事情能做，哪些不能做”。

常见约束包括：

- 长度限制
- 风格限制
- 范围限制
- 禁止补充
- 禁止解释
- 只输出某种格式

例如：

```text
1. 只整理和前端实现直接相关的信息。
2. 如果原始需求没有提到，就明确标注“待确认”，不要自行编造。
3. 控制在 350 字以内。
```

约束越清楚，任务边界越稳。

### 4.6 输出格式：让结果可阅读、可检查、可接程序

很多 Prompt 到这里就结束：

```text
请根据下面内容进行总结
```

但如果没有输出格式，你就很难保证结果稳定。

输出格式的常见写法有三类：

1. 自然语言分段
2. 列表 / 表格
3. JSON / 结构化对象

第三章先重点掌握前两类。

比如：

```text
输出格式：
## 页面目标
- ...

## 核心功能
- ...

## 待确认问题
- ...
```

这会让你在后续人工检查时非常高效。

### 4.7 负向限制：告诉模型“不要做什么”

这是很多初学者忽略，但工程里非常重要的一层。

例如：

```text
不要编造输入中没有出现的事实。
不要输出解释。
不要承诺恢复时间。
```

负向限制尤其适合以下任务：

- 客服回复
- 审核结论
- 结构化抽取
- 分类
- 风险判断

因为这些任务最怕的不是“不够精彩”，而是“错得很自信”。

### 4.8 成功标准：不是必须单独一节，但必须被隐含清楚

有些任务里，你可以直接写“成功标准”。

例如：

```text
如果输入信息不足，请输出“待确认”，而不是补充想象内容。
```

这就是一种成功标准。

Prompt 设计时要时刻问自己：

> 我怎么判断这个结果算成功？

如果你自己都答不上来，模型更不可能天然对齐。

### 4.9 信息顺序也会影响效果

常见推荐顺序通常是：

1. 角色
2. 任务
3. 输入
4. 约束
5. 输出格式

为什么这个顺序常见？

因为它符合“先定身份和目标，再给材料，再限制边界，最后规定交付格式”的阅读习惯。

当然不是所有任务都必须这样，但你要意识到：

> Prompt 不只是包含什么，还包含以什么顺序组织这些信息。

### 4.10 一个对比：坏 Prompt vs 稳定 Prompt

坏 Prompt：

```text
帮我整理一下这个需求
```

稳定 Prompt：

```text
你是一名前端需求整理助手。

任务：
请根据下面的原始需求，整理一份给前端开发者的功能说明。

输入材料：
用户希望在 AI 聊天页面中支持历史会话列表、多轮上下文、流式输出和 Token 展示。

约束：
1. 只整理与前端实现有关的信息。
2. 如果原始材料没提到，就写“待确认”。
3. 控制在 300 字以内。
4. 不要补商业分析。

输出格式：
## 页面目标
- ...

## 核心功能
- ...

## 待确认问题
- ...
```

两者差别不在“第二个更长”，而在：

- 第二个任务接口完整
- 第二个边界明确
- 第二个更容易复用
- 第二个更容易调试

### 4.11 为什么不要一上来就追求复杂 Prompt 技巧

很多人一学 Prompt，就会关注：

- 高级咒语
- 神秘提示技巧
- 某些固定模板

但真正有效的路径通常是：

1. 把基础结构写完整
2. 做 5-10 条样本测试
3. 观察稳定性
4. 再决定是否需要 Few-shot 或拆解

这比追求“高级神技”更有实际价值。

## 5. Few-shot：用示例对齐边界 📌

### 5.1 为什么要有 Few-shot

有些任务只靠文字规则，模型依然容易跑偏，尤其是：

- 分类
- 信息提取
- 语气改写
- 复杂格式转换

这时 Few-shot 的作用不是单纯“告诉模型怎么做”，而是：

> 通过具体样本把边界教给模型。

规则说的是抽象要求。

示例给的是具体形态。

两者结合，通常比单独使用其中一个更稳。

### 5.2 Zero-shot、One-shot、Few-shot 的区别

| 方式 | 含义 | 适合场景 |
|------|------|---------|
| Zero-shot | 不给示例，只给任务说明 | 简单问答、基础摘要、明确格式 |
| One-shot | 给 1 个示例 | 轻量格式对齐 |
| Few-shot | 给多个示例 | 分类、抽取、边界任务、格式一致性要求高的任务 |

不要把 Few-shot 理解成“必须使用”。

真正的问题应该是：

> 当前任务只靠规则能不能稳定？

如果能，就没必要为了“高级”而加入更多示例。

### 5.3 Few-shot 最适合解决什么问题

Few-shot 特别适合解决以下问题：

1. 标签边界不清
2. 输出格式老是飘
3. 某些中间态样本容易被错分
4. 不同模型对抽象规则理解不完全一致

例如情感分类里，“物流一般，但包装不错”到底是正面还是中性？

如果你不放示例，模型可能因为“不错”而偏正面。

但如果你给出中性示例，它就更容易理解：

> 同时有优点和缺点，不一定算正面。

### 5.4 一个常见的 Zero-shot 分类 Prompt

```text
请判断下面评论的情感倾向，只能输出：正面 / 负面 / 中性

评论：物流速度一般，但包装不错。
输出：
```

它已经比“判断一下这条评论”好很多，因为至少：

- 标签集合明确
- 输出约束明确

但它仍然没有告诉模型：

- 什么情况下算中性
- 同时有优点和缺点怎么处理
- 只输出标签时是否允许带解释

### 5.5 一个不合格的 Few-shot 是怎样的

比如：

```text
请做评论分类。

示例1：
输入：这个产品体验绝了。
输出：正面

示例2：
输入：服务态度很差。
输出：差评

示例3：
输入：功能比较多，先继续观察。
输出：一般
```

问题非常明显：

1. 标签不一致
2. 没有限定输出集合
3. 示例风格不统一
4. 没有说明分类规则

这种 Few-shot 有时甚至比 Zero-shot 更差。

原因是：

> 它把一个原本清楚的任务，变成了一个边界混乱的任务。

### 5.6 一个更合理的 Few-shot 结构

```text
你是一名评论情感分类助手。

任务：
判断评论的情感倾向，只能输出以下三个标签之一：
- 正面
- 负面
- 中性

分类规则：
1. 明显表达满意、称赞、推荐，输出 正面。
2. 明显表达抱怨、不满、故障、差评，输出 负面。
3. 既有优点也有缺点、或整体评价平淡，输出 中性。
4. 不要输出解释，不要输出其他标签。

示例：
输入：客服响应很快，问题当天就解决了。
输出：正面

输入：登录连续失败三次，页面还一直报错。
输出：负面

输入：物流速度一般，但包装比较完整。
输出：中性

现在请判断：
输入：价格不算低，不过质量还可以接受。
输出：
```

这个版本的改进点是：

- 先定义规则
- 再给示例
- 标签一致
- 覆盖边界样本
- 输出限制明确

### 5.7 Few-shot 示例设计的 6 个原则

#### 原则 1：格式保持绝对一致

例如都写成：

```text
输入：...
输出：...
```

不要一会儿写“评论：”，一会儿写“输入：”。

格式不一致会增加模型的解释成本。

#### 原则 2：标签体系绝对一致

如果允许的标签是：

- 正面
- 负面
- 中性

那示例里也必须只出现这三个词。

不要出现：

- 差评
- 好评
- 一般

#### 原则 3：覆盖典型样本和边界样本

如果你只给最明显的正面和负面，模型面对模糊样本时仍然容易飘。

所以通常要至少覆盖：

1. 明确正面
2. 明确负面
3. 中性或混合倾向

#### 原则 4：示例尽量贴近真实输入

如果真实评论都比较口语化，而你的示例像教科书，就会降低迁移效果。

Few-shot 的本质是“用相似案例教模型”。

相似度越高，通常越有效。

#### 原则 5：示例不是越多越好

示例越多：

- Token 成本越高
- 噪声越多
- 维护越麻烦

很多时候 3-5 个高质量示例，比 20 个松散示例更有效。

#### 原则 6：示例也要版本管理

你加一个新示例，可能会改善某类错误，也可能会把另一类样本带偏。

所以 Few-shot 不应该靠记忆维护，而应该像代码一样可比较、可回滚。

### 5.8 Few-shot 的工程代价

Few-shot 虽然好用，但不要忽略它的成本：

1. 上下文更长
2. 请求更贵
3. 维护更复杂
4. 不同模型上的迁移效果不一定相同

这意味着在真实项目里，要始终问自己：

> 当前收益，是否值得这部分上下文成本？

### 5.9 为什么 Few-shot 不能只看单条效果

很多人调 Prompt 的方式是：

1. 拿一条样例
2. 觉得结果不错
3. 宣布“这个 Prompt 可以了”

这非常危险。

因为 Prompt 最怕的不是某一条不行，而是：

> 在一组相似但不完全相同的样本上表现不稳定。

所以最小做法应该是：

1. 准备 5-10 条样本
2. 标好预期结果
3. 每次改 Prompt 后跑一遍

这就是最小可行评估集。

### 5.10 第三章的 Few-shot 实践怎么设计

本章代码里准备了一个小型评论数据集：

- `source/02_llm/03_prompt_engineering/data/reviews.json`

对应脚本：

- `source/02_llm/03_prompt_engineering/02_few_shot.py`

脚本做了三组对比：

1. Zero-shot
2. 坏 Few-shot
3. 好 Few-shot

你需要重点观察的不是“好 Few-shot 一定全对”，而是：

1. 为什么坏 Few-shot 会制造额外错误
2. 为什么好 Few-shot 能让边界更清楚
3. 为什么评估集比单条样本更重要

### 5.11 一个很实用的工作流

在真实项目里，你可以按这个顺序做：

1. 先写 Zero-shot 版本
2. 评估 5-10 条样本
3. 记录错在哪些类型
4. 再补 Few-shot 示例
5. 再次评估
6. 对比提升是否值得额外成本

这比“一上来就写一大堆示例”更合理。

### 5.12 什么时候不应该优先用 Few-shot

以下情况通常不建议先上 Few-shot：

1. 基础任务定义还没清楚
2. 输出格式问题其实可以靠明确格式修复
3. 问题本质是上下文缺失
4. 任务过于复杂，本来就应该拆解

换句话说，Few-shot 不是万能胶。

它适合修“边界理解”问题，不适合掩盖“任务定义不清”问题。

## 6. 复杂任务要拆成步骤，而不是继续堆 Prompt 📌

### 6.1 一个危险信号：你开始不断补说明文字

如果你发现自己在做这样的事：

- 再补一个限制
- 再补一个例外
- 再补一个说明
- 再补一个输出要求

而结果仍然不稳定，就要警惕：

> 这可能不是 Prompt 太短，而是任务本来就不该一步完成。

### 6.2 哪些任务特别适合拆解

以下类型特别适合拆解：

1. 抽取 + 判断 + 生成
2. 阅读长材料 + 输出结构化结果 + 再写解释
3. 先总结，再基于总结做决策
4. 先分析，再生成回复

例如工单场景：

```text
请阅读下面工单，判断问题类型和紧急程度，并给出发给用户的回复。
```

这里至少包含：

1. 读取事实
2. 识别问题类型
3. 判断紧急程度
4. 写客服回复

如果一步做完，你很难知道错误发生在哪个环节。

### 6.3 拆解的核心收益是什么

拆解最重要的收益不是“更高级”，而是：

1. 每一步任务更单一
2. 每一步更容易定义输入和输出
3. 更容易定位错误
4. 更容易人工审阅中间结果
5. 更容易复用某一步能力

### 6.4 一个典型拆解模式：先分析，再生成

比如客服工单，可以拆成两步：

#### 第一步：分析

```text
请根据工单提取：
- 问题类型
- 影响范围
- 紧急程度
- 已知事实
- 待确认信息
```

#### 第二步：回复

```text
请基于上一步的分析结果，写一段发给用户的回复。
```

这样做的好处是：

- 如果问题类型判断错了，你只改分析 Prompt
- 如果回复语气不对，你只改回复 Prompt

这比一条巨大 Prompt 好维护得多。

### 6.5 拆解不等于暴露思维链

这里需要特别说明。

我们这章关注的是：

> 如何让模型按步骤完成任务

不是要求模型把所有内部推理过程都长篇展开。

工程里更常见、更有价值的是：

- 显式中间结构
- 中间字段
- 中间摘要
- 中间判断结果

这和“把思考过程全写出来”不是一回事。

### 6.6 为什么中间结果要可检查

如果你拆成两步，第一步的输出必须满足：

1. 人能看懂
2. 程序能用
3. 便于判断是否正确

例如：

```text
问题类型：支付
影响范围：单个用户
紧急程度：中
已知事实：
- 用户完成付款
- 订单状态未更新
待确认信息：
- 是否有重复支付
```

这种中间结果就很有价值，因为它：

- 结构稳定
- 便于 review
- 便于下游继续使用

### 6.7 上下文应该放在哪一步

任务拆解后，每一步都应该只拿必要上下文。

常见错误是：

第一步拿全量资料，第二步也继续拿全量资料，结果模型根本不需要依赖第一步的中间结果。

这会导致：

- 拆了解构，没拆语义
- 成本更高
- 错误仍然难定位

正确思路是：

1. 第一步拿原始材料
2. 第二步优先拿第一步结果
3. 只补必要原始信息

### 6.8 常见拆解模式

#### 模式 1：抽取 -> 分类

适合：

- 用户反馈
- 审核文本
- 工单处理

#### 模式 2：总结 -> 决策

适合：

- 长文档阅读
- 会议纪要整理
- 多段用户访谈总结

#### 模式 3：分析 -> 回复

适合：

- 客服
- 销售跟进
- 审核解释

#### 模式 4：草稿 -> 自检 -> 定稿

适合：

- 写作改写
- 风格统一
- 格式校对

第三章先重点掌握前 3 类。

### 6.9 什么时候不必拆解

如果任务本身已经很单一，就不要为了“显得专业”硬拆。

例如：

- 简单翻译
- 短摘要
- 明确标签分类

这些任务通常先写好单步 Prompt 就够了。

拆解是为了解决复杂性，不是为了增加形式感。

### 6.10 对应代码实践

本章代码提供了一个工单示例：

- `source/02_llm/03_prompt_engineering/data/support_tickets.json`
- `source/02_llm/03_prompt_engineering/03_task_decomposition.py`

脚本会对比：

1. 单步 Prompt
2. 分析 Prompt
3. 回复 Prompt

你应该重点观察：

1. 单步 Prompt 是否把所有事混在一起
2. 拆解后是否更容易看出错在哪一步
3. 第二步是否更像“受控生成”而不是自由发挥

### 6.11 一个工程判断标准

如果你发现某个 Prompt 的问题总是落在不同层面：

- 有时分类错
- 有时格式错
- 有时回复语气错

那它通常值得拆解。

因为这说明：

> 当前一个步骤承载了太多目标。

## 7. Prompt 模板工程化 📌

### 7.1 为什么 Prompt 不能长期散落在代码字符串里

在最早期试玩阶段，你可以直接这样写：

```python
prompt = "请帮我总结下面的内容..."
```

但一旦进入真实项目，这种方式会很快带来问题：

1. 无法管理版本
2. 不同人偷偷改 Prompt
3. 无法做 A/B 对比
4. 变量遗漏不容易发现
5. 很难审查当前线上到底在用什么

所以 Prompt 迟早要从：

> 代码里的临时字符串

升级到：

> 可复用、可审查、可替换、可校验的模板资产

### 7.2 模板化到底解决什么问题

模板化至少解决 4 个问题：

1. **复用**
   - 同类任务不重复手写

2. **分离**
   - 把任务说明和业务代码分开

3. **校验**
   - 可以检查变量缺失和多传

4. **版本化**
   - 可以比较不同 Prompt 版本

### 7.3 一个基础模板长什么样

例如一个需求整理模板：

```text
你是一名${role}。

任务：
${task}

原始材料：
${source_text}

约束：
1. 只整理和当前任务直接相关的信息。
2. 如果原始材料没有提到，就明确写“待确认”。
3. 控制在 ${max_length} 字以内。

输出格式：
## 目标
- ...

## 核心点
- ...

## 待确认
- ...
```

这里的工程价值在于：

- 角色可变
- 任务可变
- 原始材料可变
- 长度可变

但结构保持稳定。

### 7.4 为什么模板变量也要设计规范

变量名不是随便取的。

不好的变量名：

- `${text}`
- `${data}`
- `${content}`

它们太模糊。

更好的变量名：

- `${source_text}`
- `${ticket_title}`
- `${analysis_result}`
- `${max_length}`

变量名越清楚，模板越容易维护。

### 7.5 模板渲染前为什么必须做校验

看两个常见错误：

#### 错误 1：漏传变量

模板里有：

```text
${analysis_result}
```

但代码没传。

这会导致：

- 模型看到未替换变量
- Prompt 语义残缺

#### 错误 2：多传变量

你以为自己传了关键变量，实际上模板根本没用。

这会导致：

- 代码和模板理解不一致
- 调试时浪费时间

所以模板系统至少要能检查：

1. 模板声明了哪些变量
2. 当前调用少了哪些变量
3. 当前调用多了哪些变量

### 7.6 模板文件应该如何组织

本章给的一个最小组织方式是：

```text
prompts/
├── requirement_summary_v1.txt
├── requirement_summary_v2.txt
├── sentiment_zero_shot.txt
├── sentiment_few_shot.txt
├── support_analysis.txt
└── support_reply.txt
```

你可以看出它背后的组织思路：

1. 按任务分类
2. 重要模板带版本号
3. Few-shot 和 Zero-shot 分开
4. 上游分析模板和下游回复模板分开

### 7.7 为什么推荐把 Prompt 预览导出

很多团队调 Prompt 时，只看最终模型输出，不保留实际渲染后的 Prompt。

这会丢失非常重要的信息：

- 当时到底传了什么变量
- 模板长什么样
- 线上版本和本地版本有什么差异

所以一个很实用的做法是：

> 把渲染后的 Prompt 预览导出成 JSON 文件

这样你可以：

1. 做代码 review
2. 做效果对照
3. 做回归排查

### 7.8 Prompt 版本管理应该关注什么

不是每个 Prompt 都要复杂版本系统，但至少要做到：

1. 重要模板有版本号
2. 修改原因可追踪
3. 修改后能回归测试

例如：

- `requirement_summary_v1.txt`
- `requirement_summary_v2.txt`

这就比只有一个 `summary.txt` 强很多，因为你能比较：

- v2 比 v1 多了什么
- 为什么要升级
- 升级后真的更好吗

### 7.9 模板工程化不是脱离业务的“纯文本管理”

模板化不是为了把 Prompt 丢进一个“神秘模板库”就结束。

它必须和业务流程绑定：

- 谁来传变量
- 什么时候渲染
- 哪一步做校验
- 哪一步导出预览
- 哪一步发起模型调用

所以最小系统通常会包含：

1. 模板文件
2. 模板加载器
3. 变量校验
4. 渲染
5. 预览导出
6. 模型调用

### 7.10 第三章对应代码实践

对应脚本：

- `source/02_llm/03_prompt_engineering/04_prompt_templates.py`

对应工具：

- `source/02_llm/03_prompt_engineering/prompt_utils.py`

对应模板目录：

- `source/02_llm/03_prompt_engineering/prompts/`

你会在代码里看到：

1. 模板列表
2. 模板变量描述
3. 渲染结果
4. 缺失变量 / 多余变量检查
5. 导出模板预览 JSON

### 7.11 一个重要判断：何时该从“手写 Prompt”升级为模板

通常出现以下情况就应该升级：

1. Prompt 已经被多个地方复用
2. Prompt 里有明显变量
3. Prompt 需要多版本对比
4. Prompt 需要协作修改
5. Prompt 需要走 review 流程

如果还停留在 Python 字符串硬编码，后面几乎一定会返工。

## 8. 一套实用的 Prompt 调试方法 📌

### 8.1 为什么很多 Prompt 调优越调越乱

常见错误流程是：

1. 结果不满意
2. 同时改角色、任务、格式、示例
3. 再跑一次
4. 结果变好了，但不知道为什么

这不是调试，是碰运气。

### 8.2 正确的调试顺序

更实用的顺序通常是：

1. 定义任务成功标准
2. 准备小样本评估集
3. 写最小可行 Prompt
4. 跑评估
5. 只改一个变量
6. 再跑评估
7. 记录变化

这和写代码调 bug 很像。

### 8.3 第一步：定义“什么叫成功”

例如情感分类任务，“成功”至少意味着：

1. 标签在允许集合内
2. 大多数样本分类正确
3. 不输出多余解释

如果你不先定义成功，就没法判断 Prompt 是否真的变好了。

### 8.4 第二步：准备 5-10 条小样本

这个数量不需要很大，但必须覆盖：

1. 典型样本
2. 边界样本
3. 容易出错样本

它的目的不是做学术评测，而是做工程回归。

### 8.5 第三步：先从最小可行版本开始

不要一开始就把所有技巧全上。

先写一个尽量简单但结构完整的版本：

- 角色
- 任务
- 上下文
- 约束
- 输出格式

看看它在哪些样本上失效。

### 8.6 第四步：问题归因

如果结果不好，先问：

1. 是任务定义不清？
2. 是边界样本理解不清？
3. 是格式不稳定？
4. 是需要 Few-shot？
5. 是需要拆解？

你必须知道自己为什么改，不要靠直觉乱试。

### 8.7 第五步：一次只改一个变量

例如：

#### 版本 A

只补输出格式

#### 版本 B

在 A 基础上补负向限制

#### 版本 C

在 B 基础上补 Few-shot

这样你才能知道：

- 是格式起了作用
- 还是 Few-shot 起了作用

### 8.8 第六步：记录 Prompt 变更

至少记录三件事：

1. 改了什么
2. 为什么改
3. 效果是否变好

没有这层记录，几轮之后你就会忘记：

- 哪个版本为什么存在
- 哪个示例是后来加的
- 哪个负向限制其实没用

### 8.9 第七步：观察成本，不只观察效果

Prompt 变好不代表就一定该上线。

你还要看：

1. Token 是否明显变长
2. 延迟是否更高
3. 维护成本是否变复杂
4. 跨模型迁移是否变差

工程里永远是效果、成本、稳定性的平衡。

### 8.10 一个推荐的最小记录表

你可以自己维护一个简单表格：

| 版本 | 改动 | 预期改善 | 实际结果 | 备注 |
|------|------|----------|---------|------|
| v1 | 基础结构化 Prompt | 固定输出格式 | 格式稳定，但边界样本错 | 考虑补 Few-shot |
| v2 | 增加 3 个 Few-shot 示例 | 改善中性样本 | 中性准确率提升 | Token 增加 |
| v3 | 拆成两步 | 减少混合任务错误 | 可定位性更好 | 调用次数变多 |

这张表非常朴素，但很有用。

### 8.11 什么时候该停止调 Prompt，转向别的方法

如果你已经：

1. 任务定义清楚
2. 格式也固定
3. Few-shot 也加了
4. 任务也拆了

但结果仍然不稳定，那就要考虑：

- 是不是需要结构化输出能力
- 是不是需要外部知识检索
- 是不是需要 tool calling
- 是不是模型本身不适合这个任务

Prompt 不是万能解法。

Prompt 工程的成熟表现之一，就是知道：

> 哪些问题继续调 Prompt 也不会本质改善。

5. 高频 Prompt 必须模板化

如果只记住一句话，那就是：

> Prompt 不是一段临时聊天文本，而是一种要被设计、调试、评估和管理的工程接口。

## 10. 小结

这一章我们真正建立的是一套 Prompt 工程心智，而不是若干“写提示词技巧”。

你要记住的主线有 5 条：

1. Prompt 的本质是任务接口，不是聊天措辞
2. 稳定 Prompt 至少要有角色、任务、上下文、约束和输出格式
3. Few-shot 用来对齐边界，不是为了显得高级
4. 复杂任务不该一直堆长 Prompt，而应该拆成多个步骤
5. Prompt 一旦开始复用，就必须模板化、版本化、可校验

这也是为什么第三章和后续章节关系非常紧密：

- 第四章的结构化输出，需要清楚的输出格式意识
- 第五章的流式输出，需要稳定的任务定义
- 后面的 RAG、Agent、Workflow，本质上都离不开 Prompt 设计

Prompt Engineering 从来不是一个独立小游戏，而是整个 AI 应用工程的入口能力之一。

### 练习 1：把一个坏 Prompt 改成工程化版本

把下面这句：

```text
帮我写一下这个需求
```

改造成一个包含：

的版本。

- 工单类型分类
- 用户反馈优先级分类

要求：

1. 定义标签集合
2. 写 3 个 Few-shot 示例
3. 准备 5 条评估样本

### 练习 3：把复杂任务拆成两步

任选一个任务：

- 用户反馈总结并给出建议
- 工单分析并生成客服回复
- 需求整理并输出待确认问题

要求：

1. 写出单步 Prompt
2. 再写出拆解后的两步 Prompt
3. 比较哪种方式更好定位错误

### 练习 4：做一个最小 Prompt 模板库

在 `prompts/` 目录下新增至少 3 个模板，并满足：

1. 有明确命名
2. 有变量
3. 能做变量校验
4. 能导出渲染结果

### 练习 5：记录一次 Prompt 迭代

选择一个任务，至少做 3 个版本：

- v1：基础结构化
- v2：补 Few-shot 或补负向限制

记录：

1. 每个版本改了什么
2. 为什么改
3. 结果有没有改善
4. 成本有没有变化

如果你认真做完这 5 个练习，第三章就不只是“读懂了概念”，而是已经开始具备真正的 Prompt 工程能力。
