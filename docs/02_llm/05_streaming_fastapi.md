## 2. 为什么流式输出会显著改善体验 📌

### 2.1 同样等 5 秒，为什么用户感受完全不同

假设模型需要 5 秒才能生成完整回答。

有两种返回方式：

#### 方式 A：非流式

```text
用户发送问题
-> 前端一直 loading
-> 5 秒后整段答案一次出现
```

#### 方式 B：流式

```text
用户发送问题
-> 0.8 秒后开始看到前几个字
-> 内容继续一点点往下长
-> 5 秒后完整结束
```

从总耗时看，两者可能都差不多。

但从用户体验看，差异非常大。

因为用户最在意的通常不是：

> 从发请求到完整答案结束，一共多少秒

而是：

> 我多久能确认“系统已经开始工作了，而且马上会给我结果”

这就是流式输出的核心价值。

### 2.2 流式输出改善的不是“模型能力”，而是“交互感受”

流式不会让模型突然更聪明。

它改善的是：

1. **首字等待感**
2. **响应过程可见性**
3. **用户对系统卡死的焦虑**
4. **较长回答时的阅读启动时间**

也就是说，流式输出本质上是一种：

> 产品交互层优化

而不是模型能力层优化。

### 2.3 哪些场景特别适合流式

#### 场景 1：聊天助手

最典型的场景。

用户已经习惯看到 AI 一边“打字”一边输出内容。

#### 场景 2：长摘要 / 长解释

如果答案本来就比较长，不流式时用户会长时间看着空白。

#### 场景 3：代码生成 / 文案生成

用户通常不需要等整段生成完才开始阅读。

#### 场景 4：前端有逐步渲染需求

例如：

- token-by-token 展示
- 自动滚动
- 打字机效果
- 流式 Markdown 渲染

### 2.4 哪些场景不一定要流式

不是所有任务都应该默认流式。

以下场景通常可以先不流式：

#### 场景 1：很短的结构化输出

例如：

- 分类标签
- 简短 JSON 对象
- 是否通过审核

这类结果很短，流式收益不大。

#### 场景 2：强结构化返回

比如第四章那种结构化提取：

- 先得到完整 JSON
- 再一次性做验证

这类链路通常更关心完整性而不是中途可见性。

#### 场景 3：下游必须等完整结果才能继续

例如某些必须一次性校验的复杂链路。

### 2.5 流式输出真正影响哪些产品指标

从工程和产品视角，你可以关注这些指标：

1. **TTFT**
   - Time To First Token
   - 首字到达时间

2. **Total Latency**
   - 总完成时间

3. **用户主观等待感**
   - 即使总耗时一样，流式也会让等待更“有反馈”

4. **前端渲染复杂度**
   - 流式会让前端处理更复杂

5. **服务端实现复杂度**
   - 需要事件编码、连接管理、错误处理

所以流式输出不是没有成本，它是：

> 用更复杂的前后端链路，换更好的交互体验。

### 2.6 为什么第五章必须把“流式调用”和“服务接口”放在一起讲

很多人学流式时，只学这一句：

```python
stream=True
```

然后就结束了。

但真实项目里，`stream=True` 只是第一层。

你真正还需要解决：

1. chunk 怎么到达
2. 服务端怎么把 chunk 转成 SSE
3. 前端怎么消费 SSE
4. 完整结果怎么保存
5. 如果中途出错怎么办

所以第五章不能只讲 SDK，而必须把：

- 流式调用
- SSE
- FastAPI 路由

放到一条完整链路里。

### 2.7 一个很关键的产品认知

用户不一定会逐字读完每个 token，但他们会非常敏感地感受到：

- 有没有“开始”
- 中间有没有“持续进展”
- 过程有没有“卡住”

这也是为什么流式输出经常是 AI 聊天产品的标配。

## 3. OpenAI 风格流式调用到底发生了什么 📌

### 3.1 普通聊天调用和流式调用的差别

第一章我们看到的普通调用大致是：

```python
response = client.chat.completions.create(
    model="qwen-plus",
    messages=messages,
    temperature=0.3,
    max_tokens=400,
)
```

它的特征是：

- SDK 在内部等待完整结果
- 你最后一次性拿到 `response`

而流式调用通常是：

```python
stream = await client.chat.completions.create(
    model="qwen-plus",
    messages=messages,
    temperature=0.3,
    max_tokens=400,
    stream=True,
)

async for chunk in stream:
    ...
```

它的特征是：

- 返回的不是完整结果对象
- 而是一个可以逐段迭代的流

### 3.2 chunk 里最常见的内容是什么

在 OpenAI 风格流式响应里，最常见的读取方式是：

```python
delta = chunk.choices[0].delta.content or ""
```

这背后的意思是：

- `chunk` 是这一次流式到达的片段
- `delta` 是本次新增的内容
- `content` 可能为空，也可能包含若干字符

所以流式调用最核心的动作其实很朴素：

> 把每个新增的 `delta.content` 取出来，逐段拼成完整文本。

### 3.3 为什么流式里要区分 `delta` 和 `full_text`

因为这两者面向的用途不同：

#### `delta`

适合：

- 立即发给前端
- 立即打印到终端
- 立即更新聊天气泡

#### `full_text`

适合：

- 存数据库
- 写会话历史
- 做最终日志记录
- 做后续分析

也就是说：

> 流式链路里，你既需要“增量片段”，也需要“完整结果”。

### 3.4 为什么不能只把 `delta` 打印出来就结束

因为如果你只在循环里 `print(delta)`，没有额外收集，就会丢掉很多后续能力：

- 无法保存完整回答
- 无法给 `/sessions/{id}` 返回完整历史
- 无法统计输出 token 估算
- 无法在 done 事件里返回总耗时

所以真实工程里几乎都会做一件事：

```python
collected.append(delta)
```

最后：

```python
full_text = "".join(collected)
```

### 3.5 首字耗时和总耗时分别怎么理解

#### 首字耗时

是从发起请求到收到第一个非空 `delta` 的时间。

它回答的问题是：

> 用户多久能看到系统“开始输出了”

#### 总耗时

是从发起请求到完整流结束的时间。

它回答的问题是：

> 用户多久能拿到完整答案

这两个指标都重要，但对用户体验影响更直接的，往往是首字耗时。

### 3.6 为什么流式代码天然更偏向 async

流式输出本身就很适合异步。

因为在等待下一个 chunk 的过程中，程序可以：

- 继续处理别的请求
- 继续维持其他连接
- 继续做心跳事件

这也是为什么第五章和 `01_python` 的异步章节关系很紧密。

从工程角度看，流式输出通常会自然带你进入：

- `async def`
- `async for`
- `StreamingResponse`
- 异步生成器

### 3.7 流式调用里，usage 一定能拿到吗

不一定。

这是很现实的一点。

普通调用里很多平台会直接在完整响应对象里给你：

- prompt_tokens
- completion_tokens
- total_tokens

但流式调用时：

- 有的平台会在流末尾给 usage
- 有的平台不会
- 有的平台支持额外参数控制

所以课程里第五章的实践会优先用：

- 输入 token 估算
- 输出 token 估算

来建立对流式统计的基本认知。

### 3.8 一个最小流式调用的工程闭环

你应该把第五章第一个脚本理解成下面这条链路：

```text
构造 messages
-> 发起 stream=True 请求
-> async for chunk in stream
-> 提取 delta.content
-> 一边打印，一边收集
-> 最终拼接完整文本
-> 统计首字时间、总耗时、chunk 数
```

这条链路就是：

- `01_stream_basic.py`
- `streaming_utils.py`

的核心。

### 3.9 为什么 mock 模式在流式章节尤其重要

流式调用比普通调用更依赖“可观察过程”。

如果你没有真实 API Key，很多同学就会觉得：

> 那这一章我没法学了。

这其实不对。

因为流式章节最值得先建立的能力不是“某个平台返回了什么内容”，而是：

1. chunk 这个概念是什么
2. 增量片段和完整结果的关系是什么
3. SSE 事件如何组织
4. 会话状态如何更新

这些完全可以用 mock 先跑通。

### 3.10 第一个实践脚本做了什么

对应代码：

- `source/02_llm/05_streaming_fastapi/01_stream_basic.py`
- `source/02_llm/05_streaming_fastapi/streaming_utils.py`

它做了几件事：

1. 构造最小流式请求
2. 逐段打印输出
3. 统计首字耗时
4. 统计总耗时
5. 拼完整结果

这一步的目标不是把 FastAPI 一起上来，而是先建立：

> 流式输出到底长什么样、程序该如何收集它。

## 4. SSE：为什么流式聊天接口大多长这样 📌

### 4.1 什么是 SSE

SSE 的全称是：

> Server-Sent Events

它的特点是：

- 服务器持续向客户端单向推送事件
- 事件按文本协议逐段到达
- 非常适合“持续输出文本片段”的场景

对 AI 聊天来说，它和流式 token 输出天然契合。

### 4.2 为什么 SSE 特别适合 LLM 文本流式输出

因为 LLM 文本流式场景有这些特点：

1. 服务端往客户端单向输出
2. 输出是一段段增量文本
3. 不太需要客户端在同一连接里反向持续发送指令

这和 WebSocket 不一样。

WebSocket 更适合：

- 双向实时通信
- 聊天室
- 游戏同步

而 SSE 更适合：

- 服务器不断推送增量文本
- 客户端被动接收并渲染

### 4.3 一个 SSE 事件长什么样

最常见的 SSE 文本格式大概是：

```text
event: token
data: {"delta":"你好"}

event: token
data: {"delta":"，我是"}

event: done
data: {"elapsed_ms":1234}
```

规则很简单：

- 一条事件由若干行组成
- `event:` 指事件名
- `data:` 指事件内容
- 空行表示一条事件结束

### 4.4 为什么第五章要显式区分事件类型

很多简单 demo 只会不断返回文本，没有事件类型。

这在最早期可以跑，但工程上很快会碰到问题：

- 前端怎么知道这是开始、正文还是结束
- 怎么知道当前这条是不是心跳
- 怎么知道是不是报错

所以课程里建议至少定义这些事件：

1. `start`
2. `token`
3. `ping`
4. `done`
5. `error`

### 4.5 `start` 事件有什么用

它通常用来告诉前端：

- 当前流已经建立
- session_id 是什么
- 当前 provider / model 是什么

这对调试和 UI 初始化都很有帮助。

### 4.6 `token` 事件有什么用

这是核心正文事件。

前端最常做的事就是：

1. 读到 `token`
2. 取出 `delta`
3. 拼到当前消息气泡末尾

### 4.7 `ping` 事件为什么有必要

流式链路里，有时模型一段时间没有返回新 token。

如果这段时间服务端完全不发任何内容，可能会带来：

- 前端误以为连接断了
- 某些中间代理超时
- 用户以为系统卡死

所以很多流式接口会在没有新 token 时定期发一个心跳事件：

```text
event: ping
data: {"ts": 1710000000.123}
```

它的作用不是给用户看，而是：

> 维持连接活性，并让前端知道“服务端还活着”。

### 4.8 `done` 事件为什么不能省

如果没有 `done` 事件，前端很难明确知道：

- 完整回答是不是结束了
- 是否可以关闭 loading 状态
- 是否可以记录最终耗时
- 是否可以把会话正式写入本地状态

所以 `done` 通常应该携带：

- `elapsed_ms`
- `first_token_ms`
- `chunk_count`
- token 估算
- 最终 session token 估算

### 4.9 `error` 事件为什么要单独定义

如果中途出错，前端最好收到一条明确错误事件，而不是直接静默断开。

例如：

```text
event: error
data: {"message":"Upstream timeout"}
```

这样前端就可以：

- 显示错误提示
- 停止 loading
- 决定是否允许用户重试

### 4.10 为什么课程里用 `fetch + ReadableStream` 或 `curl -N` 思路来理解

很多人知道浏览器有 `EventSource`，但它有一个限制：

> 原生 `EventSource` 更适合 GET，不适合带 JSON body 的 POST。

而聊天接口常常需要 POST 请求体：

- `message`
- `session_id`
- `system_prompt`
- `temperature`

所以真实前端里，更常见的方式是：

1. 用 `fetch` 发 POST
2. 用 `ReadableStream` 逐段读 SSE

课程里为了降低复杂度，会先用：

- `curl -N`
- FastAPI `StreamingResponse`

把服务端链路讲清楚。

### 4.11 为什么 SSE 比“直接返回纯文本流”更适合工程化

如果你只返回纯文本流，前端虽然也能拼接，但会失去很多结构化语义：

- 无法区分正文和心跳
- 无法区分正文和错误
- 无法明确拿到 done 元信息

SSE 的真正价值就在于：

> 让流式文本不仅能“传”，还能“带语义地传”。

### 4.12 服务端要做的事其实很明确

服务端最终要做的事情可以总结成：

1. 从上游模型拿到流式 chunk
2. 翻译成你自己的事件协议
3. 用 SSE 编码发给前端
4. 在 done 时保存完整结果
5. 在空闲时发 ping
6. 在异常时发 error

这就是第五章第三个脚本的核心任务。

## 5. 为什么要先有普通 `/chat`，再做 `/chat/stream` 📌

### 5.1 很多流式 demo 的常见问题

一些演示项目一上来就直接做：

- `/chat/stream`
- 一路 token 往前端推

然后忽略了这些更基础的问题：

- 请求体结构是否稳定
- session_id 怎么管理
- 历史消息怎么保存
- provider 怎么切换
- 普通非流式接口怎么提供

这会导致后续排错很困难。

### 5.2 普通 `/chat` 接口解决什么问题

普通接口通常更适合先解决这些基础问题：

1. 请求体验证
2. 会话管理
3. `messages` 组织
4. provider 切换
5. 非流式元信息返回

也就是说，普通 `/chat` 是：

> 流式接口的稳定基础层

### 5.3 为什么普通接口在真实项目里仍然有价值

即使你最后主打流式，普通 `/chat` 也通常还会继续保留。

因为它适合：

- 测试和调试
- 后台管理工具
- 某些不需要流式的业务场景
- 回归检查

### 5.4 普通 `/chat` 接口的请求体应该长什么样

在第五章里，普通接口会用一个 Pydantic 请求模型承载：

- `session_id`
- `message`
- `provider`
- `system_prompt`
- `temperature`
- `max_tokens`
- `keep_last_messages`

这和前面章节的知识是连起来的：

- `messages` 本质还是第一章的消息结构
- `provider` 还是第二章的统一配置思想
- `system_prompt` 还是第三章的 Prompt 工程

### 5.5 为什么会话状态要在这章重新出现

流式聊天接口不是 stateless demo。

真实聊天里，服务端要面对的问题依然是：

1. 用户这一轮发了什么
2. 历史消息是什么
3. 当前 session 是否已存在
4. 回复完成后如何写回历史

所以第五章会重新引入一个最小会话存储：

- `InMemorySessionStore`

它的价值不是生产可用，而是帮助你理解：

> 流式接口也必须有“完整结果落回历史”的这一层。

### 5.6 为什么流式和非流式都应该复用同一套消息组织逻辑

如果你把：

- 普通接口的消息组织
- 流式接口的消息组织

写成两套完全不同的逻辑，后面很容易出现：

- 普通接口能复现，流式接口复现不了
- 普通接口带历史，流式接口忘了带
- system prompt 行为不一致

所以课程里第五章会把：

- `build_messages_for_turn()`
- `append_assistant_message()`
- `trim_messages_by_recent_messages()`

这类会话基础逻辑收进公共工具。

### 5.7 普通 `/chat` 接口返回什么比较合理

至少建议返回：

- `session_id`
- `provider`
- `model`
- `reply`
- `mocked`
- `elapsed_ms`
- `request_preview`
- `session_estimated_tokens`

这比只返回：

```json
{"reply": "..."}
```

更适合教学和调试。

因为你不仅要让接口“可用”，还要让学员看到：

- 请求是怎样组织的
- 会话估算 token 怎么变化
- provider 是否真的切换了

### 5.8 第 2 个实践脚本做了什么

对应代码：

- `source/02_llm/05_streaming_fastapi/02_fastapi_chat.py`

它的定位是：

1. 一个最小非流式聊天 API
2. 带请求体验证
3. 带会话状态
4. 带调试信息

这一步的目标不是“花哨”，而是先把普通链路稳定下来。

### 5.9 为什么这一步很像项目章的预演

因为你会发现，到了这里已经出现了很多项目型代码的要素：

- FastAPI app
- Pydantic 请求模型
- 会话状态
- LLM service 调用
- provider 切换

只是我们还没有把它们完全整合成最终项目结构。

## 6. FastAPI 流式接口：从上游 chunk 到 SSE 事件 📌

### 6.1 `StreamingResponse` 真正做了什么

FastAPI 里最常见的流式返回方式是：

```python
return StreamingResponse(generate(), media_type="text/event-stream")
```

这里的关键不是 `StreamingResponse` 这个名字，而是：

> 你给它一个“会不断产出字符串片段”的生成器。

对 SSE 来说，这些字符串片段就是一条条事件。

### 6.2 为什么生成器通常应该是异步的

因为流式聊天天然需要等待：

- 等上游模型返回 chunk
- 等心跳超时
- 等网络继续发送

所以很自然地会写成：

```python
async def build_sse_stream(...):
    yield encode_sse_event("start", ...)
    ...
```

异步生成器最大的好处是：

- 可以 `await`
- 可以 `yield`
- 非常适合边等待、边产出

### 6.3 直接在生成器里 `async for chunk in stream` 就够了吗

不完全够。

如果你只这样写：

```python
async for event in stream_chat_events(...):
    yield encode_sse_event(...)
```

会有一个问题：

> 当上游一段时间没有新 chunk 时，你没机会发 heartbeat。

因为生成器会一直卡在等待下一个上游 chunk。

### 6.4 为什么第五章要引入 `queue + producer` 的结构

为了解决“上游没新内容时，仍然能发 heartbeat”的问题，课程里会用一个很实用的结构：

1. 启一个后台 producer task
2. producer 负责读取上游模型流
3. 把 token / done / error 事件放进队列
4. 外层 SSE 生成器定时 `wait_for(queue.get(), timeout=heartbeat_seconds)`
5. 如果超时，就先发一个 `ping`

这个结构的工程意义很大：

- 上游模型流和下游 SSE 输出解耦
- 心跳逻辑更容易插入
- 错误也更容易统一收敛

### 6.5 这个结构为什么比“简单直接转发”更稳

因为它让服务端具备了三种输出能力：

1. **正常 token 输出**
2. **超时 heartbeat 输出**
3. **错误事件输出**

而不是只会“有 token 时输出，没有 token 时什么也不做”。

### 6.6 `done` 事件里为什么要补统计信息

当流式生成结束时，前端通常需要做这些事：

- 停止 loading
- 固定最终消息
- 更新会话状态
- 记录性能数据

所以 done 事件里建议带：

- `session_id`
- `mocked`
- `first_token_ms`
- `elapsed_ms`
- `chunk_count`
- `input_tokens_estimate`
- `output_tokens_estimate`
- `session_estimated_tokens`

这让前端在收到 done 时，可以一次性完成收尾动作。

### 6.7 为什么完整结果要在 done 时再写入历史

流式输出过程中，你虽然已经拿到了很多 token，但这时历史记录通常还不应该马上落库。

因为：

- 结果还没完成
- 可能中途报错
- 可能被用户取消

更合理的做法通常是：

1. 在内存里拼接 `assembled_parts`
2. 收到 done 后拿到 `full_text`
3. 再把完整 assistant 消息写入会话历史

这就是课程脚本里 `append_assistant_message()` 的使用时机。

### 6.8 心跳保活不是“可有可无的小优化”

在本地 demo 里，有时你会觉得：

> 模型已经不断出 token 了，好像不需要心跳。

但在真实环境里，心跳很重要，因为：

1. 上游可能中途停顿
2. 网络代理可能有空闲超时
3. 前端需要知道连接还活着

所以心跳是一种：

> 为长连接稳定性服务的协议层机制

### 6.9 错误事件应该怎么设计

最小设计通常够用：

```text
event: error
data: {"message": "..."}
```

如果你后面要更工程化，还可以加：

- `code`
- `provider`
- `session_id`
- `retryable`

但课程这一章先把最小形态讲清楚就够了。

### 6.10 第 3 个实践脚本做了什么

对应代码：

- `source/02_llm/05_streaming_fastapi/03_fastapi_stream.py`
- `source/02_llm/05_streaming_fastapi/streaming_utils.py`

它实现了：

1. 普通 `/chat`
2. 流式 `/chat/stream`
3. `start / token / ping / done / error` 事件
4. 最小会话状态保存
5. SSE 头设置

这已经是一个很像真实聊天后端雏形的结构。

### 6.11 为什么这比单纯的 SDK demo 有价值

因为真实产品里，前端根本不会直接连 OpenAI SDK。

它通常连的是：

> 你自己的后端服务

所以你必须有能力把：

- 上游 provider 流
- 下游前端可消费事件流

中间这层转换桥接起来。

这就是第五章真正的工程价值。

## 7. 前端如何理解这章的服务端输出 📌

### 7.1 为什么前端同学要特别理解 `token` 和 `done`

站在前端视角，这两个事件最关键：

#### `token`

代表：

- 当前消息继续增长

前端通常会：

```text
currentMessage += delta
```

#### `done`

代表：

- 当前消息正式结束
- 可以把 loading 改掉
- 可以更新统计信息

### 7.2 为什么不要让前端自己猜“是不是结束了”

如果前端只能通过“连接断开了”去猜结束，会有很多问题：

- 连接断开不一定是正常结束
- 出错断开和正常结束难区分
- 无法稳定拿到最后统计信息

所以 `done` 应该是显式事件，而不是隐式猜测。

### 7.3 为什么 `session_id` 应该在服务端生成并返回

对于前端来说，`session_id` 的作用至少有三层：

1. 复用对话上下文
2. 拉取历史记录
3. 将普通接口和流式接口统一到同一会话

课程里采用的策略是：

- 如果请求没带 `session_id`
- 服务端自动创建
- 再通过 `start` 或普通 `/chat` 返回给前端

### 7.4 为什么这章不直接写完整前端页面

因为第五章的主任务不是做 UI，而是先把：

- 流式服务端链路
- SSE 协议组织
- 会话状态逻辑

讲透。

如果服务端这一层没想清楚，前端 UI 再漂亮也只是包着一个不稳定的流。

### 7.5 一个典型前端消费思路

不展开完整代码，只讲思路：

```text
POST /chat/stream
-> 持续读取响应 body
-> 按 SSE 协议拆事件
-> token 事件就追加消息
-> ping 事件忽略或仅记录
-> done 事件做收尾
-> error 事件展示失败提示
```

这就是服务端为什么一定要把事件类型设计清楚。

### 7.6 第五章和最终项目章的关系

最终项目章里，如果要做一个真正的聊天应用，这一章会直接成为后端实现基础：

- `/chat`
- `/chat/stream`
- 会话状态
- provider 配置
- 首字耗时统计

所以第五章不是单独的小技巧，而是后续项目的骨架之一。

## 10. 小结

第五章真正建立的是：

> 把上游模型流，变成下游前端可消费服务流的能力。

你要记住的主线有 6 条：

1. 流式输出改善的是交互体验，不是模型能力本身
2. 首字耗时和总耗时是两个不同但都重要的指标
3. OpenAI 风格流式调用的核心是不断提取 `delta.content`
4. SSE 不是纯文本流，而是“带事件语义的文本流”
5. 流式接口应该建立在稳定的普通聊天接口之上
6. 服务端最终要负责：转发 token、发送心跳、在 done 时落完整结果

学到这里，你已经具备了很多 AI 聊天产品最关键的后端基础能力：

- 普通聊天 API
- 流式聊天 API
- SSE 事件协议
- 会话状态保存
- 基础性能统计

后续到项目章时，这些能力都会直接继续复用。
