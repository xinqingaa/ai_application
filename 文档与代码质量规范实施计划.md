## 2. 已确认的规范来源

本轮已经确认三类来源，各自职责不同。

### 2.1 规范种子

作用：

1. 提供 `01_python` 的高质量闭环模型。
2. 固化“一节一闭环”“先建立入口，再补练习”“显式映射，禁止靠脑补”等核心原则。

### 2.2 高质量样本

- `docs/01_python/*`
- `source/01_python/*`
- `docs/02_llm/*`
- `source/02_llm/*`

作用：

1. 抽取可跨阶段复用的栏目职责和代码组织方式。
2. 提供从“单节教学”过渡到“最小应用骨架”的真实样本。

### 2.3 复杂阶段补充来源

- `docs/03_foundation/README.md`
- `docs/03_foundation/06_foundation_lab_design.md`
- `docs/03_foundation/07_foundation_lab_tasks.md`

作用：

1. 说明为什么单靠 `01_python` 模型不足以覆盖复杂阶段。
2. 提供“原理文档 + 设计说明 + 任务拆解 + 独立项目”并行结构的样本。
3. 为 `04`、`05`、`06` 预留适配思路。

## 3. 已确认的核心决策

### 3.1 主规范独立成文

主规范文件为：

```text
文档与代码质量规范.md
```

它承担唯一真源职责。

### 3.2 README 只做入口

`README.md` 的职责应保持为：

1. 仓库导航
2. 学习主线说明
3. 主规范入口链接

### 3.3 CLAUDE 只做执行摘要

`CLAUDE.md` 只保留：

1. 修改文档和代码前应先遵循哪份规范
2. 最低执行要求
3. 特殊目录的阶段化说明
4. 冲突时以主规范为准

### 3.4 skill 放到最后

skill 的角色是执行规范，不是定义规范。

正确顺序仍然是：

1. 先稳定主规范正文
2. 再用真实阶段试点
3. 最后才考虑是否做 skill

## 5. 规范建设的当前边界

本轮已经完成主规范首版，但仍有几件事刻意没有做。

### 5.1 本轮没有做的事情

1. 没有直接修改仓库根 `README.md`
2. 没有直接修改 `CLAUDE.md`
3. 没有创建或安装 skill
4. 没有大规模回炉所有既有阶段文档
5. 没有直接重构 `03_foundation` 的全部代码和文档

### 5.2 为什么暂时不做

原因不是这些事情不重要，而是顺序上必须先让主规范稳定。

如果在规范尚未成型时就同步改入口、改协作文档、改所有阶段，很容易把“规范建设”重新变成“到处补丁式修改”。

## 6. 下一阶段工作重点

主规范首版完成后，后续工作不再是“继续起草大纲”，而是“验证规范是否真的能用”。

### Phase 1：用 `03_foundation` 做试点验证

目标：

- 检查主规范能否真正约束复杂阶段，而不是只适合 `01_python`。

重点检查：

1. 原理文档、设计说明、任务拆解、独立项目之间的映射是否充分
2. `foundation_lab` 的交付物和完成标准是否足够清晰
3. 是否仍存在必须靠口头解释才能成立的隐性规则

产出：

1. 试点问题清单
2. 需要回写主规范的修订项

### Phase 2：补根目录入口与执行摘要

目标：

- 让主规范进入日常工作流。

执行动作：

1. 在根 `README.md` 中增加主规范入口
2. 在 `CLAUDE.md` 中补执行摘要

产出：

1. README 规范入口块
2. CLAUDE 执行摘要块

### Phase 3：按主规范 review `04`、`05`、`06` 新内容

目标：

- 把主规范直接用于新阶段写作，而不是留作说明文。

执行动作：

1. 新增章节前先按主规范检查固定结构与标准产物
2. 新增项目前先按主规范检查 README、运行入口、配置说明和完成定义
3. 每完成一个阶段核心模块，就用评审清单做一次 review

产出：

1. 更稳定的一致性风格
2. 更少的返工
3. 更少的“边写边重新约束”的情况

## 10. 本文件的作用边界

本文件不是主规范正文，也不负责替代主规范。

它只承担三件事：

1. 记录当前已经完成到哪一步
2. 记录为什么这样设计
3. 给后续会话提供明确接续点

因此，未来即使主规范继续迭代，本文件也应保留，因为它记录了：

1. 规范是基于哪些样本抽出来的
2. 哪些决策已经明确
3. 哪些事项是刻意延后处理的
