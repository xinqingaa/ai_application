# 文档与代码质量规范实施计划

> 目的：把当前零散分布在各处的文档与代码约束，收口为一套统一、可复用、可评审、可持续演进的质量规范。
>
> 当前状态：本文件用于固化规范建设方案与后续工作计划，方便后续会话直接续做；本轮先不改 `README.md`、`CLAUDE.md`，也不先做 skill。

---

## 1. 当前问题

当前仓库已经形成了较清晰的学习主线：

```text
01_python -> 02_llm -> 03_foundation -> 04_rag -> 05_agent -> 06_application
```

但用于约束内容质量的规则仍然比较零散，主要问题如下：

1. 规范分散在多个位置，没有唯一真源。
2. `01_python` 的整体质量已经比较高，但其成功模式还没有被正式抽象成通用标准。
3. `03_foundation` 的文档编写和实施边界还在反复调整，说明缺少稳定的章节级和阶段级约束。
4. 文档、代码、实施方案之间虽然已有联系，但还没有形成统一的“完成定义”。
5. 未来如果继续推进 `04_rag`、`05_agent`、`06_application`，没有统一规范会导致每个阶段都重新摸索写法。

一句话概括：

- 现在缺的不是更多内容，而是一套能稳定复制 `01_python` 质量的规范体系。

---

## 2. 本次决策

### 2.1 主规范载体

主规范**不写进** `README.md`、`CLAUDE.md` 或 skill。

主规范应当是**独立文档**，作为唯一真源。

建议文件名：

```text
文档与代码质量规范.md
```

原因：

1. `README.md` 应该承担仓库导航和主线说明，不适合承载长篇规范。
2. `CLAUDE.md` 更适合承载协作摘要，不适合作为完整规范正文。
3. skill 更适合作为规范执行器，而不是规范正文载体。

### 2.2 README 的角色

`README.md` 未来只做两件事：

1. 告诉读者仓库整体结构和学习主线。
2. 链接到主规范文档，作为入口。

### 2.3 CLAUDE 的角色

`CLAUDE.md` 未来只做执行摘要，不复制主规范全文。

建议包含：

1. 修改文档前先看哪份规范。
2. 修改代码前最少需要满足哪些要求。
3. 哪些目录有阶段性特殊规则。
4. 若发生冲突，以主规范为准。

### 2.4 skill 的角色

skill 放到最后处理。

正确顺序：

1. 先形成主规范正文。
2. 用 `03_foundation` 做试点验证。
3. 规范稳定后，再抽取为 skill。

说明：

- 不是“直接写一个 `skill.md`”就能当规范用。
- skill 应该是未来给 Codex 用的“规范执行器”，不是项目内规范的真源。

## 3. 已有基础

当前仓库里已经有三类可复用基础：

### 3.1 已验证的高质量样本

- `docs/01_python/*`
- `source/01_python/*`

它们的价值在于：

1. 文档结构相对成熟。
2. 学习路径比较明确。
3. 文档与代码的映射关系较清晰。
4. 更接近“可执行学习路径”，而不是资料堆积。

### 3.2 已有的局部规范

- `学习材料改造规范.md`

这份文档已经把 `01_python` 的改造思路写得比较清楚，特别是：

1. 一节一闭环
2. 主示例 / 练习 / 参考答案分工
3. 文档结构固定化
4. 显式映射，禁止靠脑补

它是当前最重要的规范种子，但适用范围还偏窄，主要面向 `01_python`。

### 3.3 已有的阶段实施思路

- `docs/03_foundation/implementation.md`

它说明了一个关键问题：

- `03_foundation` 不只是写文档，还要联动独立实验项目、原理理解、最小工程骨架和阶段收口。

这意味着未来主规范不能只服务于 `01_python`，还要能覆盖更复杂的阶段。

## 4. 主规范要解决什么

未来的主规范应当统一回答以下问题：

1. 一节文档应该怎么写，固定栏目是什么。
2. 一节代码应该怎么配套，最少要有哪些文件。
3. 文档、示例代码、练习代码、参考答案之间如何映射。
4. 一章或一节什么时候算真正完成。
5. 哪些内容属于通用规范，哪些属于阶段特有规则。
6. 当阶段从 `01_python` 走到 `03_foundation`、`04_rag`、`05_agent` 时，规范如何扩展而不失控。

---

## 5. 主规范不解决什么

为了避免规范失控，以下内容不应写进主规范正文：

1. 某一章具体讲哪些知识点。
2. 某个案例或 demo 的具体业务内容。
3. 某种技术的唯一实现方式。
4. 时效性很强的工具或平台推荐。
5. 临时性的个人偏好和一次性决策。

一句话：

- 主规范管的是“结构、质量、闭环、映射、完成标准”，不是“具体业务内容本身”。

## 6. 未来主规范的建议结构

建议未来的 `文档与代码质量规范.md` 采用如下结构：

### 6.1 总则

1. 适用范围
2. 目标
3. 核心原则
4. 术语说明

### 6.2 文档质量标准

1. 文档固定结构
2. 每个栏目承担的职责
3. 白话解释要求
4. 代码示例粒度要求
5. 常见坑与边界提醒要求
6. 学习顺序显式化要求

### 6.3 代码质量标准

1. 每节标准产物
2. 主示例文件职责
3. 练习文件职责
4. 参考答案文件职责
5. 可运行性要求
6. 目录职责清晰要求

### 6.4 文档与代码映射规则

1. 文档必须对应哪些文件
2. 映射表如何写
3. 文档中如何描述运行路径
4. 如何避免学习者靠脑补建立映射

### 6.5 完成定义

1. 一节完成的标准
2. 一章完成的标准
3. 一个阶段完成的标准
4. 不符合完成标准的常见情形

### 6.6 分阶段适配规则

1. `01_python` 适配规则
2. `02_llm` 适配规则
3. `03_foundation` 适配规则
4. `04_rag` / `05_agent` / `06_application` 的预留扩展位

### 6.7 评审清单

1. 文档评审清单
2. 代码评审清单
3. 命名一致性检查
4. 阶段收口检查

### 6.8 例外处理规则

1. 什么情况下允许偏离模板
2. 偏离模板时需要补充哪些说明
3. 如何避免例外变成常态

## 7. 规范建设的实施阶段

本项工作建议分 6 个阶段推进，而不是一次性写完。

### Phase 1：规范来源收口

目标：

- 把仓库里现有规则和高质量模式整理清楚。

需要收集的来源：

1. `学习材料改造规范.md`
2. `docs/01_python/*`
3. `source/01_python/*`
4. `docs/03_foundation/implementation.md`
5. `docs/02_llm/*` 中已开始项目化的章节

产出：

1. 规范来源清单
2. 通用规则候选项
3. 阶段专属规则候选项

### Phase 2：抽取 `01_python` 的质量模型

目标：

- 明确 `01_python` 为什么质量高，并把它写成可以复用的模板。

需要回答的问题：

1. 一节文档为什么“好读”。
2. 为什么当前代码结构更容易学。
3. 文档和代码的映射是如何建立的。
4. “先看什么、先跑什么、先写什么”的顺序是如何明确的。

产出：

1. 章节文档模板
2. 示例代码模板
3. 练习与参考答案模板
4. 一节完成标准模板

### Phase 3：把规范扩展到 `02_llm`

目标：

- 让规范从“基础 Python 教学”扩展到“有工程化结构的 AI 应用教学”。

重点关注：

1. `.env.example`
2. 运行说明
3. CLI / API 双入口
4. 服务层抽象
5. 项目结构说明

产出：

1. `02_llm` 补充规范
2. 与 `01_python` 的共性和差异说明

### Phase 4：把规范扩展到 `03_foundation`

目标：

- 解决 `03_foundation` 这类“原理文档 + 独立实验项目 + 实施方案”并行的复杂场景。

重点关注：

1. 原理文档如何写边界
2. 实施方案如何与代码结构联动
3. 最小实验项目如何定义交付物
4. 如何判断“理解原理”和“完成代码”都达标

产出：

1. `03_foundation` 适配规范
2. 章节级与阶段级完成定义

### Phase 5：形成主规范首版

目标：

- 完成 `文档与代码质量规范.md` 的首版正文。

要求：

1. 内容完整但不过度冗长
2. 能指导后续章节直接使用
3. 能用于 review，而不只是留档

产出：

1. `文档与代码质量规范.md`

### Phase 6：挂载与试点

目标：

- 让规范真正进入工作流。

执行动作：

1. 在 `README.md` 里增加规范入口
2. 在 `CLAUDE.md` 里增加执行摘要
3. 用 `03_foundation` 做试点
4. 根据试点结果回修主规范

产出：

1. README 入口块
2. CLAUDE 摘要块
3. 第一轮试点反馈

## 10. skill 的后续方案

当主规范稳定后，可以再考虑做 skill。

未来 skill 的目标应是：

1. 新建章节时自动提醒固定结构
2. 改文档时自动检查是否有“本节目标 / 本节怎么学 / 代码映射表 / 完成标准”
3. 改代码时自动检查是否缺示例、练习、参考答案或运行说明
4. 做 review 时按评审清单检查缺口

也就是说：

- skill 负责“执行规范”
- 主规范文档负责“定义规范”
