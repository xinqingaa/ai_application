# Course Writing

本仓库课程正文（`course/**/*.md`）的写作执行 skill。原则与模板真源：[docs/learning-guide.md](../../docs/learning-guide.md) §6–§7。

## 触发

- 新建或重写 `course/**/0N_*.md`
- 用户说「写第 X 节」「加厚文档」「修订专题」

## 写前（强制）

1. 读 `docs/learning-guide.md` §6.3、§6.4、§6.6、§7
2. 读当课 `outline.md` 中该节条目
3. 读当节已落地的 `source/` 代码（若存在）
4. 用一句话写出**本节唯一主体**；若需要两个同等深度的主题，应拆节或在正文中明确 defer

## 正文结构（按顺序写）

```text
真实问题（三层 + 场景叙事）
→ 基础原理（机制 prose + 反例 + 排查或判断路径）
→ 最小实现（少量关键真实代码片段，解释为何 work）
→ 主流框架实现（宜短）
→ 失败分析与能力边界
→ 本节实战（精简入口；细节进 README）
→ 完成标准（运行与观察 + ≥3 自检题）
→ 本节沉淀（2–3 行）
→ 相关专题（可选，仅同课上下节）
```

### 认知链路 + 交付链路

两层缺一不可：

| 链路 | 职责 |
| --- | --- |
| **认知链路** | 为什么、机制是什么、怎么错、怎么判断/排查 |
| **交付链路** | 涉及哪些文件、怎么运行、看到什么、如何验收 |

禁止写成只有「问题 → 代码路径 → 运行 → 完成标准」而缺少机制讲解。
也禁止写成「课程正文 + demo README + 源码导览」的混合体。课程正文负责讲清机制；demo README 负责承载完整参数表、命令变体、文件对照和操作细节。

### 机制递进（多层约束专题）

当节存在「自由文本 → Prompt 约束 → API 约束 → 应用校验」或类似层层手段时，「基础原理」须写**递进理解链**：

- 每一步：解决什么、仍遗留什么、一个反例；
- 再落到当节 `source/` 最小实现；
- 不展开的能力：defer 到目标节 + 当节最小判断（例如「重试环在 06；本节只学会读 `error_stage`」）。

不能只写「项目要什么 schema → 跑哪个 demo」。

### 按节类型

| 类型 | 适用 | 要求 |
| --- | --- | --- |
| 新概念首篇 | 每门课 `00` | 是什么 / 输入输出 / 与相近概念区别；场景叙事建议 ≥400 字 |
| 普通专题 | `01` 及以后 | §6.3 双层验收 |
| 跨节能力 | 当节不展开 | 写明 defer 到第 X 节 + 当节最小判断（不能只写「见第 X 节」） |

### 代码片段

- 每篇普通专题通常 **1–2 个**内嵌片段，机制复杂时最多 **3 个**；片段来自当节真实 `source/`，标注文件路径
- 只贴能帮助理解机制的关键片段；不要为满足数量而贴源码
- 路径链接作补充，不能整节只有链接、无片段；也不能用大量片段代替 prose
- 禁止贴整文件；禁止未标注的伪代码冒充可运行入口
- 片段服务机制理解，不让代码块反客为主

### 本节实战

正文中的「本节实战」只保留：

- 本节新增能力（一句话）
- 关键文件路径（不贴完整文件树）
- 一条主命令
- 应观察到什么
- 指向 demo / package README 的链接

不要把 demo 顶部常量、样例全集、参数组合、完整配置矩阵、错误分支清单全部搬进课程正文。这些放在 README；正文只讲它们背后的实验方法和判断依据。

### Demo 与 README

按 learning-guide §6.4 决策，顺序判断：

1. 已有 demo 加参数能否观测？→ 复用
2. package 内 smoke / 测试是否够？→ 不新建 demo
3. 是否需要**新的观测维度**？→ 才新建或拆 demo
4. 长链路课 → 优先串联节，非每节一脚本

## 写后自检（全部通过才算完成）

- [ ] 场景叙事 ≥300 字（`00` 建议 ≥400 字）
- [ ] 「基础原理」去掉表格、代码块后仍可读通（不依赖点开 `source/`）
- [ ] 内嵌代码片段少而关键：普通专题 1–2 个，复杂专题不超过 3 个
- [ ] 本节实战没有搬运 README；详细参数、命令变体、配置矩阵已下沉到 README
- [ ] ≥3 道自检题，答案依赖机制理解而非背命令
- [ ] 「失败分析」至少 1 条含：表现 → 原因 → 怎么验证
- [ ] 多层约束专题：「基础原理」含机制递进链（不只写最终调用路径）
- [ ] defer 项均标明目标节号 + 当节最小判断
- [ ] 无 outline bullet 扩写冒充正文
- [ ] 无源码导览冒充正文；代码片段后有 prose 解释为什么这样设计
- [ ] 当节文档与当节 `source/` 一致（若本节有代码变更）

## 禁止

- 把 learning-guide 理解成「写短、少展开、贴近实战 = 不讲原理」
- 用表格代替机制 prose 和反例
- 用代码块、配置表、文件树和命令说明稀释课程正文
- 把 demo README 的完整操作手册搬进 `course/**/*.md`
- 未经 §6.4 决策树就新建 demo 目录
- 主动 Git commit 或打 tag（除非用户明确要求）
- 厚度不足时删解释凑简洁；应加厚 prose 与反例

## Git

- 每节完成后可建议用户 commit：`course(02_llm/01): ...`
- **不要**主动 commit
- **不要**主动打 tag；阶段 tag（如 `course/02_llm`）仅用户明确下令时执行
